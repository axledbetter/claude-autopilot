# v7.0 — Hosted product MVP (`autopilot.dev`)

**Status:** spec drafted 2026-05-07
**Origin:** v6.2.x closed out the multi-phase orchestrator track (PRs #107 → #113, npm published as `@delegance/claude-autopilot@6.2.2`). The CLI is now feature-complete for the local-first / BYO-key story. v7.0 is the **first commercial product** on top of the OSS CLI — a hosted dashboard + team operational controls priced for orgs, not seats. The CLI stays free + local + open-source forever.

## What we're building

A hosted SaaS layer that lives at `autopilot.dev` (or similar — domain TBD). It receives `events.ndjson` + `state.json` snapshots from CLI uploads, stores them per-org with tamper-evident integrity, and serves a web dashboard. The product splits into **Free** (adoption-driver: solo dev, own runs, dashboard, public share-by-URL) and **Org** (governance + finance: SSO, audit, org-wide cost reporting, admin guardrails, RBAC-light).

```
$ claude-autopilot autopilot --mode full --budget 25
[autopilot] runId=01HQK8...  budget=$25.00
[autopilot] phase 1/6: scan
[autopilot] phase 2/6: spec
...
[autopilot] ✔ run complete in 4m12s, $3.47 of $25.00
[autopilot] uploaded to https://autopilot.dev/runs/01HQK8...
```

The CLI never sends source code, secrets, or LLM API keys server-side. Only run *metadata* (events.ndjson, state.json, externalRefs, costs) flows up — and only when the user has opted in via `claude-autopilot config dashboard <api-key>`.

## Why this matters

**Commercial moment.** The CLI proves the YC pitch ("local-first agentic dev, BYO-Claude-key, multi-model council") but produces zero revenue. v7.0 is the first revenue surface — and a credible one because:

1. The thing being commoditized fastest in this market is *AI compute* (Anthropic, OpenAI, and Google are racing each other to give it away). Pricing on tokens or per-run gets undercut every quarter. v7.0's pricing is **decoupled from compute**.
2. The buyer who actually approves the purchase (CTO, CISO, finance) has different needs than the buyer who evaluates (the dev). Free + great for the dev → Paid for the org. Sentry / Linear / GitHub / Vercel all built billion-dollar businesses on this shape.
3. The feature set the org buyer cares about (audit, SSO, org-wide cost view, admin guardrails) is **the moat that survives compute commoditization**. Anthropic giving away $50/mo of free credit doesn't displace a CFO needing centralized cost reporting.

## Pricing

| Tier | What you get | Limits | Price |
|---|---|---|---|
| **Free** | CLI, hosted dashboard for your own runs, 90-day run history, public share-by-URL, Google sign-in | 100 runs/mo soft cap, 5 GB storage, no team scope, no admin | $0 forever |
| **Org Small** (<50 devs, self-attested) | Free + everything in "Org tier features" below | 1000 runs/mo soft cap, 50 GB storage, standard SLA | $99/mo (or $990/year, 2 mo free) |
| **Org Mid** (<500 devs, self-attested) | Same features, higher caps | 10000 runs/mo, 500 GB storage, priority SLA | $499/mo (or $4990/year) |
| **Enterprise** (500+ devs / regulated) | Everything + custom contract terms, dedicated support, on-prem deployment option | Custom | Custom |

**Pricing principles** (from codex pre-flight review WARNING #1):

- **No per-token markup, no per-run billing, no per-seat math.** Predictable monthly price; usage soft-capped via published infrastructure limits. When a tier's limits are hit, the path is "upgrade" not "we bill you for the overage."
- **Soft enforcement of org-size bands**: self-attested at signup, periodic true-up at annual renewal. Low procurement friction; chronic under-tiering bounded by the renewal review.
- **Annual prepay (2 months free)** mandatory at $99/$499 from day one. SMB sales cycles benefit from upfront commit; we benefit from the cash + retention signal.
- **Pledge phrasing**: "no token markup, no surprise usage bills, predictable org pricing" (not "we will never meter anything" — that locks future enterprise constructs and conflicts with downstream SSO vendor MAU economics).

## Org tier features (v7.0 paid set)

Codex review iteration #2 narrowed this from 11 candidates to 5 conversion-trigger features:

1. **SSO/SAML** — Google Workspace, Okta, Microsoft Entra. Plus: domain verification + session revocation + admin "disable user" action (closes the deprovisioning gap that opens when SCIM is deferred to v7.1).
2. **Audit log** — immutable per-action ledger of who ran what, when, with what budget cap, what cost outcome. **Tamper-evident**: server-side hash-chain on event chunks, signed upload sessions per run, `source_verified` status field on every audit row.
3. **Org-wide search + dashboard** — across all team members' runs, gated behind RLS with explicit cross-tenant negative tests.
4. **Centralized cost reporting** — CFO-readable monthly: "claude-autopilot spent $X across N devs / M PRs shipped, broken down by model / phase / repo." This is the **stronger economic-trigger feature** than RBAC for the early-buyer persona (mid-market eng manager / VP Eng watching their AI line item).
5. **Light admin guardrails** — 3 roles (`owner`/`admin`/`member`), org-default model selection, per-user max budget cap, org-toggleable public-share-link disable. The wedge of the v7.1 policy engine — marketed as "policy controls coming soon."

**Deferred to v7.1+** (codex revision): SCIM (auto-provision/deprovision via Okta/Workspace), deep RBAC + policy engine, configurable retention beyond Free's 90 days, admin analytics composites (productivity dashboards, time-to-PR, cost-per-PR-shipped).

**Stack** (separate from the Delegance app's ECS baseline — claude-autopilot is its own repo with its own deployment story):
- **Hosting**: Vercel (Next.js 16 App Router). The Delegance app uses ECS; claude-autopilot's hosted backend is *new infrastructure for a different codebase*, and Vercel is the right shape for the surface (Next.js + low-traffic SaaS + matches the existing Phase 7 cert suite which already runs against Vercel/Fly/Render). Per-handler runtime: ingest API uses Node (not Edge — needs Buffer + crypto.subtle full APIs for hash-chain verification); dashboard API uses Node. No Edge requirement anywhere in v7.0.
- **Database**: Supabase (Postgres + RLS). Same supabase-js v2.97+ patterns as Delegance — every table includes `Relationships: []`.
- **Auth**: Supabase Auth for Free tier (Google sign-in). WorkOS for SSO/SAML on Org tier (handles Okta + Entra + Google Workspace federation behind one API; ~$125/mo for the standard plan, scales by connection not MAU).
- **Billing**: Stripe Checkout + customer portal + webhooks. Entitlements stored in Postgres, keyed by Stripe product/price.
- **Storage**: Supabase Storage for events.ndjson + state.json blobs. **Tenant-scoped object keys** (`org/{org_id}/runs/{run_id}/events.ndjson` for org-tier rows, `user/{user_id}/runs/{run_id}/events.ndjson` for free-tier) — RLS on metadata alone isn't enough; the bucket policy must enforce path-prefix access by `auth.uid()` / membership join. Negative tests cover signed-URL generation and direct object fetch across org boundaries.
- **Email**: Resend (matches Delegance app).
- **Domain**: TBD — `autopilot.dev`, `claude-autopilot.dev`, or subdomain of an existing one. Decision deferred to launch prep.

## Data model

**Multi-tenant authz matrix** (codex CRITICAL #1 — this lands FIRST, before any feature ships, with negative cross-tenant tests):

```sql
-- Core tenancy
CREATE TABLE organizations (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  slug TEXT UNIQUE NOT NULL,
  name TEXT NOT NULL,
  plan TEXT NOT NULL DEFAULT 'free' CHECK (plan IN ('free','small','mid','enterprise')),
  declared_size_band TEXT NOT NULL DEFAULT 'small' CHECK (declared_size_band IN ('small','mid','enterprise')),
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE TABLE memberships (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  organization_id UUID NOT NULL REFERENCES organizations(id) ON DELETE CASCADE,
  user_id UUID NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  role TEXT NOT NULL CHECK (role IN ('owner','admin','member')),
  status TEXT NOT NULL DEFAULT 'active' CHECK (status IN ('active','disabled','removed')),
  invited_at TIMESTAMPTZ,
  joined_at TIMESTAMPTZ DEFAULT NOW(),
  UNIQUE (organization_id, user_id)
);

-- Run data — Free tier writes user_id only; Org tier always has organization_id
CREATE TABLE runs (
  id TEXT PRIMARY KEY,  -- ULID from CLI
  organization_id UUID REFERENCES organizations(id) ON DELETE CASCADE,
  user_id UUID NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  cli_version TEXT NOT NULL,
  hostname_hash TEXT,  -- one-way hash of $HOSTNAME for support triage, not deanon
  started_at TIMESTAMPTZ NOT NULL,
  ended_at TIMESTAMPTZ,
  total_cost_usd NUMERIC(10,4) NOT NULL DEFAULT 0,
  status TEXT NOT NULL CHECK (status IN ('running','success','failed')),
  source_verified BOOLEAN NOT NULL DEFAULT FALSE,  -- tamper-evident flag
  upload_session_id UUID REFERENCES upload_sessions(id),
  events_blob_path TEXT,  -- supabase storage key
  state_blob_path TEXT,
  events_chain_root TEXT,  -- hash-chain root (codex CRITICAL #2)
  visibility TEXT NOT NULL DEFAULT 'private' CHECK (visibility IN ('private','public','org'))
);

CREATE TABLE upload_sessions (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  organization_id UUID REFERENCES organizations(id) ON DELETE CASCADE,
  run_id TEXT NOT NULL,
  signed_token TEXT NOT NULL,  -- HMAC-signed JWT, 4h TTL
  expires_at TIMESTAMPTZ NOT NULL,
  consumed_at TIMESTAMPTZ
);

-- Entitlements (codex CRITICAL #3)
CREATE TABLE entitlements (
  organization_id UUID PRIMARY KEY REFERENCES organizations(id) ON DELETE CASCADE,
  stripe_subscription_id TEXT,
  stripe_product_id TEXT,
  plan TEXT NOT NULL,
  features JSONB NOT NULL DEFAULT '{}'::jsonb,
  limits JSONB NOT NULL DEFAULT '{}'::jsonb,
  -- e.g. { "runs_per_month": 1000, "storage_gb": 50, "retention_days": 365 }
  current_period_end TIMESTAMPTZ,
  cancel_at TIMESTAMPTZ,
  status TEXT NOT NULL CHECK (status IN ('active','past_due','canceled','trialing')),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- Audit log (codex CRITICAL #2 — tamper-evident)
CREATE TABLE audit_events (
  id BIGSERIAL PRIMARY KEY,
  organization_id UUID REFERENCES organizations(id) ON DELETE CASCADE,
  actor_user_id UUID REFERENCES auth.users(id),
  action TEXT NOT NULL,  -- e.g. 'run.uploaded', 'membership.disabled'
  subject_type TEXT NOT NULL,
  subject_id TEXT NOT NULL,
  metadata JSONB NOT NULL DEFAULT '{}'::jsonb,
  source_verified BOOLEAN NOT NULL,
  prev_hash TEXT,  -- hash chain for tamper-evident
  this_hash TEXT NOT NULL,
  occurred_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- Org admin guardrails
CREATE TABLE organization_settings (
  organization_id UUID PRIMARY KEY REFERENCES organizations(id) ON DELETE CASCADE,
  default_model TEXT,  -- e.g. 'claude-opus-4-7'
  max_budget_per_user_usd NUMERIC(10,2),
  allow_public_share_links BOOLEAN NOT NULL DEFAULT TRUE
);
```

**RLS policies** — every multitenant table has either `organization_id` (org-tier rows) or `user_id` (free-tier personal rows). Policy expression has **two explicit branches** to avoid unsafe `OR` patterns that leak across tenants (codex final-pass WARNING):

```sql
CREATE POLICY runs_select_policy ON runs FOR SELECT USING (
  -- Org-tier branch: row belongs to an org the caller is an active member of
  (organization_id IS NOT NULL AND EXISTS (
    SELECT 1 FROM memberships m
    WHERE m.organization_id = runs.organization_id
      AND m.user_id = auth.uid()
      AND m.status = 'active'
  ))
  OR
  -- Free-tier branch: row has no org and the caller owns it
  (organization_id IS NULL AND user_id = auth.uid())
);
```

Tested with explicit cross-tenant negative tests in `tests/rls/cross-tenant-leak.test.ts` covering search, export, audit-query endpoints, AND **direct Supabase Storage signed-URL access** across org boundaries. Both RLS branches (org-tier + free-tier) get dedicated positive AND negative tests. **No feature ships until RLS negative tests pass.**

**Audit log immutability** is enforced at the DB layer (codex final-pass NOTE): app roles get `INSERT` only on `audit_events`; `UPDATE`/`DELETE` are revoked. Inserts go through a dedicated `audit.append(...)` function that computes the next hash chain entry. Daily checkpoint of the chain root is anchored to S3 (out-of-process write, separate IAM role) for stronger evidentiary posture — v7.0 ships the daily checkpoint; cross-tenant external timestamping (e.g., transparency log) is v7.1+.

**Upload session token storage** (codex final-pass WARNING): `upload_sessions.signed_token` stores **only the JWT's `jti` + a hash of the token bytes**, never the raw signing material. Cryptographic verification happens against the inbound JWT presented by the CLI; server then matches `jti` against the stored row, checks `consumed_at IS NULL`, marks single-use on finalize. Even if the DB is exfiltrated, no replay window opens.

**Plan constraint parity** (codex final-pass WARNING): `entitlements.plan` gets the same CHECK constraint as `organizations.plan` (`free|small|mid|enterprise`). Stripe price ID → plan mapping lives in `lib/billing/plan-map.ts` as the single source of truth, consulted by the webhook handler and the entitlement seeder.

## Event ingestion trust model (codex CRITICAL #2)

CLI-uploaded `events.ndjson` is client-originated → spoofable without integrity controls → audit log credibility breaks. v7.0 ships **tamper-evident** (not tamper-proof — that's v7.1+):

1. **Signed upload sessions** — CLI calls `POST /api/upload-session` with API key → server returns short-lived (4h) HMAC-signed JWT pinned to `(user_id, run_id, organization_id)`. JWT is required for the subsequent blob upload.
2. **Hash chain on event chunks** — CLI computes a per-chunk hash chain (`chunk_n_hash = SHA256(chunk_n || chunk_{n-1}_hash)`) and uploads chunks with the chain hashes. Server verifies the chain on receipt; rejects on mismatch. Final root hash stored in `runs.events_chain_root`.
3. **Server-side append-only storage** — once a run's blobs are written to Supabase Storage, they're marked immutable via storage bucket policy. Only the upload session that created them can finalize; once finalized, no further writes.
4. **`source_verified` flag** — set TRUE only when (signed session was valid + chain verified end-to-end + run.complete event present). FALSE for any partial/orphaned run. Audit log entries copy this flag — UI surfaces "verified" badges.
5. **Out of scope for v7.0**: rotating signing keys, hardware-attested signing, cross-org event forwarding, true tamper-proof (Merkle tree + external timestamping). v7.1+ if customers ask.

## Entitlement enforcement (codex CRITICAL #3)

Stripe webhooks update `entitlements`. Every server-side handler that does anything billable consults the entitlement record before proceeding:

```typescript
// app/api/upload-session/route.ts
const ent = await getEntitlement(organizationId);
if (!ent) {
  // Free tier user — check personal limits
  await checkFreeTierLimits(userId);
} else {
  // Org tier — check Stripe-keyed entitlements
  await checkEntitlementLimits(ent, action);
  // Throws 402 + structured "upgrade tier" response when over cap
}
```

When limits hit:
- Return HTTP 402 with `{ error: 'limit_reached', limit: 'runs_per_month', current: 1042, max: 1000, upgrade_url: '...' }`
- CLI surfaces a friendly "your org has hit its monthly run limit; admin can upgrade at <url>" message
- Dashboard shows an in-app upgrade banner

**No metered overage billing.** Always upgrade-tier path.

## Breaking changes from v6.x

v7.0 is a major version, so a small batch of v6 deprecations land:

- `--no-engine` flag removed. Engine is always on. Out-of-tree consumers using this flag get an immediate `unrecognized flag` error.
- `ENGINE_DEFAULT_V6_0` and `ENGINE_DEFAULT_V6_1` constants removed from public exports. Test seam `builtInDefault: false` continues to work for legacy-mode tests.
- `RUN_STATE_SCHEMA_VERSION` bumped to **2**. v7.0 adds `runner.user`, `runner.host_hash`, `runner.cli_version` fields to state.json — useful for support triage and dashboard display. v6.2.2's range check (`MIN_SUPPORTED..MAX_SUPPORTED`) means v7.0 binaries can still resume v6.2.x runs (`MIN_SUPPORTED=1`); v6.2.x binaries cannot resume v7.0 runs.
- Migration guide section "v6 → v7" walks through these and the new opt-in dashboard feature.

These cleanups would otherwise have been their own PR (Option A from the brainstorming menu); folding into v7.0 is legal because the major version bump makes breaking changes legal, and the cleanup is small enough to ship alongside the product MVP.

## Testing strategy

**Required before any ship:**

1. **RLS negative tests** (codex CRITICAL #1): for every multitenant table + endpoint, a test that creates two orgs (A, B), inserts data into A, signs in as a member of B, asserts B cannot read/list/export A's data. Tests live in `tests/rls/`. Coverage: `runs`, `audit_events`, `entitlements`, `organization_settings`, `memberships`, search endpoints, export endpoints.
2. **Tamper-evident verification**: signed-session round-trip, hash-chain mismatch rejection, chain-end-to-end verification, `source_verified` flag accuracy on partial uploads.
3. **Entitlement enforcement**: free-tier limit hit → 402 + upgrade URL, org-tier limit hit → 402 + upgrade URL, Stripe webhook → entitlement update lag scenarios (subscription canceled but `entitlements.status` stale).
4. **SSO + deprovisioning**: WorkOS connection setup, login round-trip, admin disable-user → user's existing sessions revoked + future logins blocked.
5. **Public share links**: org admin disables → existing public links return 404.
6. **CLI upload**: opt-in works, opt-out works, partial upload at Ctrl-C, retry on transient network failure.

Static analysis: existing `validate.ts` Phase 1 catches RLS-bypass patterns, missing tenant filters in Weaviate equivalents (n/a here — no Weaviate), hardcoded secrets. Adopt unchanged.

## Open questions

1. **Domain.** `autopilot.dev` is taken? `claude-autopilot.dev`? Subdomain of `delegancebrokerage.com` is wrong (per memory_sender_domain). Decision deferred to launch prep.
2. **WorkOS vs Auth0 vs Supabase Auth alone for Org tier.** WorkOS is the quickest path to multi-IDP federation; Supabase Auth alone would require building the IDP plumbing. WorkOS pricing scales by connection (cheap until you have many enterprise connections); Auth0 scales by MAU (gets expensive fast). **Recommend WorkOS** unless cost projections at 100+ Org Small customers say otherwise.
3. **Migration guide format.** v6 → v7 is small; do we ship as a section in the existing migration-guide.md or as a fresh doc? Probably section.
4. **Self-attestation enforcement.** When an Org Small customer's actual usage exceeds the band's typical usage by a wide margin, do we auto-flag for a sales-led true-up? Or wait until annual renewal? Probably the latter for v7.0 simplicity.
5. **Free tier abuse.** A bad actor could create 50 Free accounts and aggregate runs across them. Mitigations: tie Free accounts to verified email, limit accounts per IP, soft rate-limit ingest. Acceptable for v7.0.
