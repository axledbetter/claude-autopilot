# v6.2.1 — Side-effect phase idempotency contracts (`migrate` + `pr`)

**Status:** shipped 2026-05-07. See CHANGELOG entry for v6.2.1 and PR description for the as-built file/test breakdown. Spec text below preserved as-written; small implementation deviations (notably the `migration-batch` deterministic-id fallback) are documented inline in `src/cli/migrate.ts` and the CHANGELOG "Out of scope" section.
**Origin:** Codex CRITICAL on the v6.2 orchestrator spec — side-effect phases under the orchestrator could double-apply migrations or duplicate PRs on resume if event history is incomplete. v6.2.1 is the gating PR before `migrate` + `pr` enter the orchestrator's `--mode=full` registry.

## The contract (one paragraph)

Every side-effecting phase MUST satisfy two invariants:

1. **Pre-side-effect breadcrumb:** record at least one `phase.externalRef` event BEFORE the side-effect invocation. This guarantees that a crash between "started doing the thing" and "finished doing the thing" still leaves a resume target. The breadcrumb's `id` MUST be derivable from the inputs alone (deterministic) so that a re-run that re-emits the same id is recognized as "same operation, retry" rather than "new operation, duplicate."

2. **Post-side-effect reconciliation refs:** record one `phase.externalRef` per durable artifact AFTER the side-effect completes. These are authoritative for resume — Phase 6's provider readback consults the platform of record (gh, ledger, deploy adapter) and matches against the persisted refs to decide skip-already-applied vs retry vs needs-human.

If a phase declares `hasSideEffects: true` and does NOT satisfy both invariants, the orchestrator MUST refuse to register it. v6.2.1 ships:

- The contract codified as a static check (registry-time assertion).
- `migrate` + `pr` audited and gap-closed against the contract.
- Tests that prove resume-after-partial-crash is safe.

## Audit results — current state

### `pr` (v6.0.9, already in master) — **PASSES**

`src/cli/pr.ts:287` emits `phase.externalRef({ kind: 'github-pr', id: String(pr.number), provider: 'github' })` BEFORE the `runCommandImpl()` call that does the gh work. The id is deterministic (the PR number is supplied by the caller, looked up via `gh pr view` or passed via `--pr-number`). Phase 6's `github` readback handles `github-pr` refs and maps `gh pr view <n> --json state` to `open`/`closed`/`merged`/`unknown`. No changes required.

**Resume preflight (orchestrator-side):** when phase N=`pr` has prior `phase.success` for the run, orchestrator calls `verifyRefs(priorRefs)` → if state is `open` or `merged`, short-circuit to `phase.success` with `replayed: true`. If `closed` (without merge) or `unknown`, route to `needs-human` (a closed-without-merge PR resumed under autopilot is a real edge case — branches diverge, base ref drifts — better to prompt).

### `migrate` (v6.0.8, already in master) — **GAP**

`src/cli/migrate.ts:230-235` emits `phase.externalRef({ kind: 'migration-version', id: '<env>:<name>' })` AFTER `await dispatchFn(input)` returns. If the dispatcher partially applies migrations and then crashes, the ledger has 3 of 5 migrations applied but the run dir has 0 externalRefs recorded. Resume sees no externalRefs and restarts the dispatcher from scratch. The dispatcher's ledger guard prevents double-apply at the database level (safety preserved), but the engine is blind to what happened — and the YC-demo cost meter shows phantom cost on a phase that re-runs the dispatcher's no-op skip path.

**Fix:** emit a `migration-batch` breadcrumb BEFORE the dispatch:

```typescript
run: async (input, ctx) => {
  // BEFORE — pre-side-effect breadcrumb. Deterministic id = sha256 of
  // (env, planned migration set). Same input → same id → resume recognizes
  // this as the same batch, not a new operation.
  const batchId = sha256Hex(`${input.env}:${input.plannedMigrations.join(',')}`);
  ctx.emitExternalRef({
    kind: 'migration-batch',
    id: `${input.env}:${batchId}`,
  });

  const artifact = await dispatchFn(input);

  // AFTER — one reconciliation ref per actually-applied migration
  // (existing v6.0.8 behavior, preserved).
  for (const migration of artifact.appliedMigrations) {
    ctx.emitExternalRef({
      kind: 'migration-version',
      id: `${input.env}:${migration}`,
    });
  }
  // ...
}
```

**New ref kind:** `migration-batch` joins `migration-version` in `ExternalRefKind`. Provider readback for `migration-batch` reads the ledger and compares the planned set to applied set; returns `open` (some pending), `merged` (all applied), `failed` (ledger shows error), or `unknown`.

**Resume preflight:** orchestrator sees `migration-batch:<env>:<hash>` ref AND zero or more `migration-version` refs → calls `verifyRefs([batch, ...versions])`. If batch is `merged` AND all planned versions are present in the ledger, short-circuit to success. If batch is `open` (partial), retry the dispatcher (its ledger guard handles the re-attempt). If `unknown`, route to needs-human.

## Registry-time contract enforcement

`src/core/run-state/phase-registry.ts` (introduced in v6.2.0) gains a static check on registration:

```typescript
function registerPhase<I, O>(reg: PhaseRegistration<I, O>): PhaseRegistration<I, O> {
  if (reg.build === undefined) throw new Error(`registry: missing build for ${reg.displayName}`);

  // v6.2.1 — side-effect phases MUST declare a pre-effect breadcrumb kind
  // and at least one post-effect reconciliation kind. The registry can't
  // statically verify the phase body emits them, but it CAN require the
  // phase to declare its contract so the orchestrator's resume preflight
  // knows what kinds to read back.
  if (reg.hasSideEffects && (!reg.preEffectRefKinds?.length || !reg.postEffectRefKinds?.length)) {
    throw new Error(
      `registry: side-effect phase ${reg.displayName} missing idempotency contract — ` +
      `declare preEffectRefKinds + postEffectRefKinds`,
    );
  }
  return reg;
}
```

Each registration grows two optional arrays:
- `preEffectRefKinds: ExternalRefKind[]` — "I will emit at least one ref of these kinds BEFORE my side-effect runs."
- `postEffectRefKinds: ExternalRefKind[]` — "I will emit zero or more refs of these kinds AFTER my side-effect completes successfully."

For `pr`: `preEffect = ['github-pr']`, `postEffect = []` (the github-pr ref serves both purposes — it's recorded pre-effect with the same id that gh reports post-create).

For `migrate`: `preEffect = ['migration-batch']`, `postEffect = ['migration-version']`.

Implement-phase declaration (when implement starts emitting `git-remote-push` refs in v6.2.x): `preEffect = ['git-remote-push-attempt']`, `postEffect = ['git-remote-push']`.

## Resume preflight pseudocode

```typescript
// Orchestrator, before invoking phase N's runPhase:
if (priorPhaseSuccessExists(runDir, N) && phase.hasSideEffects) {
  const refs = collectRefs(runDir, N);
  const readback = await verifyRefs(refs);

  const preDone = readback.some(r =>
    phase.postEffectRefKinds.includes(r.refKind) &&
    (r.currentState === 'merged' || r.currentState === 'live')
  );

  if (preDone) {
    emit('phase.success', { replayed: true, reason: 'side-effect-already-applied' });
    continue;  // Skip to phase N+1
  }

  const partial = readback.some(r =>
    phase.preEffectRefKinds.includes(r.refKind) &&
    r.currentState === 'open'
  );

  if (partial) {
    // Pre-effect breadcrumb says we started; post-effect refs incomplete.
    // Retry — the phase body's own ledger / preflight handles dedup.
    proceedWithRetry();
  } else {
    emit('replay.override', { reason: 'unknown-state', refs: readback });
    throw new GuardrailError('needs_human');
  }
}
```

## Tests (gating v6.2.1)

1. **`migrate` resume after partial crash** — fixture: 5 planned migrations, dispatcher applies 3 then throws. Run dir has `migration-batch` (open) + 3× `migration-version`. Resume: readback returns `migration-batch: open, migration-version: merged x3`. Decision = retry; dispatcher applies remaining 2; run completes.
2. **`migrate` resume after full success** — fixture: 5 applied, all refs present. Resume: readback returns all `merged`. Decision = skip-already-applied. Phase 6 emits `phase.success { replayed: true }`.
3. **`pr` resume after PR created** — fixture: github-pr ref present. Resume: gh pr view returns `state: open`. Decision = skip-already-applied.
4. **`pr` resume after PR closed** — fixture: github-pr ref present. Resume: gh pr view returns `state: closed`. Decision = needs-human (replay.override emitted).
5. **`migrate` registry rejection** — attempt to register a phase with `hasSideEffects: true` but no `preEffectRefKinds` → throws at registry-time.
6. **Orchestrator under run-scope budget** — partial-success resume DOES NOT double-charge cost (phase.cost from prior attempt is preserved; replay attempt records new phase.cost only for actual work performed).
