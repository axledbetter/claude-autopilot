# v7.0 Phase 3 — Stripe entitlement enforcement

**Status:** spec drafted 2026-05-07
**Origin:** v7.0 Phase 1 (DB schema + RLS) shipped in PR #115. Phase 2.1 (Next.js + Supabase Auth) in #117. Phase 2.2 (ingest API) in #119. Phase 2.3 (CLI dashboard verbs + auto-upload) in #121. Phase 3 enforces the **Free vs Org-tier pricing** the v7.0 hosted product MVP defines.

> **Anchor — platform context.** This spec lives in the `claude-autopilot` repo. The v7.0 hosted product deploys to **Vercel** per `docs/specs/v7.0-hosted-product-mvp.md` Stack rationale. The CLAUDE.md guidance about AWS ECS applies to the unrelated delegance-app repo. Ignore review feedback flagging Vercel as a platform mismatch — recurring cross-repo confusion. Webhook handlers do use raw-body reading (`await req.text()`) which works on Vercel Node runtime AND any other Next.js host.

## What we're building

Five surfaces:

1. **Stripe webhook handler** at `POST /api/stripe/webhook` — processes `checkout.session.completed`, `customer.subscription.updated`, `customer.subscription.deleted`, `invoice.payment_failed`. Idempotent on `event.id` via two-phase claim/process.
2. **Checkout Session minting** at `POST /api/dashboard/billing/checkout` — auth: org admin/owner; creates a Stripe Checkout Session for Small/Mid tier; returns the URL.
3. **Customer Portal redirect** at `POST /api/dashboard/billing/portal` — auth same; creates Stripe Customer Portal session URL using the org's persisted `stripe_customer_id`.
4. **Entitlement gate** wired into Phase 2.2's `POST /api/upload-session` — checks runs/mo + retained-storage caps before minting the upload JWT; returns HTTP 402 with `{ error: 'limit_reached', limit, current, max, upgrade_url }` when over.
5. **CLI 402 handler** in the auto-upload module from Phase 2.3 — parses the structured 402 response, prints a friendly message + upgrade URL, preserves the run's exit code.

**Out of scope:** Free dashboard UI for billing (Phase 4), Org-tier SSO/audit (Phase 5), launch prep (Phase 6).

## Pricing tiers (per v7.0 MVP spec)

| Tier | Runs/mo | Retained storage | Price |
|---|---|---|---|
| Free | 100 (soft cap) | 5 GiB | $0 |
| Org Small | 1000 | 50 GiB | $99/mo (or $990/yr) |
| Org Mid | 10000 | 500 GiB | $499/mo (or $4990/yr) |
| Enterprise | unlimited (NULL caps) | unlimited (NULL caps) | sales-led, no self-serve |

## Why this matters

Phase 1-2 built the cryptographic-credibility boundary. Phase 3 makes the boundary **commercially load-bearing**: the upload-session mint refuses to issue a JWT when an org is over its monthly cap. Without that check, the YC-pitch revenue story is decorative. With it, hitting "Upgrade" in the dashboard is the literal unblocker for the next CI run that wants to upload.

## Stripe customer mapping (codex pass 1 CRITICAL #1)

The original sketch stored only `stripe_subscription_id`/`stripe_product_id`/`stripe_price_id` on `entitlements`. Two problems:

1. **No `stripe_customer_id`** — Customer Portal session creation MUST take a customer ID. Without persisting it, the portal endpoint can't reliably resolve which Stripe customer to operate on.
2. **`client_reference_id` only on Checkout completion** — subscription lifecycle webhooks (`customer.subscription.updated|deleted`, `invoice.payment_failed`) do NOT carry `client_reference_id`. They carry `customer` and (transitively) `subscription` IDs. Resolving by `client_reference_id` alone breaks every event after the initial Checkout.

Fix: add a dedicated `billing_customers` table keyed by `organization_id`, plus persist both `stripe_customer_id` AND `stripe_subscription_id` on `entitlements`. Webhook handlers resolve in this order:
1. Look up by `stripe_subscription_id` if the event carries one.
2. Else look up by `stripe_customer_id`.
3. If neither resolves, retrieve the subscription from Stripe (with `customer` expanded) and reconcile from `subscription.metadata.organization_id` (codex pass 1 WARNING — webhook ordering safety).

Both Checkout Session AND Subscription get `metadata.organization_id` set at creation time so reconciliation always has a fallback path.

## Webhook idempotency — two-phase claim (codex pass 1 WARNING)

Naive `INSERT INTO stripe_webhook_events (id) ON CONFLICT DO NOTHING; return 200` is unsafe in two failure modes:

- Insert event row → handler crashes → Stripe retries → conflict → no-op → entitlement update permanently lost.
- Process first → handler crashes after entitlement update → Stripe retries → entitlement re-applied (possibly out-of-order if Stripe also fires a newer subscription.updated in between).

Fix: status field with claim-then-complete pattern.

```sql
CREATE TABLE stripe_webhook_events (
  id TEXT PRIMARY KEY,                  -- Stripe evt_...
  type TEXT NOT NULL,
  payload JSONB NOT NULL,
  status TEXT NOT NULL DEFAULT 'processing'
    CHECK (status IN ('processing', 'completed', 'failed')),
  error TEXT,
  received_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  completed_at TIMESTAMPTZ
);
```

Handler flow:

```typescript
// 1. Verify signature (raw body required).
const event = stripe.webhooks.constructEvent(rawBody, sig, secret);

// 2. Claim (or detect duplicate).
const { error: insertErr } = await supabase.from('stripe_webhook_events').insert({
  id: event.id, type: event.type, payload: event,
});
if (insertErr) {
  if (/duplicate/i.test(insertErr.message)) {
    // Already-claimed. Look up status.
    const { data } = await supabase.from('stripe_webhook_events')
      .select('status').eq('id', event.id).maybeSingle();
    if (data?.status === 'completed') return new Response('ok', { status: 200 });
    if (data?.status === 'processing') return new Response('still processing', { status: 200 });
    // status === 'failed' → let Stripe retry; reset to processing.
    await supabase.from('stripe_webhook_events').update({ status: 'processing', error: null })
      .eq('id', event.id);
    // Fall through to re-process.
  } else {
    return new Response('insert error', { status: 500 });
  }
}

// 3. Process.
try {
  await dispatch(event);  // calls plan-map handlers per event.type
  await supabase.from('stripe_webhook_events').update({
    status: 'completed', completed_at: new Date().toISOString(),
  }).eq('id', event.id);
  return new Response('ok', { status: 200 });
} catch (err) {
  await supabase.from('stripe_webhook_events').update({
    status: 'failed', error: (err as Error).message,
  }).eq('id', event.id);
  // Return 5xx so Stripe retries.
  return new Response('handler failed', { status: 500 });
}
```

This guarantees: every event reaches `completed` exactly once; transient failures retry; permanent failures surface in operator dashboards via the `failed` status row.

## Storage cap is RETAINED bytes, not monthly upload (codex pass 1 CRITICAL #3)

Earlier sketch said "sum runs.totalBytes for the current calendar month". Wrong — that resets monthly while object storage accumulates forever. A 5 GiB/mo upload limit with the spec's 90-day retention would let a Free user accumulate 15 GiB.

Fix: aggregate **retained** bytes. Maintain `runs.totalBytes` (already present from Phase 2.2 finalize), and define "retained" as `runs WHERE deleted_at IS NULL AND (current_date - created_at::date) <= retention_days`. Phase 4 will eventually surface deletion UX; Phase 3 just queries with the retention filter.

```typescript
async function checkStorageCap(orgId: string | null, userId: string): Promise<{ used: number; max: number }> {
  const supabase = createServiceRoleClient();
  const { data } = orgId
    ? await supabase.from('runs')
        .select('total_bytes')
        .eq('organization_id', orgId)
        .is('deleted_at', null)
        .gte('created_at', sub(now, '90 days'))
    : await supabase.from('runs')
        .select('total_bytes')
        .eq('user_id', userId)
        .is('organization_id', null)
        .is('deleted_at', null)
        .gte('created_at', sub(now, '90 days'));
  const used = (data ?? []).reduce((s, r) => s + (r.total_bytes ?? 0), 0);
  const max = await getStorageCap(orgId, userId);
  return { used, max };
}
```

Phase 1's `runs` table doesn't have `total_bytes` or `deleted_at` — Phase 3 adds them.

## Entitlement gate at upload time

Wired into Phase 2.2's `POST /api/upload-session` between ownership check and JWT mint. The check considers (in order):

1. Subscription status — if `stripe_subscription_status` is `canceled`/`unpaid` AND past `current_period_end`, treat as free tier.
2. Cancel-at-period-end — if `cancel_at_period_end = true` AND past `current_period_end`, fall through to free tier caps. Otherwise honor existing cap.
3. Payment failed — if `payment_failed_at` set AND it's been >7 days since the failure (Stripe's typical dunning window), fall through to free. Otherwise grace.
4. Per-tier caps:
   - Org tier: caps from `entitlements` row, scoped to the **whole organization** (codex pass 1 WARNING — NOT per-user inside org).
   - Free tier: caps from `personal_entitlements` keyed by `user_id`.
5. Enterprise = NULL caps = no enforcement.

```typescript
export interface EntitlementResult {
  exceeded: boolean;
  kind: 'runs_per_month' | 'storage_bytes';
  current: number;
  max: number;
  upgradeUrl: string;
  effectivePlan: 'free' | 'small' | 'mid' | 'enterprise';
}

export async function checkEntitlement(input: {
  organizationId: string | null;
  userId: string;
  action: 'mint_upload_session';
}): Promise<EntitlementResult>;
```

The 402 response payload structure matches the v7.0 hosted MVP spec exactly:

```json

  "current": 1042,
  "max": 1000,

}
```

## Plan map — runtime config validation (codex pass 1 WARNING)

`process.env.STRIPE_PRICE_SMALL!` is a TypeScript fiction; at runtime the value can still be undefined and the assertion is silent. Replace with explicit `loadBillingConfig()` that throws at startup:

```typescript
// apps/web/lib/billing/plan-map.ts
import { z } from 'zod';

const ConfigSchema = z.object({
  STRIPE_SECRET_KEY: z.string().min(20).startsWith('sk_'),
  STRIPE_WEBHOOK_SECRET: z.string().min(20).startsWith('whsec_'),
  STRIPE_PRICE_SMALL: z.string().min(20).startsWith('price_'),
  STRIPE_PRICE_MID: z.string().min(20).startsWith('price_'),
  AUTOPILOT_PUBLIC_BASE_URL: z.string().url().default('https://autopilot.dev'),
});

let cached: z.infer<typeof ConfigSchema> | null = null;

export function loadBillingConfig() {
  if (cached) return cached;
  const parsed = ConfigSchema.safeParse(process.env);
  if (!parsed.success) {
    throw new Error(`Invalid billing config: ${parsed.error.issues.map(i => `${i.path}: ${i.message}`).join('; ')}`);
  }
  cached = parsed.data;
  return cached;
}

export const PLAN_MAP = {
  small: {
    get priceId() { return loadBillingConfig().STRIPE_PRICE_SMALL; },
    runsPerMonthCap: 1000,
    storageBytesCap: 50 * 1024 * 1024 * 1024,  // 50 GiB
  },
  mid: {
    get priceId() { return loadBillingConfig().STRIPE_PRICE_MID; },
    runsPerMonthCap: 10000,
    storageBytesCap: 500 * 1024 * 1024 * 1024,
  },
} as const;

export function tierForPriceId(priceId: string): 'small' | 'mid' | null {
  if (priceId === PLAN_MAP.small.priceId) return 'small';
  if (priceId === PLAN_MAP.mid.priceId) return 'mid';
  return null;
}
```

## Routes

### `POST /api/stripe/webhook`

- **Body:** raw bytes (`await req.text()` BEFORE any JSON parsing — Stripe signature is over the raw payload).
- **Signature header:** `stripe-signature`. Verified via `stripe.webhooks.constructEvent(rawBody, sig, secret)`.
- **Returns:** 200 on success or known duplicate, 400 on malformed/invalid-signature, 500 on processing failure (Stripe retries).

### `POST /api/dashboard/billing/checkout`

- **Auth:** Supabase session cookie. Caller must be `memberships.role IN ('owner','admin')` AND `status='active'` for the requested `organization_id`.
- **Body:** `{ organizationId: string, tier: 'small' | 'mid' }`.
- **Returns:** `{ url: string }` (Stripe Checkout Session URL). Frontend navigates.
- **Errors:** 401 unauth, 403 not admin/owner, 422 invalid tier, 409 if org already on a paid subscription.

### `POST /api/dashboard/billing/portal`

- **Auth:** same as checkout.
- **Body:** `{ organizationId: string }`.
- **Returns:** `{ url: string }` (Stripe Customer Portal session URL).
- **Errors:** 401, 403, 404 if no `billing_customers` row for the org (means never paid → no portal yet).

### `POST /api/upload-session` (extended from Phase 2.2)

After existing ownership check, before mint:

```typescript
const ent = await checkEntitlement({
  organizationId: run.organization_id,
  userId: caller.userId,
  action: 'mint_upload_session',
});
if (ent.exceeded) {
  return NextResponse.json({
    error: 'limit_reached',
    limit: ent.kind,
    current: ent.current,
    max: ent.max,
    upgrade_url: ent.upgradeUrl,
  }, { status: 402 });
}
```

## CLI 402 handling (extends Phase 2.3 uploader)

In `src/dashboard/upload/uploader.ts`'s `bootstrapSession`:

```typescript
if (mintRes.status === 402) {
  const body = await mintRes.json() as {
    error: string; limit: string; current: number; max: number; upgrade_url: string;
  };
  throw new UploadLimitError(
    `upload rejected — ${body.limit} cap reached (${body.current}/${body.max}). Upgrade at ${body.upgrade_url}`,
    body,
  );
}
```

`UploadLimitError` is a typed subclass of `UploadError` so the auto-upload entry point in `src/cli/autopilot.ts` can detect it and print without retry:

```typescript
if (err instanceof UploadLimitError) {
  console.warn(`[autopilot] ${err.message}`);
  // exit code preserved from run, NOT the upload error
  return;
}
```

## Hard constraints

- Webhook handler reads RAW body via `await req.text()` (NOT `await req.json()` — would break signature verification).
- Stripe customer ID is persisted at Checkout completion AND every subsequent webhook reconciles by subscription/customer ID, NOT by `client_reference_id` alone.
- Storage cap = retained bytes (with 90-day retention filter), NOT monthly upload volume.
- Org-tier run cap counts ALL runs in the org for the calendar month, not per-user.
- Enterprise rows have NULL caps enforced by CHECK constraint.
- Webhook idempotency uses claim-then-complete pattern with status field.
- `loadBillingConfig()` validates env at runtime and throws on missing/malformed.
- 402 response payload shape is the documented contract: `{ error, limit, current, max, upgrade_url }` — CLI parses these field names.
