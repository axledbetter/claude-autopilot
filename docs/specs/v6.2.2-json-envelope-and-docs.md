# v6.2.2 — `claude-autopilot autopilot --json` envelope + docs polish

**Status:** spec drafted 2026-05-07
**Origin:** v6.2.0 shipped the multi-phase orchestrator; v6.2.1 added side-effect idempotency contracts. The remaining v6.2.x deliverable is the `--json` envelope so CI consumers get a deterministic machine-readable shape, plus the migration guide section explaining the one-runId model.

## What we're building

Three deliverables, one PR:

1. **Outer JSON envelope** for `claude-autopilot autopilot --json` — exactly one valid envelope on stdout, including pre-run failures (config validation, registry errors, uncaught exceptions). Per codex NOTE #6 from the v6.2 review.
2. **Migration guide section** "v6.1 → v6.2: one runId across the pipeline" — concrete walkthrough of how the user's existing pipeline scripts can keep working OR migrate to `claude-autopilot autopilot`.
3. **Cache contract version marker** — `.guardrail-cache/runs/<ulid>/state.json` already has `schemaVersion`; this PR adds explicit cross-version-resume rejection ("older run state schema version 0; this binary requires schema version >=1") so a stale run dir from a prior major doesn't crash the orchestrator with an opaque shape error.

## Why this matters

**For CI consumers:** today `claude-autopilot autopilot --json` is reserved (orchestrator throws `not_implemented` when the flag is passed). CI scripts that want to gate a deploy on the orchestrator's outcome have to parse stderr NDJSON or shell-out to `claude-autopilot runs show <id> --json` after the fact. One outer envelope lets them branch on `.exitCode` / `.failedPhase` / `.totalCostUSD` directly.

**For users migrating from v5.x or v6.0.x:** the migration guide today documents `runPhaseWithLifecycle` and the per-phase wraps but doesn't have a concrete "you can replace your shell script with one verb" walkthrough. A 30-line section closes that gap.

**For long-lived run dirs:** v6.2.x didn't bump `schemaVersion` (v6.0 layout was still good), but a v7 will. Today's `replayState()` would crash with `TypeError: cannot read property 'phases' of undefined` on an unknown shape — replacing that with an explicit `schemaVersion` mismatch error means the user sees "run dir is from a newer/older version, this CLI can't resume it" instead of a cryptic stack trace.

## Outer JSON envelope shape

```typescript
type AutopilotJsonEnvelope = {
  version: '1';
  verb: 'autopilot';
  runId: string | null;          // null for pre-run failures
  status: 'success' | 'failed';
  exitCode: 0 | 1 | 2 | 78;
  phases: PhaseResult[];          // empty for pre-run failures
  totalCostUSD: number;
  durationMs: number;
  // Failure metadata, present when status === 'failed':
  errorCode?: AutopilotErrorCode;
  errorMessage?: string;
  failedAtPhase?: number;
  failedPhaseName?: string;
};

// Bounded enum (per codex NOTE #5 — keep CI parsers' branching reliable).
// CI consumers MAY rely on these specific strings; new codes ship as
// minor versions of the envelope.
type AutopilotErrorCode =
  | 'invalid_config'      // pre-run validation, --phases parse, engine off
  | 'budget_exceeded'     // run-scope budget cap hit
  | 'lock_held'           // engine lock collision
  | 'corrupted_state'     // state.json mismatch / schemaVersion out of range
  | 'partial_write'       // events.ndjson torn write
  | 'needs_human'         // interactive prompt would fire in --json
  | 'phase_failed'        // generic phase failure (LLM error, network, etc.)
  | 'internal_error';     // catch-all for the uncaughtException handler

// Exit-code-to-errorCode mapping (per codex WARNING #3 — must be
// deterministic for CI scripts):
//   exit 0   → no errorCode (status: 'success')
//   exit 1   → invalid_config | phase_failed | internal_error
//   exit 2   → lock_held | corrupted_state | partial_write
//   exit 78  → budget_exceeded | needs_human

type PhaseResult = {
  name: string;
  status: 'success' | 'failed' | 'skipped-replay';
  costUSD: number;
  durationMs: number;
  // For skipped-replay phases (resume short-circuit):
  reason?: 'side-effect-already-applied' | 'idempotent-replay';
};
```

Pre-run failure example (engine disabled by env):

```json
{"version":"1","verb":"autopilot","runId":null,"status":"failed","exitCode":1,"phases":[],"totalCostUSD":0,"durationMs":12,"errorCode":"invalid_config","errorMessage":"engine disabled via CLAUDE_AUTOPILOT_ENGINE=off; autopilot requires engine-on"}
```

Successful run example:


## Channel discipline

- **stdout**: exactly ONE JSON envelope (final). Per-phase `phase.cost` events do NOT appear on stdout in `--json` mode.
- **stderr**: NDJSON event stream (existing `events.ndjson` shape, line-buffered). One event per line.
- **No ANSI codes** in `--json` mode — wrapped through `stripAnsi` from the existing `src/cli/json-envelope.ts` helper.
- **Interactive prompt would fire** → exit `78` with the standard "needs-human" envelope (same as other v6 verbs).

**Exactly-once guarantee under fatal paths** (per codex WARNING #2). The orchestrator's existing `try/catch` is not sufficient — async unhandled rejections, early module-init throws, and `process.exit()` calls outside the catch all bypass it. Implementation:

1. **Single-write latch**: `writeAutopilotEnvelope` flips a module-scoped boolean before writing; subsequent calls no-op. Prevents double-emission if a finalization error fires after the success envelope.
2. **Process-scoped guards**: when `--json` is set, the autopilot CLI entrypoint installs `process.on('uncaughtException', ...)` and `process.on('unhandledRejection', ...)` handlers that emit a fallback failure envelope (`runId: null` if unknown, `errorCode: 'internal_error'`, `errorMessage: <stringified>`) and call `process.exit(1)`. The handlers consult the single-write latch — if an envelope already shipped, they no-op and exit silently.
3. **stdout flush before exit**: orchestrator awaits `new Promise(r => process.stdout.write('', r))` after the envelope to guarantee the bytes hit the pipe before `process.exit`.

## Implementation

### `src/cli/autopilot.ts` — replace the `not_implemented` throw with envelope writer

The current code (per master commit 4251910) throws when `--json` is passed. Replace with:

```typescript
if (options.json) {
  return runAutopilotWithJsonEnvelope(options, dispatchOpts);
}
```

`runAutopilotWithJsonEnvelope` wraps the existing orchestrator logic, captures the per-phase outcomes into a `PhaseResult[]`, and emits the envelope on success or failure. All existing per-phase NDJSON events continue to flow to stderr unchanged.

Pre-run failures (engine validation, `--phases` parse) are already caught by the `try/catch` around the orchestrator — extend it to emit a pre-run envelope when `--json` is set.

### Migration guide additions

`docs/v6/migration-guide.md` gets a new top-level section:

```markdown
## v6.1 → v6.2: one runId across the pipeline

If your CI / shell script today looks like:

    claude-autopilot scan
    claude-autopilot spec
    claude-autopilot plan
    claude-autopilot implement
    claude-autopilot migrate
    claude-autopilot pr

each invocation creates its own runId. Your $25 budget cap applies per phase, not per pipeline. A failure in `implement` leaves three orphan run dirs.

Replace with (after v6.2.1 ships migrate + pr into the registry):

    claude-autopilot autopilot --mode full --budget 25

One runId across all six phases. (Note: in v6.2.0, `--mode full` runs scan→spec→plan→implement only; v6.2.1 wires migrate + pr; the JSON envelope examples in this section assume the v6.2.1 phase set.) `runs watch <id>` shows the full pipeline ticking down a single budget. Resume from a mid-pipeline failure short-circuits the completed phases via their `phase.success` events.

Per-verb invocations continue to work — v6.2 doesn't deprecate them. The orchestrator is sugar.
```

### Cache contract versioning

`src/core/run-state/state.ts` already reads `RUN_STATE_SCHEMA_VERSION` from `state.json`. v6.2.2 adds a min/max compatibility window (per codex WARNING #1 — strict equality would block resume across rolling deploys / mixed binary fleets):

```typescript
export const RUN_STATE_MIN_SUPPORTED_SCHEMA_VERSION = 1;
export const RUN_STATE_MAX_SUPPORTED_SCHEMA_VERSION = RUN_STATE_SCHEMA_VERSION; // current = 1

// In replayState():
if (state.schemaVersion < RUN_STATE_MIN_SUPPORTED_SCHEMA_VERSION ||
    state.schemaVersion > RUN_STATE_MAX_SUPPORTED_SCHEMA_VERSION) {
  throw new GuardrailError(
    `run dir at ${runDir} has schemaVersion ${state.schemaVersion}; this binary supports schemaVersion ${RUN_STATE_MIN_SUPPORTED_SCHEMA_VERSION}..${RUN_STATE_MAX_SUPPORTED_SCHEMA_VERSION}. ` +
    `Use the version of claude-autopilot that created this run dir, or delete the run dir to start fresh.`,
    { code: 'corrupted_state' }
  );
}
```

Future minor versions can additively expand the schema while preserving forward-read compatibility — bump `RUN_STATE_SCHEMA_VERSION` (writer) without bumping `MIN_SUPPORTED` (reader). Major bumps (v7) will reset `MIN_SUPPORTED` to break with the past explicitly. The migration guide records the policy.

## Tests

- 6 new envelope tests in `tests/cli/autopilot-json-envelope.test.ts`:
  - Successful run produces parseable envelope with expected fields
  - Pre-run failure (engine off) produces envelope with `runId: null`, `errorCode: 'invalid_config'`, `exitCode: 1`
  - Mid-pipeline failure produces envelope with `failedAtPhase` + `failedPhaseName`
  - `--json` mode emits NO ANSI codes on stdout (regression for v6 Phase 5 contract)
  - **Stdout purity under stderr load** (per codex NOTE #6): orchestrator runs with simulated `phase.cost` events at high rate; assert stdout is exactly one JSON line, all events on stderr
  - **Single-write latch + uncaughtException** (per codex WARNING #2): orchestrator phase throws after envelope is written; uncaughtException handler observes the latch is set, no second envelope emitted
- 4 new schema-version tests in `tests/run-state/state.test.ts`:
  - `replayState` accepts a `schemaVersion` inside the supported range
  - `replayState` throws `corrupted_state` when `schemaVersion` is below `MIN_SUPPORTED`
  - `replayState` throws `corrupted_state` when `schemaVersion` is above `MAX_SUPPORTED`
  - Range error message names both bounds for operator triage
