# v5.4 — first-class Vercel deploy adapter

**Status:** spec, not yet implemented
**Tracks:** v5.3.0 generic `deployCommand` (PR #56)
**Why now:** the generic adapter works for any project with one CLI command, but loses the rich state Vercel exposes (deploy IDs, build logs, rollback). Three competitors that pitch deploy-loop closure (Devin, Factory, Render's own dashboard) all rely on platform-specific APIs. We need this to match the demo claim.

---

## What v5.3 did

```yaml
deployCommand: "vercel --prod"
healthCheckUrl: "https://my-app/healthz"
```

Runs the command, extracts the first `https?://` from stdout, polls health check, posts result to PR. Works on day one for any deploy CLI. **Limit:** we only know what `vercel` printed to stdout. We don't know:
- The deploy ID (so we can't link to the build dashboard or trigger a rollback)
- Build status (queued / building / ready / error)
- Build logs in real-time (we wait for the CLI to exit then have a static blob)
- Whether the same commit is already deployed (so we re-deploy uselessly)
- Health checks Vercel itself runs

## What v5.4 ships

A first-class `deploy: vercel` adapter that uses Vercel's REST API (`https://api.vercel.com/v13/deployments`) for everything, and falls back to the v5.3 generic command path when the user's project isn't a Vercel app.

```yaml
deploy:
  adapter: vercel
  project: my-app                 # vercel project id or slug
  team: my-team                   # optional, for team accounts
  target: production              # production | preview
  watchBuildLogs: true            # stream logs to stderr in real time
  rollbackOn:
    - healthCheckFailure
    - smokeTestFailure
healthCheckUrl: "https://my-app/healthz"
```

## API surface

```bash
claude-autopilot deploy                              # uses adapter from config
claude-autopilot deploy --adapter vercel             # CLI override
claude-autopilot deploy --watch                      # stream logs
claude-autopilot deploy --pr 42                      # post upserting comment
claude-autopilot deploy rollback                     # rollback to previous prod deploy
claude-autopilot deploy rollback --to <deploy-id>    # rollback to specific deploy
claude-autopilot deploy status                       # show current prod deploy + last 5 builds
```

## Adapter contract

```ts
export interface DeployAdapter {
  readonly name: string;                                          // 'vercel' | 'fly' | 'render' | 'generic'
  deploy(input: DeployInput): Promise<DeployResult>;
  status?(input: DeployStatusInput): Promise<DeployStatusResult>;  // optional — generic adapter omits
  rollback?(input: DeployRollbackInput): Promise<DeployResult>;    // optional — generic omits
}

export interface DeployResult {
  status: 'pass' | 'fail' | 'in-progress';
  deployId?: string;                // adapter-native ID (Vercel uses dpl_xxx)
  deployUrl?: string;
  buildLogsUrl?: string;
  durationMs: number;
  output?: string;                  // human-readable for PR comment
  rolledBackTo?: string;            // populated when adapter auto-rolled back
}
```

## Implementation phases

1. **Phase 1 (~3h)** — `src/adapters/deploy/vercel.ts` with deploy + status. Calls `POST /v13/deployments` with the project's git ref, polls `GET /v13/deployments/<id>` until `state === 'READY' | 'ERROR' | 'CANCELED'`, returns. No log streaming yet.
2. **Phase 2 (~2h)** — log streaming via `GET /v2/deployments/<id>/events?builds=1` (server-sent events). Print to stderr in real time when `--watch`.
3. **Phase 3 (~1h)** — rollback via `POST /v13/deployments/<id>/promote`.
4. **Phase 4 (~1h)** — auto-rollback wired to `rollbackOn: [healthCheckFailure]`. When health check fails after deploy, fetch previous prod deploy ID and promote it. **[done]**
5. **Phase 5 (~30m)** — `claude-autopilot deploy status` and `rollback` CLI subcommands. Just thin wrappers over the adapter methods.
6. **Phase 6 (~1h)** — Auth detection. `VERCEL_TOKEN` env or `~/.local/share/com.vercel.cli/auth.json`. Doctor check.

## Auth model

```bash
# Required
export VERCEL_TOKEN=...                  # personal access token (vercel.com/account/tokens)

# Optional (auto-detected from .vercel/project.json if `vc link` was run)
export VERCEL_ORG_ID=team_xxx
export VERCEL_PROJECT_ID=prj_xxx
```

If the token's missing, the adapter throws a `GuardrailError({ code: 'auth', provider: 'vercel' })` with a one-line remediation pointing at `vercel.com/account/tokens`. Same pattern as the LLM adapters.

## Failure modes the user should never have to debug

1. **Token expired or revoked** → adapter throws `auth` error with pointer.
2. **Project not in account** → throws `auth` with "your token can't see project X — check team scope".
3. **Build itself fails** → `status: 'fail'`, `buildLogsUrl` populated, last 50 log lines in `output`.
4. **Build succeeds but health check fails** → auto-rollback if configured; otherwise `fail` with both URLs (new build + old build) in PR comment.
5. **Network blip mid-poll** → retry 3x with exponential backoff, then bail with a clear "Vercel API unreachable" error.

## How the generic adapter co-exists

`deploy.adapter: vercel` uses the rich path. `deploy.adapter: generic` (or omitted with `deployCommand` set) keeps the v5.3 command-based flow. No breaking change. The setup auto-detector (`init`) sniffs for `.vercel/project.json` and writes `adapter: vercel` when it finds one.

## What v5.4 deliberately does NOT do

- **Doesn't ship Fly.io adapter** — same pattern, separate PR (~half the work; their API is similar but events are over WebSocket).
- **Doesn't ship Render adapter** — same.
- **Doesn't pre-deploy preview URLs in PR comments** — that's a Vercel-side feature (preview deployments per PR); we wait for it to land then read.
- **Doesn't manage env vars** — read-only on the deploy side; env management is a separate skill.
- **Doesn't replace `vercel deploy` for existing users** — if you're happy with the generic adapter, keep it. The first-class adapter is opt-in.

## Open questions for review

1. Do we need a streaming log API in `DeployAdapter`, or is `--watch` flag enough on the CLI side?
2. Should `rollbackOn: [smokeTestFailure]` integrate with the existing `validate` phase, or stay scoped to the deploy phase only?
3. Where does the deploy ID get persisted for `rollback` to find "previous"? `.guardrail-cache/deploys.jsonl`? Or always query Vercel API?

## Test plan

- Unit: 12+ tests against a mocked `fetch` for adapter contract conformance (success, build-fail, auth-fail, network-blip retry, log-streaming, rollback).
- Integration: 1 test against Vercel's actual API using a sandbox project (skipped in CI without `VERCEL_TOKEN_TEST`).
- E2E: real deploy of a hello-world Next.js project on every release (~$0 cost, ~30s).

## Estimated total effort

~10 hours over 1–2 days. PR could ship as **v5.4.0** with Vercel adapter only, then v5.4.1+ for Fly/Render in parallel.
