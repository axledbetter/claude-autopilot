# v7.0 Phase 5.6 — WorkOS SSO sign-in flow

**Status:** spec PR draft (next: PR creation)

**Builds on:** Phase 5.4 (`docs/specs/v7.0-phase5.4-workos-setup.md`) — admin has already connected an SSO connection via the Admin Portal; `organization_settings.sso_connection_status='active'` and `workos_connection_id` is set.

**Out-of-scope (deferred to Phase 5.7):** admin disable-user, real-time session revocation across active sessions, IdP-initiated SCIM/just-in-time user provisioning beyond email/name, MFA step-up. Phase 5.7 will close those.

## Goal

End-to-end WorkOS SSO sign-in for end users in orgs that have:

1. An active connection (Phase 5.4)
2. At least one verified domain claim
3. (Optionally) `sso_required = true` — non-SSO sign-in attempts for users whose email matches a verified domain are redirected to SSO.

After successful sign-in, the user is dropped on `/dashboard` with a Supabase session cookie indistinguishable from a Google-OAuth or magic-link sign-in.

## Why this scope

The Phase 5.4 codex pass split 5.6 from the sign-in surface explicitly because Supabase session minting from an external IdP profile is the highest-risk security unknown. This phase contains it, with three discrete sub-features that all must ship together for SSO to be usable end-to-end:

- **Domain claim** — required so we know which user emails belong to which org. Without this, callback can't route a user to the right tenant.
- **Sign-in entry + WorkOS authorize redirect** — the actual flow.
- **`sso_required` enforcement at sign-in time** — without it, claiming SSO doesn't actually keep users on it.

Bundling all three is intentional: shipping any subset would be unusable in production.

## Surfaces

### Backend routes

1. `POST /api/dashboard/orgs/:orgId/sso/domains` — admin-only. Body `{ domain: string }`. Normalizes (lowercase, strip leading `@`/`https://`, validate via simple regex), inserts a `pending` claim row with a fresh 32-byte hex challenge token, returns `{ id, domain, status: 'pending', challengeRecordName: '_workos-verify.<domain>', challengeRecordValue: <token> }`.

2. `POST /api/dashboard/orgs/:orgId/sso/domains/:domainId/verify` — admin-only. Performs DNS TXT lookup on `_workos-verify.<domain>`. If any returned TXT record value matches the stored challenge token, the route calls `mark_domain_verified` RPC (flips status to `'verified'`, records `verified_at`). On mismatch, returns 422 `verification_failed`.

3. `DELETE /api/dashboard/orgs/:orgId/sso/domains/:domainId` — admin-only. Soft-disable (status `'revoked'`, `revoked_at` timestamp). Verified-then-revoked entries do NOT free up the domain for another org without an admin support process — full delete needs ops approval. (Records linger so audit trail is preserved.)

4. `PATCH /api/dashboard/orgs/:orgId/sso/required` — owner-only. Body `{ ssoRequired: boolean }`. Sets `organization_settings.sso_required`. Audit append.

5. `POST /api/auth/sso/start` — public (anonymous-acceptable). Body `{ email?: string, orgId?: string }`. Resolves org from email's domain (verified claim) or directly from orgId. Validates org has active SSO. Generates WorkOS authorization URL with state cookie (HttpOnly Secure SameSite=Lax 10min, value = SHA-256 of a server-stored nonce so we can replay-detect on callback). Returns `{ authorizationUrl }`. Browser navigates.

6. `GET /api/auth/sso/callback` — WorkOS posts query params `code`, `state`. Validates state cookie matches. Exchanges code for profile via `workos.userManagement.authenticateWithCode()` (or `workos.sso.getProfileAndToken()` for SSO-classic). From profile: `email`, `firstName`, `lastName`, `organizationId` (WorkOS), `connectionId`. Server-side:
   - Look up local `organization_settings` by `workos_organization_id` — must be active.
   - `record_workos_sign_in` RPC: finds-or-creates `auth.users` row by email, ensures active membership exists in the bound org (auto-provision as `member` if first-time), audit-appends.
   - Mint Supabase session via admin-mediated magic link path (see "Session minting" below).
   - Set Supabase cookies on response, redirect to `/dashboard`.

### UI

7. `/login/sso` — small page with email input. Submit POSTs to `/api/auth/sso/start`. Surface error states cleanly (no SSO connection / domain not claimed / org disabled).

8. Existing sign-in surface — middleware/server check on `auth.callback` cookie path: if profile email's domain matches a verified claim whose org has `sso_required=true`, server signs the user out (revoke session) and redirects to `/login/sso?email=<encoded>`. Done in `/api/auth/callback` Phase 2.1 route, NOT in middleware (middleware doesn't have user yet).

### Admin page

9. `/dashboard/admin/sso` (Phase 5.4 page) gains:
   - Domain claims section: list of claims with status, "Add domain" form, "Verify" button per pending claim showing the `_workos-verify.<domain>` TXT record name + challenge value, "Revoke" for verified claims.
   - SSO required toggle (owner-only): switch + "Save" → PATCH route. Confirms when enabling: "Existing non-SSO sessions stay valid until next sign-in. Force sign-out is a Phase 5.7 feature."

### RPC contracts (full bodies in plan)

- `claim_domain(p_caller_user_id, p_org_id, p_normalized_domain, p_challenge_token) → jsonb` — admin/owner-only. **Caller normalizes domain in route** (lowercase, punycode IDN, reject paths/ports — see "Domain normalization" below). Refuses if a row with `ever_verified=TRUE` exists for `lower(domain)` in any other org (`domain_already_claimed`). Refuses if a pending claim exists in this org for same domain (`domain_already_pending`). Inserts row, audit appends `org.sso.domain.claim_started`.

- `mark_domain_verified(p_caller_user_id, p_org_id, p_domain_id) → jsonb` — admin/owner-only. Route does the DNS check; RPC flips status + sets `ever_verified=TRUE`. Re-checks unique-on-`ever_verified` atomically inside the txn (two pending claims on same domain race — only one wins). Audit appends `org.sso.domain.verified`.

- `revoke_domain_claim(p_caller_user_id, p_org_id, p_domain_id) → jsonb` — admin/owner-only. Sets status='revoked', `revoked_at=now`. `ever_verified` stays TRUE (codex pass-1 CRITICAL #1). Audit appends.

- `set_sso_required(p_caller_user_id, p_org_id, p_required) → jsonb` — owner-only. **Asymmetric guard (codex pass-1 WARNING #7):** turning OFF (`p_required=FALSE`) succeeds regardless of `sso_connection_status` so an admin can recover from a misconfigured-then-disabled connection. Turning ON (`p_required=TRUE`) refuses with `no_active_sso` if `sso_connection_status` is not 'active'. Audit appends `org.sso.required.toggled`.

- `consume_sso_authentication_state(p_nonce, p_workos_organization_id, p_workos_connection_id) → jsonb` — service-role. Atomically loads + marks-consumed the state row matched by nonce. Validates: not expired, not already consumed, `workos_organization_id` matches, `workos_connection_id` matches. Returns `{ stateId, organizationId, initiatedEmail }` on success or one of `state_not_found | state_expired | state_already_consumed | state_workos_org_mismatch | state_workos_connection_mismatch`. Codex pass-1 CRITICAL #2 + WARNING #5.

- `record_workos_sign_in(p_organization_id, p_email, p_workos_user_id, p_workos_organization_id, p_workos_connection_id) → jsonb` — service-role only. **Codex pass-1 CRITICAL #3: enforces verified-domain binding.**
  - First, requires `lower(split_part(p_email, '@', 2))` to match a row in `organization_domain_claims` with `(organization_id=p_organization_id, status='verified')`. Otherwise raises `email_domain_not_claimed_for_org`.
  - Resolves local org from `workos_organization_id` and asserts it matches `p_organization_id` (otherwise `unknown_org`).
  - Identity-linkage path (codex pass-1 WARNING #6): looks up `workos_user_identities` by `(workos_user_id, workos_organization_id)`. If found, returns the linked `user_id`. If no link exists, looks up `auth.users` by lowercase-email; if found, inserts an identity link row (this captures the WorkOS-issued identity for future sign-ins even if email changes).
  - If neither identity link nor auth.users row exists, returns `{ result: 'user_not_provisioned', email, organizationId }` — route then calls `supabase.auth.admin.createUser`, then re-runs `record_workos_sign_in`.
  - Always upserts membership: if no active membership exists for (user_id, organization_id), inserts one with `role='member'`. Audit appends `org.sso.user.signed_in` with details.
  - Returns `{ result: 'linked', userId, membershipCreated, identityCreated }`.

## Session minting

The hardest part. Phase 5.4 codex pass-2 explicitly flagged this as the bigest unknown.

### Constraint

Supabase Auth's `signInWithIdToken({ provider: 'oidc', token })` requires the Supabase project to have the WorkOS OIDC issuer pre-configured under Authentication → Providers. We can't programmatically configure that per-org. Hosted product tenancy precludes the workos.com top-level OIDC config.

### Approach: admin-mediated magic-link

After WorkOS callback validates and `record_workos_sign_in` returns the local `user_id`:

**Codex pass-2 WARNING #3 — email-policy on link mint.** When the user is identity-linked (existing `workos_user_identities` row), Phase 5.6 mints with the LINKED Supabase user's CURRENT email (looked up via `auth.admin.getUserById(userId)`), NOT the WorkOS profile email. Reason: IdP email may have changed since linkage; using the linked email guarantees the magic link targets the right Supabase user. Mismatch handling: if WorkOS profile email differs from linked email, audit `org.sso.email_drift_observed` (NOT a hard failure). IdP-driven email update flow ships in Phase 5.7.

For first-time provisioning (`record_workos_sign_in` returned `user_not_provisioned`), the route creates the Supabase user with the WorkOS profile email; subsequent linkage uses that email.

1. Look up linked Supabase user's current email: `const { data: u } = await supabase.auth.admin.getUserById(userId); const linkedEmail = u.user.email`.
2. `supabase.auth.admin.generateLink({ type: 'magiclink', email: linkedEmail })` — returns `{ properties: { hashed_token, action_link, ... } }`. Magic link is good for one consumption.
2. Server-side, immediately consume the token using `supabase.auth.verifyOtp({ token_hash: hashed_token, type: 'magiclink' })` (codex pass-1 CRITICAL #4 — token_hash is the field, not token; this matches @supabase/supabase-js v2.97+ contract). Use an ANON client (NOT service-role, since verifyOtp is a public-API operation). Returns `{ session, user }`.
3. **Codex pass-1 WARNING #8 — verify the minted session matches the intended user.** Assert `session.user.id === expectedLocalUserId AND session.user.email?.toLowerCase() === normalizedProfileEmail`. If either fails: revoke the new session via `supabase.auth.admin.signOut(session.access_token, 'global')`, audit `org.sso.session.mismatch`, return 500 `session_user_mismatch` without setting any cookies.
4. Set the session cookies on the outgoing response using the Phase 2.1 `@supabase/ssr` cookie helpers (`setAll([{ name: 'sb-...', value, options }, ...])`).
5. Audit append `org.sso.session.minted`.
6. 302 redirect to `/dashboard`.

This is the documented workflow Supabase recommends for "admin-controlled sign-in." It does not bypass any Supabase auth security, generates a real session with a real refresh token, and works without per-tenant project config.

### User auto-provisioning

If `record_workos_sign_in` returned `user_not_provisioned`:

1. Route calls `supabase.auth.admin.createUser({ email, email_confirm: true, user_metadata: { workos_user_id, first_name, last_name, source: 'workos_sso' } })`.
2. On success, route re-runs `record_workos_sign_in` (will find via `workos_user_identities` + create membership).
3. Continue with magic-link minting and session-mismatch verification (step 3 above).

If `createUser` fails because email already exists with a different provider (race condition with a concurrent Google sign-in for the same email), the route returns 409 `user_email_collision` and the user is shown a "Try signing in with Google instead, or contact support" page. Codex pass 1 will scrutinize this branch.

## Domain normalization

Codex pass-1 WARNING #7 + pass-2 WARNING #5 + NOTE #1. New helper `apps/web/lib/dns/normalize-domain.ts` uses the maintained `tldts` package (active maintenance, IDN-aware, exposes `getPublicSuffix()` + `getDomain()`):

```typescript
import { domainToASCII } from 'node:url';
import { getPublicSuffix } from 'tldts';

export type NormalizeResult = { ok: true; domain: string } | { ok: false; reason: string };

export function normalizeDomain(input: string): NormalizeResult {
  if (typeof input !== 'string' || input.length === 0 || input.length > 253) {
    return { ok: false, reason: 'invalid_input' };
  }
  const trimmed = input.trim().toLowerCase();
  if (trimmed.includes('://') || trimmed.includes('/') || trimmed.includes(':') || trimmed.includes('@')) {
    return { ok: false, reason: 'must_be_bare_domain' };
  }
  const noTrailing = trimmed.replace(/\.+$/, '');
  const ascii = domainToASCII(noTrailing);
  if (ascii === '') return { ok: false, reason: 'invalid_format' };
  // Must have at least one dot (rejects 'localhost' etc).
  if (!ascii.includes('.')) return { ok: false, reason: 'must_have_tld' };
  // Reject if the entire input IS a public suffix (e.g. `co.uk`, `com`).
  const publicSuffix = getPublicSuffix(ascii, { allowPrivateDomains: true });
  if (publicSuffix === ascii) return { ok: false, reason: 'public_suffix_only' };
  return { ok: true, domain: ascii };
}

export function normalizeEmailDomain(email: string): NormalizeResult {
  if (typeof email !== 'string') return { ok: false, reason: 'invalid_input' };
  const at = email.lastIndexOf('@');
  if (at < 1 || at === email.length - 1) return { ok: false, reason: 'invalid_email' };
  return normalizeDomain(email.slice(at + 1));
}
```

`normalizeEmailDomain` is the SHARED helper for: `/api/auth/sso/start`, `record_workos_sign_in` (route normalizes before passing to RPC), `/api/auth/callback` `enforceSsoRequired`, and `record_workos_sign_in`'s domain match (route normalizes; SQL receives pre-normalized value).

`record_workos_sign_in`'s signature changes to accept `p_normalized_email_domain` from the route, so the SQL doesn't need to re-implement normalization (codex pass-2 WARNING #5):

`record_workos_sign_in(p_organization_id, p_email, p_normalized_email_domain, p_workos_user_id, p_workos_organization_id, p_workos_connection_id) → jsonb`

The RPC asserts `p_normalized_email_domain != ''` and uses it as-is to match `organization_domain_claims.domain`.

## DNS verification

`apps/web/lib/dns/verify-txt.ts`. Codex pass-2 WARNING #4 — `resolveTxt` doesn't honor AbortController, so use `Promise.race` for an actual timeout bound:

```typescript
import { resolveTxt } from 'node:dns/promises';

export type VerifyResult = { ok: true } | { ok: false; reason: string };

export async function verifyTxtChallenge(
  fqdn: string,
  expectedToken: string,
  timeoutMs = 5_000,
): Promise<VerifyResult> {
  const lookup = (async (): Promise<VerifyResult> => {
    try {
      const records = await resolveTxt(fqdn);
      // resolveTxt chunks long values into string[]; join per record before compare.
      for (const chunkArr of records) {
        const value = chunkArr.join('');
        if (value === expectedToken) return { ok: true };
      }
      return { ok: false, reason: 'no_matching_txt_record' };
    } catch (err) {
      const code = (err as { code?: string }).code;
      if (code === 'ENOTFOUND' || code === 'ENODATA') return { ok: false, reason: 'no_txt_records' };
      return { ok: false, reason: code ?? 'dns_error' };
    }
  })();
  const timeout = new Promise<VerifyResult>((resolve) =>
    setTimeout(() => resolve({ ok: false, reason: 'timeout' }), timeoutMs),
  );
  return Promise.race([lookup, timeout]);
}
```

New helper `apps/web/lib/auth/enforce-sso-required.ts`:

```typescript
import { createServiceRoleClient } from '@/lib/supabase/service';
import { normalizeEmailDomain } from '@/lib/dns/normalize-domain';

export type EnforcementResult =
  | { action: 'allow' }
  | { action: 'redirect_to_sso'; email: string };

export async function enforceSsoRequired(email: string | null | undefined): Promise<EnforcementResult> {
  if (!email) return { action: 'allow' };
  const norm = normalizeEmailDomain(email);
  if (!norm.ok) return { action: 'allow' };  // malformed email — let auth handle it
  const svc = createServiceRoleClient();
  // Two queries (codex pass-2 WARNING — no FK between domain_claims and
  // organization_settings, PostgREST embedded join would require defining one).
  const { data: claim } = await svc
    .from('organization_domain_claims')
    .select('organization_id')
    .eq('domain', norm.domain)
    .eq('status', 'verified')
    .maybeSingle();
  if (!claim) return { action: 'allow' };
  const { data: settings } = await svc
    .from('organization_settings')
    .select('sso_required, sso_connection_status')
    .eq('organization_id', (claim as { organization_id: string }).organization_id)
    .maybeSingle();
  const s = settings as { sso_required: boolean; sso_connection_status: string } | null;
  if (s?.sso_required && s?.sso_connection_status === 'active') {
    return { action: 'redirect_to_sso', email };
  }
  return { action: 'allow' };
}
```

## State protocol (codex pass-2 CRITICAL #2)

One protocol, no ambiguity:

**At `/api/auth/sso/start`:**

1. Resolve `localOrgId` (from email-domain → verified claim → organization_id, or directly from authenticated `orgId` param — see Auth requirement below). Resolve `workosOrganizationId` + `workosConnectionId` from `organization_settings`.
2. Generate `stateId = randomUUID()` and `nonce = randomBytes(32).toString('hex')`.
3. Insert `sso_authentication_states` row with `id = stateId`, `nonce = sha256(nonce hex digest)`, `organization_id = localOrgId`, `workos_organization_id`, `workos_connection_id`, `initiated_email`, `expires_at = now() + 10 min`.
4. Build cookie value: `payload = JSON.stringify({ stateId, nonce })`. Sign: `signed = base64url(payload) + '.' + base64url(hmacSha256(payload, SSO_STATE_SIGNING_SECRET))`. Set `Set-Cookie: sso_state=<signed>; HttpOnly; Secure; SameSite=Lax; Max-Age=600; Path=/`.
5. Build WorkOS authorize URL with `state = stateId` (the WorkOS query param sees only the opaque ID).
6. Return `{ authorizationUrl }`.

**At `/api/auth/sso/callback`:**

1. Read `state` query param → must equal `cookie.stateId` after HMAC verify.
2. Verify HMAC on cookie. Reject `invalid_state` if signature fails.
3. Parse `{ stateId, nonce }` from cookie payload. Verify `query.state === stateId`.
4. Call `consume_sso_authentication_state(p_state_id := stateId, p_nonce_hash := sha256(nonce hex digest), p_workos_organization_id := profile.organizationId, p_workos_connection_id := profile.connectionId)`. RPC returns one of:
   - `state_not_found` (no row by ID — reject 401)
   - `state_expired` (now > expires_at — reject 401)
   - `state_already_consumed` (consumed_at IS NOT NULL — reject 401, replay)
   - `state_nonce_mismatch` (cookie nonce hash doesn't match stored — reject 401, attacker forged ID without nonce)
   - `state_workos_org_mismatch` / `state_workos_connection_mismatch` (callback profile doesn't match start — reject 401, tenant confusion)
   - On success: returns `{ stateId, organizationId, initiatedEmail }`. Continues with sign-in.
5. Continue with `record_workos_sign_in` etc.

This is the ONLY state model in the spec. Schema comment + table column docs match this protocol.

New env: `SSO_STATE_SIGNING_SECRET` (32+ random bytes). Documented in `.env.example`.

### Updated RPC: `consume_sso_authentication_state`

Signature changes per the protocol above:

`consume_sso_authentication_state(p_state_id uuid, p_nonce_hash text, p_workos_organization_id text, p_workos_connection_id text) → jsonb`. The stored row's `nonce` column holds the SHA-256 hex digest, NOT the raw nonce.

### Authentication requirement on `/api/auth/sso/start`

Codex pass-2 WARNING #8 — `orgId`-mode start leaks tenant config to anonymous callers.

**Decision:** Phase 5.6 ships email-only public start. The `orgId`-direct mode is removed from the route. Admin-mediated SSO test sign-in (e.g. for testing a connection without a verified domain) is deferred to Phase 5.7's settings page where it requires an authenticated admin session.

The route shape becomes `POST /api/auth/sso/start` with body `{ email: string }` only. Errors return generic `{ error: 'sso_unavailable' }` to anonymous callers; specific reasons (`no_sso_for_email`, `sso_not_active`, etc.) are logged server-side for support visibility. UI copy stays generic to avoid the enumeration vector.

## Lifecycle ordering

`set_sso_required(true)` requires `sso_connection_status='active'` (codex pass-1 WARNING #7 → pass-2 confirmed asymmetric guard). If a connection is later disabled (Phase 5.4 disconnect), `sso_required` is **left set**. Re-enabling restores enforcement; admins can manually toggle off via the settings page (turning OFF works in any state per the asymmetric guard).
