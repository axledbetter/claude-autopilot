# v6.0 — Run State Engine

**Status:** spec drafted 2026-05-04
**Origin:** Codex 5.3 review of the post-v5.6 gap analysis named this as the single highest-leverage next build for the autopilot harness — bundles four separately-tracked gaps (resume/retry, idempotency, cost cap, typed events) into one architectural subsystem that unlocks all of them.

## What we're building

A **Run State Engine** that turns autopilot from a stateless command-stream into a checkpointed, resumable, budget-bounded, observable pipeline. Every phase transition writes to disk; every run gets an ID; every command can resume, replay, or attach to in-flight runs. A single config knob caps total spend; another caps per-phase spend; another caps council recursion depth. Every state transition emits a typed JSON event to stderr so CI consumers can drive the pipeline programmatically.

Concretely, we're shipping:

1. **Run ledger format** at `.guardrail-cache/runs/<run-id>/` — `state.json` snapshot, `events.ndjson` append-only log, per-phase artifact pointers
2. **Phase wrapper / lifecycle** — every existing phase (brainstorm, plan, implement, migrate, validate, PR, review, deploy) gets instrumented to write checkpoints; no behavioral change
3. **Resume / status / list verbs** — `claude-autopilot run resume <id>`, `runs list`, `runs show <id>`
4. **Budget enforcement** — preflight cost estimate per phase, hard-stop on cap breach, council recursion-depth cap
5. **Typed JSON event stream** — NDJSON to stderr; `--json` flag on every command for structured stdout
6. **Idempotency contracts** — each phase declares whether replay is safe; side-effect phases (deploy, migrate) require explicit rules
7. **Real adapter certification suite** — live integration tests for Vercel + Fly + Render against sandbox accounts (separate but blocked by the events stream landing)

## Why this matters

After Codex review, three of v5.6's four "shipped" stories have a hidden reliability gap that a real user will hit in their first week:

- **Mock-only deploy adapters.** Bugbot caught a HIGH on Render (service-scoped URL) and a MEDIUM on cursor-dedup precisely because the spec author got the API shape wrong. Other URL/payload divergences likely lurk. Without real-API certification, first deploy failure mode is "agent-followed-spec-literally, spec was wrong, user finds out at 2 a.m."
- **No resume from failed phase.** Implement fails on a transient tsc error halfway through Phase 3 of 5 → user starts over from brainstorm. They do this twice and switch tools.
- **Cost can run unbounded.** A council that re-asks the synthesizer 20 times costs $50 the user didn't approve. Existing `appendCostLog` records the damage; doesn't prevent it.

The Run State Engine is the architectural piece that lets all three stop being design-time problems. With it, mock-only adapters become "mock-tested + real-API-certified per release." Resume-from-failed-phase is one verb. Cost cap is one config field that the runner checks before every spawn. Typed JSON events make CI integration possible.

## Non-goals (deliberate)

- **Distributed runs.** Single-machine, single-user. Multi-node coordination is out of scope.
- **Replacement of the existing skill chain.** Skills stay as-is; the Engine wraps invocations, doesn't supersede them.
- **Running phases in parallel.** v6 is sequential phase-by-phase. Parallel implement (multi-worktree) is a future enhancement.
- **Replacing `appendCostLog`.** It stays as the persistence primitive; the Engine adds the policy layer on top.
- **Web UI.** Pure CLI + JSON events. Any web UI is a downstream consumer of the events stream.

## Architecture

### State on disk

```
.guardrail-cache/
  runs/
    01HZK7… (ULID — sortable + readable)/
      state.json         # current snapshot of the DAG
      events.ndjson      # append-only event log; the source of truth
      artifacts/          # ALWAYS copy, never symlink (Codex WARNING — security)
        spec.md           # copy of docs/specs/<...> at run-start time
        plan.md
        impl-diff.patch   # generated by implement phase
        validate.sarif    # output of validate phase
        deploy.json       # adapter-emitted DeployResult
      artifacts.json      # { "<name>": { sha256, size, originalPath, copiedAt } }
      phases/
        brainstorm.json
        plan.json
        implement.json
        ...
  index.json             # PURE CACHE; rebuildable from runs/*/events.ndjson (Codex NOTE)
```

`index.json` is treated as a cache, never authoritative. On any inconsistency between `index.json`, `state.json`, and `events.ndjson` — `events.ndjson` wins, `state.json` is rebuilt from it, then `index.json` is rebuilt from each run's snapshot. An `index.rebuilt` event is emitted to mark the rebuild. A `claude-autopilot runs doctor` verb forces a full rebuild + integrity check across all runs.

`state.json` is a checkpoint — last-known-good snapshot. `events.ndjson` is the source of truth — replay-able to rebuild `state.json` if it's corrupted.

### Persistence protocol (Codex CRITICAL #1)

Concurrent writers would silently corrupt the log without explicit ordering. Required protocol:

- **Per-run advisory lock** at `.guardrail-cache/runs/<run-id>/.lock` via `proper-lockfile`. Acquired on `run start` / `run resume`; held for the duration of the run. A second invocation against the same run-id fails fast with "run X is owned by writer <writerId>; pass --force-takeover to claim ownership".
- **Single-writer invariant.** Every event includes `writerId` (PID + machine hostname hash). The writer that holds the lock is the only writer that may append. `--force-takeover` writes a `lock.takeover` event before claiming.
- **Durable append.** `open(events.ndjson, O_APPEND)` + write one line + `fsync`. Then write `state.json` to a temp file + atomic `rename` + `fsync(dirfd)`. Order is strict: append event → fsync → state snapshot → rename → fsync dir. Crash before the snapshot leaves an event without a snapshot — recoverable on next open via replay.
- **Monotonic seq** is assigned by the holding writer (stored in lock file metadata so a takeover knows where to resume the seq). Receivers of the events stream MUST detect gaps and error.
- **Truncation detection.** Last-line partial-JSON on read is rejected; the recovered state pretends that event never happened. The next event after recovery is `run.recovered-from-partial-write`.

### Run lifecycle

```
new ──▶ pending ──▶ phase[0]:running ──┬──▶ phase[0]:succeeded ──▶ phase[1]:running ──▶ ...
                                       └──▶ phase[0]:failed ──▶ paused
                                            (resumable from this phase)
```

Every state transition writes one event to `events.ndjson`:

```jsonl
{"ts":"2026-05-04T18:00:00.000Z","runId":"01HZK7...","seq":1,"event":"run.start","phases":["brainstorm","plan","implement","validate","pr","review"],"config":{...}}
{"ts":"...","runId":"...","seq":2,"event":"phase.start","phase":"brainstorm","phaseIdx":0,"idempotent":false}
{"ts":"...","runId":"...","seq":3,"event":"phase.cost","phase":"brainstorm","provider":"anthropic","inputTokens":1200,"outputTokens":3400,"costUSD":0.07}
{"ts":"...","runId":"...","seq":4,"event":"phase.success","phase":"brainstorm","durationMs":120000,"artifacts":["docs/specs/2026-05-04-foo.md"]}
{"ts":"...","runId":"...","seq":5,"event":"phase.start","phase":"plan","phaseIdx":1,...}
...
{"ts":"...","runId":"...","seq":N,"event":"run.complete","status":"success","totalCostUSD":4.20,"durationMs":1850000}
```

Event schema is versioned (`schema_version: 1`). Every event has `seq` (monotonic) so consumers can detect gaps.

### Phase contract

```ts
export interface RunPhase<I = unknown, O = unknown> {
  readonly name: string;
  readonly idempotent: boolean;
  readonly hasSideEffects: boolean;        // true for migrate, deploy, pr-push
  estimateCost?(input: I): { lowUSD: number; highUSD: number };  // for budget preflight
  run(input: I, ctx: PhaseContext): Promise<O>;
  /** Called when resuming after a previous failure. Receives last attempt's
   * partial output (if any) and decides: skip, retry, abort, or require
   * human approval. Default: side-effecting phases require human approval;
   * idempotent phases retry. */
  onResume?(prev: PhaseResumeContext): Promise<'skip' | 'retry' | 'abort' | 'needs-human'>;
}
```

Existing phase implementations get wrapped — no rewrite.

### Budget enforcement

Config block in `guardrail.config.yaml`:

```yaml
budgets:
  perRunUSD: 25            # hard stop; prompts user to extend before continuing
  perPhaseUSD: 10          # per phase; deploy/migrate get 0 by default (their cost is platform-side)
  councilMaxRecursionDepth: 3   # synthesizer bounded
  bgAutopilotMaxRoundsPerSelfEat: 5  # autopilot safety on its own pipeline
  conservativePhaseReserveUSD: 5    # see below
```

Two-layer enforcement (Codex WARNING — estimates can fail open):

- **Layer 1 (advisory)** — if a phase implements `estimateCost`, the runner uses `estimate.high` to preflight. Pause-and-prompt (or hard-fail in CI) if the estimate would push the run over `perRunUSD`.
- **Layer 2 (mandatory runtime guard)** — independent of estimate. The runner enforces `actualCostSoFar + conservativePhaseReserveUSD <= perRunUSD` before spawning ANY phase, regardless of whether `estimateCost` is present. If unknown, behavior is the same as if `estimate.high === conservativePhaseReserveUSD`. Phases without estimates can therefore still trigger budget gates.
- Every check emits a typed `budget.check` event with the decision rationale (`{phase, estimatedHigh, actualSoFar, reserveApplied, capRemaining, decision: 'proceed' | 'pause' | 'hard-fail'}`) so consumers can attribute spend.

Council recursion depth: every synthesizer self-call increments `councilDepth`; exceeding `councilMaxRecursionDepth` aborts the council with `partial` status, never deeper.

### Idempotency rules + external operation ledger (Codex CRITICAL #2)

Replay rules can't be heuristic ("if PR not yet created") — they need persisted external references so the decision is deterministic. Every side-effect phase MUST persist its `externalRefs` into `state.json` and emit them as `phase.externalRef` events:

```ts
interface ExternalRef {
  kind: 'github-pr' | 'github-comment' | 'git-remote-push' | 'deploy' | 'migration-version' | 'rollback-target';
  id: string;            // PR number, comment ID, commit SHA, deploy ID, migration filename, etc.
  provider?: string;     // 'github' | 'vercel' | 'fly' | 'render' | 'supabase' | 'rails' | ...
  url?: string;          // human-readable artifact link
  observedAt: string;    // ISO timestamp of the platform's confirmation
}
```

`onResume` decisions are made purely from persisted refs + a provider read-back check (e.g., "is PR #123 still open?"). Missing or inconsistent refs → `needs-human` always; a `--force-replay` override writes an explicit `replay.override` event with user-supplied reason.

| Phase | Idempotent? | Has side effects? | externalRefs emitted | Default replay rule |
|---|---|---|---|---|
| `brainstorm` | no (produces new content each time) | no | `spec-file` | retry on failure |
| `plan` | no | no | `plan-file` | retry on failure |
| `implement` | partial (git ops are idempotent given same input) | yes (touches filesystem, possibly remote branch) | `git-remote-push` (commit SHA + branch) | retry if no `git-remote-push` ref, else needs-human |
| `migrate` | no | yes | `migration-version` per env | needs-human; on resume, read-back `migration_state` table to determine actual applied state |
| `validate` | yes (read-only) | no | `sarif-artifact` | retry freely |
| `pr` | no (creates / updates GitHub PR) | yes | `github-pr` | retry if no `github-pr` ref; if ref present, read-back via `gh pr view <id>` and update the existing one |
| `review` | yes | no | `review-comments` | retry freely; comment-by-id dedup |
| `bugbot` | partial (replies are idempotent-per-comment-id; auto-fixes are not) | yes | `github-comment` per reply, `git-remote-push` per fix | needs-human; per-comment-id replay safe |
| `deploy` | no | yes | `deploy` (provider deploy ID), optional `rollback-target` | needs-human always; provider read-back on resume |

`needs-human` raises an interactive prompt (or a hard-fail in CI mode) before resume continues.

### Resume command

```bash
# List recent runs
claude-autopilot runs list                 # newest 20
claude-autopilot runs list --status=paused # only resumable

# Show a run's state
claude-autopilot runs show <run-id>        # full state.json + last 5 events
claude-autopilot runs show <run-id> --events    # full ndjson tail

# Resume from where it stopped
claude-autopilot run resume <run-id>       # picks up at first non-succeeded phase
claude-autopilot run resume <run-id> --from validate   # explicit phase

# Tear down
claude-autopilot runs gc                   # delete completed runs older than 30d (configurable)
claude-autopilot runs delete <run-id>      # explicit
```

### CLI `--json` mode + strict channel discipline (Codex WARNING)

Every existing command grows a `--json` flag that switches stdout from text to a typed JSON envelope:

```json
{
  "schema_version": 1,
  "command": "scan",
  "runId": "01HZK7...",
  "status": "pass",
  "findings": [...],
  "costUSD": 0.21,
  "durationMs": 38000,
  "exit": 0
}
```

`--json` implies non-interactive AND imposes a hard channel contract:

- **stdout** — exactly ONE JSON envelope per command invocation. Nothing else.
- **stderr** — ONLY NDJSON event lines. No human-readable warnings, no progress bars, no color codes.
- **All warnings, prompts, and human diagnostics route to typed events** (`run.warning`, `run.recovery`, `phase.needs-human`, `budget.check`, etc.). The text-mode logger is disabled.
- **Interactive prompts (`needs-human`)** in `--json` mode hard-fail with `exit: 78` (custom: "needs-human in non-interactive mode") and the envelope's `nextActions` field carries the resume hint.

A test asserts that under `--json` mode, no command emits anything to stdout that fails `JSON.parse` and nothing to stderr that fails NDJSON parse. Mixed-content from any caller is a regression.

### Real adapter certification suite (Phase 7)

Separate test target `npm run test:adapters:live`. Skipped without `VERCEL_TOKEN_TEST` / `FLY_API_TOKEN_TEST` / `RENDER_API_KEY_TEST`. Per-provider it asserts:

1. **Deploy success path** — push a hello-world image / repo, get a `dpl_xxx` / release ID, poll until terminal, assert `status === 'pass'` + `deployUrl` reachable.
2. **Auth failure path** — bad token, expect `auth` error code with friendly message.
3. **404 path** — wrong project / app / service ID, expect `not_found` error.
4. **Rollback path** — deploy v1, deploy v2, rollback, assert v1 URL is now serving.
5. **Log streaming path** — subscribe to streamLogs, assert lines arrive within timeout, assert redaction on a planted secret in deploy output.

Free-tier accounts handle this; cost ~$0/month. Runs nightly via `.github/workflows/adapter-cert.yml` on a schedule.

**Flake control (Codex NOTE — provider rate-limit / cold-start / eventual-consistency reality):**
- Per-provider retry budget: 3 attempts with exp backoff (1s / 4s / 16s) on transient categories (5xx, rate-limit, network).
- **Hard-fail (no retry)** for deterministic checks: auth-error path, 404 path, schema mismatch.
- **Soft-fail with alert** for rollout/log-streaming flakes: log a warning + alert channel ping but don't break the workflow run. After three consecutive soft-fails on the same check, escalate to hard-fail.
- All cert runs persist `events.ndjson` + last 200 log lines as workflow artifacts so a triage engineer can replay the failure without re-running the workflow.

## Open questions for review

1. **ULID vs UUIDv7 for run IDs?** ULID is sortable + URL-safe + 26 chars; UUIDv7 is the same shape but standardized. Lean ULID for terminal readability but happy to flip.
2. **Should `events.ndjson` be the SoT and `state.json` be a derived snapshot?** Codex called this out as best practice. Yes, but `state.json` provides O(1) status queries — keep both, with state.json as a cache.
3. **Where do non-pipeline commands (scan, run, fix) fit?** They're stateless one-shot reviews today. Should they get run IDs too (so events show up in the same stream) or stay outside the Engine? Lean: yes, give them run IDs but mark them `single-phase`.
4. **Replay rules for `pr` phase** — when the previous attempt pushed to remote but the PR creation step failed, do we update the existing branch or open a new one? Default: update existing if branch was pushed; CI mode hard-fails.
5. **`--non-interactive` everywhere** — currently inconsistent across commands. v6 needs uniform handling: every prompt has a CI fallback (either env-var override or hard-fail).

## Failure modes the user should never have to debug

1. **Missing `.guardrail-cache/runs/`** → engine bootstraps empty state, no error.
2. **Corrupted `state.json`** → rebuild from `events.ndjson` automatically; warn on stderr.
3. **Truncated `events.ndjson`** → log explicit warning, mark run as `unknown`, refuse to resume without `--force`.
4. **Crash mid-event-write** → next run notices the gap (seq jump or partial JSON line) and emits a `run.aborted-by-crash` synthetic event so observers can react.
5. **Run interrupted by Ctrl-C mid-phase** → handler writes `phase.aborted-by-user` event before exit, marks run as paused, prints resume hint.

## Migration path (v5.6 → v6) + precedence matrix (Codex NOTE)

- Existing autopilot installs work as-is; Run State Engine is opt-in via `engine.enabled: true` in `guardrail.config.yaml`.
- Default flips to `enabled: true` in v6.1 after a stabilization period.
- Old stateless behavior remains accessible via `--no-engine` for one minor version.

**Precedence (highest wins):**

1. CLI flag (`--engine` / `--no-engine` / `--json` / `--non-interactive`).
2. Env var (`CLAUDE_AUTOPILOT_ENGINE=on|off`, etc).
3. `guardrail.config.yaml` (`engine.enabled`, `budgets.*`).
4. Built-in default (v6.0: off; v6.1+: on).

**Engine-on behavior is uniform across all commands** — single-phase verbs (`scan`, `run`, `fix`, `costs`) get a synthetic single-node DAG with run ID + events stream, identical contract to multi-phase pipeline runs. There is no "some commands get run IDs, some don't" mode. `--no-engine` is the only escape hatch and is single-version-supported.

## What v6 deliberately defers

- **Parallel phase execution** (multi-worktree implement) — engine supports it architecturally but doesn't enable it.
- **Run sharing across machines** — local-only ledger.
- **Web UI** — events stream is the API; UI is a downstream consumer.
- **GitHub App** — orthogonal product.
- **Skill-version pinning** — package.json hygiene; not part of the engine.
