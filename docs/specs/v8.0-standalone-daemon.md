# v8.0 — Standalone autopilot daemon

> **Status:** spec only — design doc, no implementation.
> **Track:** v8 (next major). Builds on v7.1.x (hosted product MVP +
> day-1 polish + benchmark loop closure).
> **Driving question:** can `claude-autopilot` work as a standalone
> product that anyone can run on any repo, without sitting in a
> Claude Code chat session?

## Background

Through v7.1.x we shipped 16 PRs end-to-end with the autopilot loop
running inside a single Claude Code session. The loop drives:
brainstorm → spec → 3 codex passes → plan → codex pass on plan →
worktree → impl agent dispatch → CI monitor → codex PR review →
bugbot triage → admin merge → memory update.

This works extremely well for the operator who is already in a
Claude Code session. It does NOT work as a product. To become a
product the loop needs to:

1. Run **without** a Claude Code session — as a standalone CLI
   daemon (`autopilot watch <repo>`) or hosted service.
2. Authenticate to GitHub + LLM providers without the operator
   manually sitting in a chat to approve each step.
3. Have an **explicit trust/permission/rollback model** so users
   can grant the daemon scoped capabilities (e.g. "can open PRs in
   this repo, cannot merge to protected branches, cannot touch
   `auth/` paths") and the daemon enforces them.
4. Have a **billing model** — hosted with seats + usage, or
   BYO-key local CLI (free), or hybrid.

The codex review of the autopilot product-direction brainstorm
(`/tmp/codex-input-autopilot-product-direction-2026-05-09.md`)
identified two CRITICALs that gate Option C (standalone daemon):

* **C1**: Daemon needs an explicit trust/permission/rollback model.
* **C2**: Admin-merge cannot be the default behavior for a productized
  autopilot.

Plus 7 WARNINGs that shape the design (parallel pipeline needs
durable state machine; codex-pass count should be measured-A/B not
guessed; path-filtered CI; pricing should be hybrid; observability;
benchmark across repo types; templates ≠ repo understanding).

This spec folds all 9 codex findings into a concrete v8 design.

## Goal

Ship `@delegance/autopilot-daemon` (separate npm package) that
provides:

* `autopilot watch <repo>` — local daemon that polls a GitHub repo,
  runs the autopilot pipeline on incoming /autopilot issues,
  enforces a per-repo policy file, opens PRs, monitors CI, posts
  codex/bugbot reviews. Default: never auto-merges; opens PR with
  "ready for human review" status.
* Optional **hosted tier** at autopilot.dev that runs the daemon
  on the user's behalf with a GitHub App + dashboard for
  policy/audit/billing.
* Reuses the existing `@delegance/claude-autopilot` npm package as
  a library — the daemon imports `runScaffold`, `runScan`, the
  codex-review script, etc. No duplication.

## Non-goals

* **Not** a competitor to GitHub Copilot Workspace / Devin / Factory
  — those are full-stack hosted services with their own
  infrastructure. v8 is "claude-autopilot but you can leave it
  running on your laptop overnight."
* **Not** auto-merge by default. Opt-in per-repo via policy file.
* **Not** spec-writing automation. The brainstorming + user-review
  gate stays — operator (or an org's TPM/EM) writes specs and
  triggers the daemon by labeling/commenting on a GitHub issue.
* **Not** a multi-tenant data plane. Each daemon instance owns one
  repo's pipeline. (Hosted tier orchestrates many single-tenant
  instances.)

                     │  autopilot-daemon (Node)     │

### State machine

Every autopilot run is a record in a per-repo SQLite DB at
`~/.claude-autopilot/runs/<repo-id>/state.db`. Phases:

```
queued → spec → spec-review → plan → plan-review → impl →
ci-running → pr-review → bugbot → ready_for_human → merged | failed
```

Each transition writes an audit entry to `state.db` AND to
`.autopilot/audit.log` in the repo (committed by the daemon as
part of the PR). The audit log is append-only and signed; tampering
is detectable on next daemon restart.

This addresses **codex W2** (parallel pipeline needs durable state
machine, not just worktrees). Phase N+1 can dispatch in parallel
with phase N's CI as long as both are tracked in the state machine
and rebases are coordinated.

### Policy file

`.autopilot/policy.yaml` lives in the repo:

```yaml
version: 1
trigger:
  # Only run when these conditions are met.
  issue_label: autopilot
  authorized_users:  # Members of these GH teams can trigger.
    - my-org/engineering
  authorized_paths_only: false  # If true, label MUST also include path scope

permissions:
  # What the daemon can do.
  open_pr: true
  merge_pr: false                # Default OFF — codex C2.
  merge_pr_to: []                # Allowlist of branches the daemon may merge to.
  protected_paths:               # Daemon refuses to write here.
    - "auth/**"
    - "**/secrets/**"
    - "infrastructure/**"
  protected_files:               # Daemon refuses to modify these specific files.
    - ".github/workflows/release.yml"
    - "package.json"             # Operator-controlled deps
  max_pr_size_loc: 500           # Reject PRs that touch more LOC than this.
  max_files_changed: 30

budget:
  # Hard caps per run AND per day.
  max_run_usd: 25.00
  max_day_usd: 200.00
  on_exceed: stop                # vs `pause-for-human`

review:
  codex_passes:
    spec: 1                      # Was 3 in v7.1.x — A/B-measured, see codex W3.
    plan: 1
    pr_diff: 1
  bugbot:
    enabled: true
    auto_apply_patches: false    # Codex C2 — never auto-apply unless opted in.

emergency_stop:
  # Operator can halt the daemon by creating either of these:
  file: .autopilot/STOP          # Empty file in repo root
  issue_label: autopilot-stop    # Apply to any open issue

observability:
  log_target: stderr             # OR `sentry`, `datadog`
  webhook: null                  # Optional: POST run events to this URL
```

The daemon refuses to start if `.autopilot/policy.yaml` is missing
(safety: no implicit-default permissions). On first `autopilot
init` it scaffolds a maximally-restrictive policy file (no
permissions enabled) and prints the path for the operator to
review + opt in.

This addresses **codex C1** (explicit trust/permission/rollback
model) and **C2** (admin-merge not default).

### Trust + rollback

* **Trust scope:** the daemon authenticates to GitHub via the
  user's `gh` CLI auth (BYO mode) or via a GitHub App with
  scoped permissions (hosted mode). LLM keys are loaded from
  `~/.claude-autopilot/keys.json` (mode 0600); never sent to a
  hosted service in BYO mode.
* **Rollback:** every PR the daemon opens carries an audit-trail
  comment listing exact commit SHAs + LLM spend + policy decisions.
  The operator can revert the PR via standard GitHub UI; the
  daemon detects the revert in its next poll and writes a
  `reverted` audit entry to `state.db` (so the daemon won't
  re-attempt the same change blindly).
* **Emergency stop:** `.autopilot/STOP` (any non-empty existence)
  causes the daemon to abort all in-flight runs immediately and
  refuse to start new ones until the file is removed. This is the
  break-glass.

### Pricing model (hybrid per codex W6)

* **Local CLI (free):** `npm install -g @delegance/autopilot-daemon`
  + `autopilot watch <repo>`. Uses operator's own LLM keys.
  Operator pays Anthropic + OpenAI directly. Free forever.
* **Hosted tier (paid):** seat-based + usage-based.
  * Seats: $X / engineer / month covers GitHub App orchestration,
    dashboard, audit log, multi-repo policy management.
  * Usage: pass-through LLM costs at cost (no markup) + a small
    orchestration fee per run.
  * Hard spend caps + transparent itemized billing.
* **Wedge:** local CLI is the adoption path; hosted is the
  business model. Per codex W6, this is the right shape — pure
  hosted requires too much trust on day 1, pure BYO is hard to
  monetize.

## Components + data flow

### Daemon process

```
src/
├── daemon.ts               # Long-running entry point: `autopilot watch`
├── state-machine/
│   ├── runs.ts             # Run lifecycle (queued → ready_for_human)
│   ├── transitions.ts      # Validates state changes; emits audit
│   └── store.ts            # SQLite (sql.js or better-sqlite3)
├── policy/
│   ├── load.ts             # Parse + validate .autopilot/policy.yaml
│   ├── enforce.ts          # Pre-action gate (path, branch, size, budget)
│   └── audit.ts            # Append-only signed audit log writer
├── poller/
│   ├── github.ts           # Polls labeled issues + PR state via gh CLI
│   └── stop-flag.ts        # Watches .autopilot/STOP for emergency stop
├── pipeline/
│   ├── run-spec.ts         # Wraps brainstorming/codex-review skill output
│   ├── run-plan.ts         # Wraps writing-plans skill
│   ├── run-impl.ts         # Dispatches Claude API impl agent
│   ├── run-ci-monitor.ts   # Polls GH Actions for the PR
│   ├── run-pr-review.ts    # Calls codex-pr-review.ts
│   └── run-bugbot.ts       # Triages bugbot comments
└── cli/
    ├── init.ts             # `autopilot init` — scaffold policy + DB
    ├── watch.ts            # `autopilot watch` — start daemon
    ├── status.ts           # `autopilot status` — list active runs
    └── stop.ts             # `autopilot stop <run-id>` — graceful halt
```

### Reuses claude-autopilot library

The daemon NEVER spawns the `claude-autopilot` CLI as a subprocess
during pipeline execution. It imports the underlying functions:

```typescript
import { runScaffold } from '@delegance/claude-autopilot/scaffold';
import { runValidate } from '@delegance/claude-autopilot/validate';
import { runCodexReview } from '@delegance/claude-autopilot/codex';
```

This requires `@delegance/claude-autopilot` to **export its
internal functions as a library** in addition to its CLI surface.
That's a v7.2.x prerequisite.

### Run lifecycle data shape

```typescript
interface RunRecord {
  id: string;                    // ULID
  repo: string;                  // owner/name
  trigger: {
    issue_number: number;
    triggered_by: string;        // GH username
    labels: string[];
  };
  phase: RunPhase;               // 'queued' | 'spec' | 'plan' | ... | 'ready_for_human'
  branch: string;                // feat/<topic-slug>
  pr_number: number | null;      // Set after `pr-review` phase
  policy_decisions: PolicyDecision[];
  llm_spend: { anthropic_usd: number; openai_usd: number; total_usd: number };
  audit_entries: AuditEntry[];
  created_at: string;            // ISO 8601
  updated_at: string;
  completed_at: string | null;
  error: { phase: RunPhase; message: string; recoverable: boolean } | null;
}
```

Stored in SQLite at `~/.claude-autopilot/runs/<repo-id>/state.db`.
Per-repo isolation; daemon never reads runs from a different repo.

## Error handling + testing

### Failure modes

| Failure | Detection | Recovery |
|---|---|---|
| LLM API outage | LLM call throws → state transitions to `error` with `recoverable: true` | Daemon retries every 60s up to budget cap; on resume, state machine continues from last good phase |
| GitHub API rate limit | gh CLI returns 429 | Exponential backoff (30s → 5min); audit logs the wait |
| Policy violation (path / size / budget) | Policy gate rejects pre-action | Run halts with `error.phase = <phase>, recoverable: false`; audit logs the decision; operator notified via PR comment |
| Repo divergence (force-push, branch deleted) | Daemon detects on next poll | Run marked `superseded`; audit entry; daemon does not retry |
| Local DB corruption | SQLite open fails | Daemon refuses to start; operator runs `autopilot recover <repo-id>` to reconstruct from `.autopilot/audit.log` (the source of truth IS the in-repo audit log; SQLite is a cache) |
| Emergency stop file appears | Polled every 1s | All in-flight runs halt within 1s; audit entry; new triggers ignored until file removed |

### Test strategy

* **Unit:** policy enforcement (~30 cases — path/branch/size/budget combinations).
* **Integration:** state-machine transitions with a fake GH poller.
* **End-to-end (local-only):** spawn a local daemon against a
  test repo (`tmpdir/test-repo/`), trigger via labeled issue,
  verify PR opens with audit comment.
* **Benchmark suite (codex N1):** at least 4 repo types — mature
  well-tested repo, under-tested legacy, blank repo, template-
  initialized. Track time-to-PR, time-to-merge, CI pass rate,
  human intervention count, cost.

## Migration / rollout

* v8.0.0-alpha: ship the daemon to a private testing channel; eat
  our own dogfood by running it against `axledbetter/claude-autopilot`
  itself for at least 50 PRs.
* v8.0.0-beta: open BYO-key CLI as public. Hosted tier waitlist.
* v8.0.0: public release. Hosted tier opens.
* claude-autopilot v7.x continues to ship as a Claude Code plugin
  for users who prefer the chat-driven loop.

## Stabilization criteria

v8.0.0 ships when:

1. Daemon has run 50+ self-eat PRs against
   `axledbetter/claude-autopilot` with zero policy violations and
   zero unintended writes.
2. The 4-repo benchmark suite (codex N1) passes — daemon ships
   working PRs on each repo type without operator intervention.
3. Emergency-stop tested under load (10+ in-flight runs halt
   within 1s of `.autopilot/STOP` appearing).
4. Hosted tier dashboard ships with: per-org spend cap,
   per-repo policy editor, audit log viewer, run timeline.
5. Pricing copy + privacy doc + landing page live at autopilot.dev.

## Out-of-scope / explicit follow-ups

* **Multi-language scaffold** (codex N1 + v7.2.0 deferred): per-stack
  scaffolders for Python / Rust / Go. v7.2.0 ships Node ESM only.
* **Brainstorming-skill autonomy.** v7.x brainstorming requires a
  human user-review gate. v8 daemon assumes the operator (or a
  TPM/EM in their org) writes the spec themselves and labels the
  issue. Fully-autonomous brainstorming is a v9+ research direction.
* **Repository understanding** (codex N2): templates are NOT a
  substitute for the daemon learning a repo's actual patterns
  over time. v8 ships static stack detection + auto-scaffolded
  CLAUDE.md (already in v7.1.7); v9 should add memory-driven
  pattern detection (read recent PRs, infer style guide).

## Open questions for next codex pass

(Save for the next discussion — these aren't blockers for the spec
shape, but should be locked before v8.0-alpha:)

1. SQLite vs append-only JSONL for state.db. SQLite gives queries
   for `autopilot status`; JSONL is simpler + the audit log already
   exists in append-only form. Probably SQLite for the cache, JSONL
   in the repo for the source of truth.
2. GitHub App permission scopes in detail. Read issues + PRs +
   contents + actions; write contents (push to feature branches)
   + PRs (open / comment); never write to default branch unless
   `merge_pr: true` is in policy.
3. How does the daemon survive a laptop reboot? Re-poll on
   startup; SQLite persists state; in-flight phase resumes from
   last audit entry. Document precise resume semantics in v8.0.1.
4. Hosted tier multi-tenancy. Per-org dedicated daemon process
   vs shared process? Shared simpler, dedicated safer. Probably
   dedicated — cost will inform pricing.

## Review request

Two specific places I want pushback before locking the spec:

1. **Policy file shape.** Is `.autopilot/policy.yaml` enough, or
   does the daemon need a separate `.autopilot/secrets.json`
   (mode 0600) for things that shouldn't be in git? Currently
   policy is git-tracked (so PRs to it go through normal review)
   but contains no secrets — LLM keys are in
   `~/.claude-autopilot/keys.json` instead. I think this is right
   but worth a sanity check.
2. **`merge_pr: false` default vs hosted-tier UX.** A hosted-tier
   user might want auto-merge as the default ("you're paying me to
   ship, just ship"). I think the right answer is that hosted
   tier ships with `merge_pr: false` AND a clearly-labeled "enable
   auto-merge for protected branches" toggle in the dashboard
   (with extra-strong warnings + audit). Confirm or push back.
