# v6.1 — Default-flip: engine ON, `--no-engine` removed

**Status:** spec drafted 2026-05-05 (placeholder)
**Origin:** Phase 8 of v6 ([#94](https://github.com/axledbetter/claude-autopilot/pull/94)) deferred the default-flip to a follow-up minor version so v6.0 could ship with a stabilization window.

## What we're building

A **single config behavior change**: flip the built-in default of `engine.enabled` from `false` to `true` and remove the `--no-engine` CLI flag + `CLAUDE_AUTOPILOT_ENGINE=off` escape hatch. After v6.1, the Run State Engine is the only mode.

This is a **minor version** because users who explicitly set `engine.enabled: false` keep their current behavior — the only thing that changes is the built-in default for users who haven't opted in.

## Why this matters

The engine adds persistent per-run state, resumable phases, hard budget caps, and a typed JSON event stream. v6.0 shipped it OFF by default to give users control over when their workflows pick up new behavior — important because the engine writes to `.guardrail-cache/runs/<ulid>/`, the budget cap can hard-stop a phase mid-pipeline, and the `--json` channel discipline tightens stdout/stderr semantics in ways CI consumers may not expect.

After a stabilization window we want the engine on for every user automatically. Resume-from-failed-phase and budget caps are the kind of feature that's most valuable when a user doesn't have to know to ask for them — they're saving you from a 2 a.m. emergency you didn't know was coming. v6.1 closes that gap.

`--no-engine` exists in v6.0.x for one reason: to give users who can't migrate immediately a single-version escape hatch. v7 removes it — keeping it forever forks the test surface and the docs surface.

## Stabilization criteria

The flip happens **only** when **all** of these are true:

1. **Time window:** at least **30 days** on master since the most recent v6.x engine bug fix.
2. **No outstanding HIGH bugbot findings** on any file under `src/core/run-state/`, `src/cli/runs.ts`, `src/cli/json-envelope.ts`, or `src/cli/json-mode.ts`. MEDIUM and LOW are acceptable but each one needs an explicit "won't fix / acknowledged" reason in the bugbot triage log.
3. **No open issues with the `engine` label** in the GitHub repo at the time of release.
4. **At least one self-eat run** has completed end-to-end with `engine.enabled: true` and exit code 0, against a non-trivial spec (the v6.1 PR itself counts if the spec is non-trivial).
5. **Cross-environment validation:** the live adapter cert suite (Phase 7) has had at least one successful nightly run with all three providers (Vercel + Fly + Render) reporting `passed` — this proves the engine + the deploy adapters don't drift apart.
6. **Migration guide validated** by at least one external user (a non-author) who walked through the v5.x → v6 opt-in flow without hitting an undocumented surprise. Capture the feedback in `docs/v6/migration-guide.md`'s "Troubleshooting" section.
7. **No regression in `npm test`** — the suite must be at the v6.1 baseline + any new tests for the flip itself.

The intent is **not** to wait for perfection — it's to wait until the engine has earned trust through real usage. If criteria 1-3 are met but 4-6 aren't, the flip waits. The default-flip is a one-way door (`--no-engine` going away in v7); we don't get to undo it cheaply.

## Scope of the v6.1 PR

A minor version, but a small one. Three changes:

1. **Flip the default** in the config-loading layer. The single line that says `enabled: false` becomes `enabled: true`. Existing users with explicit `engine.enabled: false` are unaffected.
2. **Remove `--no-engine`** from the CLI flag parser. Pass-through becomes a no-op with a deprecation warning routed through the `run.warning` event so CI consumers can detect it: `{event: 'run.warning', reason: 'flag-deprecated', flag: '--no-engine', message: '--no-engine is removed in v7; this is the last version that accepts it as a no-op.'}`. **Wait — read criterion below.**
3. **Update the migration guide and CHANGELOG** to reflect the new default and to mark `--no-engine` as the final-version escape hatch.

> **Important:** v6.1 should NOT remove `--no-engine`. v6.1 should only flip the default and start emitting the deprecation warning. **v7 is what removes the flag.** This gives users one minor version to migrate after the default flips. The migration guide promises this — don't break that promise.

Revised three-bullet scope:

1. Flip the built-in default to `enabled: true`.
2. Keep `--no-engine` as a working escape hatch in v6.1.x but emit `run.warning(reason: 'flag-deprecated', flag: '--no-engine')` on every invocation.
3. Update the migration guide + CHANGELOG to reflect the new default, the deprecation warning, and the planned v7 removal.

## Out of scope (deliberate)

- **Per-feature defaults** (e.g., flip `--json` channel discipline default) — different release.
- **Cross-machine state sync** — orthogonal product (deferred from v6).
- **Web UI for run state** — orthogonal product.
- **Schema bumps** to `state.json` / `events.ndjson` — keep the schemas stable across v6.x. Schema bumps wait for v7+.

## Test plan

- **Unit:** the config loader returns `enabled: true` when no `engine` block is present in `guardrail.config.yaml`.
- **Unit:** `--no-engine` still works (engine off) but emits a `run.warning` with `reason: 'flag-deprecated'`.
- **Integration:** an end-to-end self-eat run with no `engine` block in config completes successfully (engine on by default).
- **Migration guide:** a fresh `claude-autopilot init` writes a `guardrail.config.yaml` that does NOT include an explicit `engine.enabled: true` (default is implicit). Existing configs that have `engine.enabled: true` continue to work; existing configs that have `engine.enabled: false` keep the engine off.

## Failure modes the user should never have to debug

1. **Existing CI breaks because `.guardrail-cache/runs/` isn't gitignored.** The v6.0 release added the dir to `.gitignore` automatically via `claude-autopilot init`. v6.1 needs to verify the same — and emit a `run.warning(reason: 'gitignore-missing-runs-dir')` if `.guardrail-cache/runs/` isn't in `.gitignore` on first engine-on run.
2. **CI bills double because `.guardrail-cache/runs/` artifacts aren't sized-bounded.** v6 ships `runs gc` with a 30-day default. v6.1 should add a doctor check that warns when the runs dir exceeds 1 GB.
3. **A user pinned to v5.x output shape gets `--json` envelopes they didn't ask for.** This shouldn't happen — `--json` requires the explicit flag. But if a CI consumer parses stderr free-form and the engine starts emitting NDJSON on first engine-on run, the consumer breaks. Migration guide flagging is the answer; v6.1 PR should re-flag this in the CHANGELOG.

## Migration story for v6.0 users

Anyone on v6.0 who hasn't opted in:
- Install v6.1.
- The next CLI invocation creates `.guardrail-cache/runs/<ulid>/` and starts emitting events.
- Cost-per-run goes up by ~20-50ms per phase (state writes are negligible vs LLM calls).
- Resume / budget cap / runs CLI become available without further config.

Anyone on v6.0 who explicitly opted in via `engine.enabled: true`:
- Install v6.1. Behavior is identical to v6.0 + their explicit opt-in. Their config still works.

Anyone on v6.0 who explicitly opted out via `engine.enabled: false`:
- Install v6.1. Behavior is identical — engine stays off because their explicit `false` overrides the new built-in default.
- They get one warning telling them `--no-engine` and `engine.enabled: false` are both deprecated and removed in v7. They have one minor version to migrate.

Anyone on v5.x:
- The v6.0 → v6.1 migration is a no-op for them. They should read [`docs/v6/migration-guide.md`](../v6/migration-guide.md) before bumping major.

## Estimated effort

~3-4 hours of implementation. Real work is the doctor check + the warning routing, not the default flip itself.

## Open questions

1. **Should v6.1 also surface a `claude-autopilot doctor engine` subverb** that proactively warns about gitignore / runs dir size / migration guide compliance? Lean: yes, but as a follow-up PR — keep v6.1 small.
2. **Should `--no-engine` warn or hard-fail in v6.1?** Spec above says warn. Codex review is welcome on whether the warning should be on stderr text-mode (like a CLI deprecation) vs only via the `run.warning` event (forces CI consumers to read events). Lean: both — text-mode warning **plus** a `run.warning` event so CI consumers don't miss it.
3. **Calendar trigger or criteria-only trigger?** Spec above is criteria-only. If we add a hard "v6.1 ships by date X regardless," that risks shipping before the engine has earned trust. Lean: criteria-only.
