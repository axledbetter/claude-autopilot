## Scope (in)

**Code changes (CLI + apps/web):**

1. **Remove `--no-engine` flag and the engine-off code path.** The v7 product
   surface (apps/web ingest, dashboard, billing, SSO) requires engine state
   for cost reporting + audit trails. Per master spec table row 6:
   "drop `--no-engine`". After this PR, every wrapped CLI verb runs through
   `runPhaseWithLifecycle` engine-on; `runPhase` is the only path.

2. **Remove `ENGINE_DEFAULT_V6_0` and `ENGINE_DEFAULT_V6_1` exports** from
   `src/core/run-state/resolve-engine.ts`. With the engine-off path gone,
   the "default flip" is irrelevant. `resolveEngineEnabled()` returns
   `true` unconditionally (existing call sites stop reading
   `builtInDefault`); the function itself is preserved (callers remain
   source-compatible) but **direct imports of `ENGINE_DEFAULT_V6_0` /
   `ENGINE_DEFAULT_V6_1` are intentionally breaking** (codex pass-3
   NOTE #1). The migration guide documents replacing these constants
   with literal `true` for downstream consumers who imported them.

3. **Bump `RUN_STATE_SCHEMA_VERSION` from 1 to 2.** Signals the v7.0 cycle
   on every newly-written `state.json`. `RUN_STATE_MIN_SUPPORTED_SCHEMA_VERSION`
   stays at 1 so v6.x-written runs are still readable; the readable window
   is `[1, 2]`. v6→v7 reads succeed; v7→v6 reads fail with `corrupted_state`
   and a "downgrade not supported" hint.

4. **apps/web middleware: real-time membership revocation.** Currently
   `apps/web/middleware.ts` only refreshes the Supabase auth cookie.
   Extend it: on `/dashboard/**` and `/api/dashboard/**` requests, when
   the request resolves an `cao_active_org` cookie, look up
   `memberships.status` for `(active_org, auth.user)` and reject with
   302 → `/login?reason=member_disabled` (page request) or 403
   `member_disabled` (API request) if status is anything other than
   `'active'`. **Cache the result in a 60s server-set HttpOnly
   `cao_membership_check` cookie** keyed by `${active_org}:${userId}`
   so the middleware only hits the DB at most once per minute per user
   per request — bounded by Phase 5.7's documented "≤1h access-token
   expiry" window which this collapses to ≤60s.

## Architecture

**Engine-off removal — call-site survey:**

`runPhaseWithLifecycle` today branches on `engineEnabled` and calls
`opts.runEngineOff()` for the legacy path. Eight wrapped phases
(`scan`, `costs`, `fix`, `brainstorm`, `spec`, `plan`, `review`,
`validate`, `implement`, `migrate`, `pr`) supply that callback. After
this PR, the helper drops the engine-off branch and the callback —
phase bodies are always invoked through `runPhase`. The orchestrator
verb (`autopilot`) already required engine-on (per v6.2.0 design); the
single-verb dispatchers now match.

**`--no-engine` flag — survey:**

Defined in `src/cli/index.ts` `parseEngineFlags()` (~L283). Help text
mentions in `src/cli/help-text.ts` (12 occurrences across per-verb
sections). Per-verb doc comments in `cli/{costs,brainstorm,fix,plan,
implement,review,validate,scan,migrate,pr,spec}.ts`. Removed
everywhere; `parseEngineFlags()` shrinks to detect `--engine` (no-op
shim emitting a deprecation warning) and reject `--no-engine` with
`invalid_config` exit 1.

**Schema-version bump — read/write semantics:**

```
v6.x writes:                    schema_version: 1
v7.0   writes:                  schema_version: 2
v7.0   reads from disk:         range [1, 2] → success
v6.x   reads v7-written disk:   range [1, 1] → corrupted_state
```

`RUN_STATE_MIN_SUPPORTED_SCHEMA_VERSION` stays at `1` so an in-place
v6 → v7 upgrade preserves resume capability. `RUN_STATE_MAX_SUPPORTED_SCHEMA_VERSION`
follows `RUN_STATE_SCHEMA_VERSION` (existing pattern). A v6 binary
attempting to resume a v7-written run produces the existing
`corrupted_state` error with the `[1..1]` range in the message — clean
failure, not silent corruption.

**Middleware revocation — request flow:**

```
Browser → /dashboard/admin/members → middleware
  ├── refresh Supabase auth cookie (existing)
  ├── if path matches /dashboard, /dashboard/**, /api/dashboard, /api/dashboard/**:
  │   ├── read cao_active_org cookie (Phase 5.3)
  │   ├── if cookie absent or invalid:
  │   │   ├── /api/dashboard/**: 403 JSON {error:'no_active_org'}    (fail closed)
  │   │   └── /dashboard/**: pass through ONLY for whitelisted org-bootstrap
  │   │     paths (/dashboard/orgs/select, /dashboard/orgs/create);
  │   │     all others 302 → /dashboard/orgs/select
  │   ├── read cao_membership_check cookie (HMAC-signed; see below)
  │   │   ├── verifyHmac OK + matches (orgId, userId) + exp > now: pass
  │   │   └── verifyHmac fail / mismatch / expired: query check_membership_status RPC
  │   │       ├── status='active': mint new HMAC-signed cookie (60s exp); pass
  │   │       └── status≠'active' or no row: clear active-org + membership cookies
  │   │           ├── page request: 302 → /access-revoked?reason=...
  │   │           │   (NOT /login — /login auto-forwards authenticated users
  │   │           │   back to /dashboard, so revoked users would loop)
  │   │           └── API request: 403 JSON {error:'member_disabled'|'member_inactive'|'no_membership'}
  └── continue to route handler
```

**Cookie format (HMAC-signed, codex pass-1 CRITICAL #2; attributes per pass-2 WARNING #2):**

```
cao_membership_check = base64url(payload) + "." + base64url(hmacSha256(secret, payload))
payload = JSON {
  orgId:    UUID,
  userId:   UUID,
  status:   'active',
  role:     'owner'|'admin'|'member',
  exp:      epochSeconds,         // checked_at + 60
}

Set-Cookie attributes (every set):
  HttpOnly
  Secure                 (omitted only in NODE_ENV=test/development)
  SameSite=Lax
  Path=/
  Max-Age=60

Clear:
  Set-Cookie: cao_membership_check=; Max-Age=0; Path=/; HttpOnly; Secure; SameSite=Lax
```

The HMAC key is a new env var `MEMBERSHIP_CHECK_COOKIE_SECRET` (≥32
bytes, hex or utf8 — same module-load validation as
`SSO_STATE_SIGNING_SECRET`). Verification: parse → split on dot →
recompute HMAC → `crypto.timingSafeEqual` against signature → reject
if mismatch, expired, or `(orgId, userId)` don't match the request's
resolved `(active_org, auth.user.id)`. **An unsigned, forged, or
mismatched cookie is treated as cache miss** — falls through to the
RPC check. Forging a signed cookie requires the secret, so a disabled
user cannot bypass the 60s check window.

**Why a 60s cache:** without it, every dashboard navigation costs one
DB round-trip in the middleware (latency + load). With it, the worst-
case revocation window collapses from ~60min (current — until access
token expires) to ≤60s, while keeping middleware cost bounded.

**Why HttpOnly cookie (not in-memory or KV):** the autopilot.dev hosted
product runs on Vercel; serverless invocations are stateless
across instances. A cookie is the simplest stable per-user cache that
survives across regions without provisioning a Redis/KV. Trade-
off: 60s revocation window vs zero infra cost. HMAC signing closes the
forgery hole. (Note for reviewers: this is the autopilot.dev hosted
product on Vercel; delegance-app is the ECS deployment — separate
codebase. Codex passes 1/2/3 CRITICAL #1 was the recurring confusion.)

**Middleware runtime — explicitly Node.js (codex pass-3 CRITICAL #2):**
the middleware uses `node:crypto` for HMAC + `crypto.timingSafeEqual`,
which are NOT available in Vercel's Edge runtime. Set
`export const runtime = 'nodejs'` in `apps/web/middleware.ts` (Next.js
16 supports this; default is currently `'edge'`). The Node runtime
adds ~50-100ms cold-start vs Edge but unlocks `node:crypto`,
`Buffer`, and full Supabase service-role client compatibility.
Build-time test asserts the runtime export.

**Why role-hash is NOT used as a cache-bust signal:** an earlier draft
proposed including a hash of `(status, role)` in the cache key.
Middleware cannot know the *current* role without a DB call, so a
demoted user's cached "admin" cookie remains accepted until the 60s
TTL expires anyway. **Documented guarantee: status and role changes
are visible within ≤60s of the next request, not synchronously.**
Tighter enforcement (instant role change) would require server-side
cache invalidation on `change_member_role` — out of scope for v7.0.

**RPC contract (codex pass-1 NOTE #2: always returns one row; pass-3 WARNING #2: SECURITY INVOKER):**

```sql
CREATE OR REPLACE FUNCTION check_membership_status(
  p_org_id UUID,
  p_user_id UUID
) RETURNS JSON
LANGUAGE sql SECURITY INVOKER SET search_path = public, pg_temp
AS $$
  SELECT COALESCE(
    (SELECT json_build_object(
       'status', status,
       'role',   role,
       'checked_at', extract(epoch from now())::int
     )
     FROM memberships
     WHERE organization_id = p_org_id
       AND user_id = p_user_id
     LIMIT 1),
    json_build_object('status', 'no_row', 'role', NULL,
                      'checked_at', extract(epoch from now())::int)
  )
$$;

REVOKE ALL ON FUNCTION check_membership_status FROM PUBLIC, anon, authenticated;
GRANT EXECUTE ON FUNCTION check_membership_status TO service_role;
```

`'no_row'` is treated identically to `'disabled'`/`'inactive'` (revoke
+ redirect/403). The middleware never sees `null` — only
`{status: 'active'|'disabled'|'inactive'|'invite_pending'|'no_row', role, checked_at}`.

**Service-role guardrails (codex pass-1 WARNING #2):** the middleware
calls a new dedicated helper `lib/supabase/check-membership.ts`
exporting `checkMembershipStatus(orgId: string, userId: string):
Promise<{status, role}>`. The helper validates both args with
`isValidUuid()` (existing helper from
`lib/dashboard/membership-guard.ts`) BEFORE touching Supabase, calls
exactly one RPC (`check_membership_status`), and never accepts
arbitrary table names / filter strings. RPC errors are caught and
re-thrown as `MembershipCheckError` with a `code` field. Middleware
**distinguishes RPC errors from negative results** (codex pass-2
WARNING #4): a real `member_disabled` / `member_inactive` /
`no_membership` returns those codes; an actual RPC error (timeout,
network, missing migration) returns `{error: 'check_failed'}` (API)
or redirects to `/access-revoked?reason=check_failed` (page).

**Hard timeout (codex pass-3 WARNING #3):** the RPC call wraps in
`Promise.race([rpcCall, timeout(1500)])` — 1.5s upper bound. Timeout
emits `MembershipCheckError({code: 'check_failed', subcode: 'timeout'})`,
logged with the structured fields below. This bounds middleware
latency on every dashboard request: cache hit ~1ms, cache miss ~50-
200ms typical, ~1500ms worst case. Both
fail closed but the operator can grep logs for `check_failed` to
distinguish "Supabase outage" from "user actually disabled". Helper
emits a structured log line with `traceId, orgId, userId, code,
errorMessage` on every error path. This is the only service-role
surface in middleware; no other reads or writes are performed.

**Status → reason mapping (codex pass-3 WARNING #5; single source of truth):**

| RPC `status` | Page redirect reason | API JSON `error` | HTTP |
|---|---|---|---|
| `active` | (allow — no redirect) | (allow — no JSON) | pass |
| `disabled` | `member_disabled` | `member_disabled` | 302 / 403 |
| `inactive` | `member_inactive` | `member_inactive` | 302 / 403 |
| `invite_pending` | `member_inactive` | `member_inactive` | 302 / 403 |
| `no_row` (RPC returns synthetic) | `no_membership` | `no_membership` | 302 / 403 |
| (RPC error / timeout) | `check_failed` | `check_failed` | 302 / 403 |

The `/access-revoked` page renders one of `member_disabled`,
`member_inactive`, `no_membership`, `check_failed` (4 strings only).
This mapping is the single source of truth for tests, logs, and UI.

**Single-response composition (codex pass-2 WARNING #1; pass-3 WARNING #4):**
middleware must construct exactly one `NextResponse` and apply
Supabase auth-refresh cookie mutations + membership cookie set/clear
ops to that same response, regardless of pass / redirect / 403 path.
Cookie operations are recorded in a per-request `CookieMutationLog`
and applied to the chosen final response after the membership check
decides the response shape (see helper below). Pattern:

```typescript
let response = NextResponse.next();
const supabase = createMiddlewareClient(request, response);
const { data: { user } } = await supabase.auth.getUser();
// ... membership check decides the final response shape ...
if (revoked) {
  response = isApi
    ? NextResponse.json({error: code}, {status: 403})
    : NextResponse.redirect(new URL('/access-revoked?reason=' + code, request.url));
  // re-apply cookie operations onto the new response
  copyCookies(originalResponse, response);
}
return response;
```

Tests assert that BOTH the Supabase refresh cookies AND the membership
cookies appear on every terminal response shape (pass / redirect /
403 / cleared).

**`SECURITY INVOKER` for the RPC (codex pass-2 WARNING #5):** since
`service_role` already bypasses RLS, the function does not need
`SECURITY DEFINER`. Switching to `SECURITY INVOKER` reduces the blast
radius if grants are accidentally widened to `authenticated` later.
Migration test asserts `SECURITY INVOKER` and the only-`service_role`
grant.

**Whitelisted org-bootstrap routes — disabled-user audit (codex pass-2
WARNING #7):** `/dashboard/orgs/select` and `/dashboard/orgs/create`
must filter their org list using `listActiveOrgs(svc, userId)` (existing
Phase 5.3 helper). A revoked user landing on `/dashboard/orgs/select`
sees an empty list with a "no active organizations" message — they
cannot re-select into the org they were just disabled from because
the helper filters to `status='active'` rows. Regression test #5(j)
in the table below pins this.

**Route inventory — coverage of org-scoped surfaces (codex pass-3 WARNING #1):**

| Route prefix | Membership-revocation enforcement |
|---|---|
| `/dashboard/**`, `/api/dashboard/**` | NEW middleware check (this PR) — ≤60s revocation window |
| `/api/auth/sso/callback` | Phase 5.7 — `record_workos_sign_in` refuses disabled/inactive at sign-in |
| `/api/auth/callback` (Google) | Phase 5.6 — `enforceSsoRequired()` chokepoint |
| `/api/upload-session`, `/api/runs/:id/events/:seq`, `/api/runs/:id/finalize` | Phase 2.2 — signed-session JWT (NOT cookie); JWT bound to `(user, run, org)` at mint time. Membership not re-checked but JWT TTL ≤30min bounds blast radius. **Deferred to v7.1**: JWT mint also embeds `mint_membership_status` so finalize/event endpoints can refuse disabled members within JWT lifetime. |
| `/api/dashboard/runs/:id/{upload-session,artifact}` GET | Phase 5.8 — explicit `memberships.status='active'` filter when `r.organization_id IS NOT NULL`; covered already |
| `/api/stripe/webhook`, `/api/workos/webhook` | Webhook-only (HMAC-signed, no user session) |
| `/api/health`, `/api/health/deep` | Public; no auth |
| `/cli-auth` | Public landing page; mints API key via authenticated POST that re-validates session |
| `/runs/:runShareId` | Public via anon Supabase client; share-link visibility check happens in route handler |

The new middleware tightens the **session-cookie-authenticated** surface
(`/dashboard/**`, `/api/dashboard/**`). Other surfaces are either
JWT-authenticated (Phase 2.2 — bounded by ≤30min token TTL),
webhook-signed (no user identity), public (no enforcement needed),
or already explicitly filtered (Phase 5.7/5.8). The known gap is
the Phase 2.2 ingest API JWTs — listed as a v7.1 follow-up.

**`CLAUDE_AUTOPILOT_ENGINE=off` env handling (codex pass-2 WARNING #6):**
the env var is removed from precedence resolution at the same time as
`--no-engine`. Setting it to `off` produces a `run.warning` event with
code `engine_off_removed` and a one-line stderr deprecation banner;
**the env value is otherwise ignored** (engine remains on). This is
softer than `--no-engine` (which becomes a hard `invalid_config`
error) because env vars in CI are sticky and silently breaking every
v6.x → v7 upgrade in CI on day one would burn user trust. Test
#1(c) below pins.

**Schema-version downgrade error (codex pass-1 NOTE #1):** the existing
`corrupted_state` thrown when reading a v=2 file with a v6 binary now
includes the message `"state was written by a newer Autopilot version
(schema_version=2; this binary supports [1..1]); downgrade resume is
not supported"`. Test 3 below pins the exact substring.

**npm publish automation (codex pass-1 WARNING #3, moved into scope):**
`.github/workflows/publish.yml` (or equivalent) currently publishes on
every `v*` tag with `--tag next`. New behavior: tags matching
`v[0-9]+.[0-9]+.[0-9]+` (no `-pre`/`-alpha`/`-rc`) publish with
`--tag latest`; everything else stays `--tag next`. One-line bash
conditional in the workflow. Per codex pass-2 NOTE #2, the workflow
becomes the source of truth for npm dist-tags; `publishConfig.tag`
stays as a fallback only (left as `next` so a hand-run `npm publish`
without the workflow defaults to the safer prerelease channel).
