# v7.0 Phase 5.7 — Admin lifecycle controls + session revocation

**Status:** spec PR draft

**Builds on:** Phase 5.1 (`remove_member` RPC), Phase 5.4 (SSO disconnect), Phase 5.6 (sign-in flow + sso_authentication_states + workos_user_identities). Closes the lifecycle/revocation gap those phases explicitly deferred.

**Out-of-scope:** rate limiting on `/api/auth/sso/start` (deferred to Phase 6 — relies on platform-layer rate limit at Vercel edge, not DB-backed counter). Compliance audit log retention (deferred — separate spec).

## Goal

Three lifecycle controls so admins can actually run an SSO-protected org:

1. **Admin disable-user** — admin/owner can disable a member. **Refresh tokens are deleted immediately so the user cannot extend their session; the user's existing access tokens (Supabase JWTs) expire within ≤1h** (Supabase default). The disabled user cannot sign in again (SSO callback refuses, dashboard routes 404 because every page checks `status='active'`) until an admin explicitly re-enables them. (Codex pass-1 CRITICAL #1 — language updated to match reality. True real-time revocation requires middleware against a per-request denylist; that's deferred to Phase 6 when middleware ships.)
2. **SSO disconnect cascade** — when a Phase 5.4 webhook processes `connection.deleted` (or admin DELETEs the SSO connection), refresh tokens for all org members with verified-domain emails are revoked in one set-based DELETE. Existing access tokens still expire on their own ≤1h schedule; admin UI copy explicitly states this latency.
3. **Expired-state cleanup** — sso_authentication_states (consumed or expired > 24h) and processed_workos_events (processed > 30 days) hard-deleted by a `cleanup_expired_sso_states` RPC. **Phase 5.7 ships the RPC only** (codex pass-1 CRITICAL #3 — no HTTP route in this phase; Phase 6 wires a cron worker that calls the RPC directly via service-role client, no public endpoint).

## Why this scope

**Disable-user without session revocation is a critical security gap.** Phase 5.1's `remove_member` flips `memberships.status='inactive'` but the user's Supabase JWT keeps working until refresh-token expiry (default 1 week in Supabase). For an org that just learned an employee was compromised or terminated, "we removed them but they can still sign in for a week" is the wrong default.

**SSO disconnect cascade has the same shape.** Phase 5.4 disconnect sets `sso_connection_status='disabled'` but the existing user sessions stay valid; the next sign-in attempt would block (sso_required), but the existing session still works. For SSO admins, this is surprising and dangerous.

## Surfaces

### Backend routes

1. `POST /api/dashboard/orgs/:orgId/members/:userId/disable` — admin/owner only. Body none. Calls `disable_member` RPC. Returns updated membership row + count of sessions revoked + count of API keys revoked.
2. `POST /api/dashboard/orgs/:orgId/members/:userId/enable` — admin/owner only. Re-activates membership (status='active'). User must sign in again to get a fresh session. Note: re-enable does NOT restore previously-revoked API keys; user must mint new ones.
3. **No HTTP route for cleanup** (codex pass-1 CRITICAL #3). The `cleanup_expired_sso_states` RPC is callable via service-role client from a Phase 6 cron worker. Ops can run manually via `npx tsx scripts/cleanup-expired-sso-state.ts` (added in this phase).

(All routes use Phase 5.4's pattern: `assertSameOrigin` → `resolveSessionUserId` → admin route check → service-role RPC → `mapPostgresError`.)

### Schema delta

`data/deltas/20260509140000_phase5_7_lifecycle.sql`:

```sql
-- 1. disable_member RPC.
--
-- State transition matrix (codex pass-1 WARNING #5):
--   pending  → disable: 422 invalid_status_transition (use revoke invite)
--   active   → disable: succeeds → 'disabled', revoke sessions+API keys
--   inactive → disable: 422 invalid_status_transition (already inactive)
--   disabled → disable: noop (returns membership unchanged, NO duplicate
--                       audit, NO duplicate token revocation)
-- Reverse:
--   disabled → enable:  succeeds → 'active'
--   any other → enable: 422 invalid_status_transition
--
-- 'inactive' (Phase 5.1 remove) and 'disabled' (Phase 5.7 lockout) are
-- distinct: inactive requires re-invitation; disabled requires admin
-- re-enable. Codex pass-1 NOTE #1 — language updated.

-- Membership status enum extension + disabled_at/by metadata
-- (codex pass-1 NOTE #2 — capture lockout context).
ALTER TABLE public.memberships
  DROP CONSTRAINT IF EXISTS memberships_status_check,
  ADD CONSTRAINT memberships_status_check
    CHECK (status IN ('pending', 'active', 'inactive', 'disabled'));

ALTER TABLE public.memberships
  ADD COLUMN IF NOT EXISTS disabled_at TIMESTAMPTZ,
  ADD COLUMN IF NOT EXISTS disabled_by UUID REFERENCES auth.users(id);

-- 2. revoke_user_sessions — server-side helper, called by
-- disable_member + apply_workos_event (cascade on connection.deleted).
--
-- Uses raw DELETE on auth.refresh_tokens — Supabase exposes this table
-- to service_role. Couples to Supabase internals; documented here so
-- a future Supabase API change is a known maintenance hit.
--
-- Returns count of refresh tokens deleted. The user's existing
-- access_token JWTs remain valid until their `exp` claim (default 1h),
-- but cannot be refreshed → session terminates within ~1h max.
--
-- A "true real-time" revocation would require a JTI denylist + middleware
-- check on every request; that's a Phase 6 enhancement when we add
-- middleware. For 5.7, refresh-revocation gives ≤1h worst-case latency
-- which is the standard for OAuth-style sessions.
CREATE OR REPLACE FUNCTION public.revoke_user_sessions(
  p_user_id uuid
) RETURNS jsonb
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path = auth, public, pg_temp
AS $$
DECLARE
  v_count bigint;
BEGIN
  IF p_user_id IS NULL THEN
    RAISE EXCEPTION 'invalid_user_id' USING ERRCODE = 'P0001';
  END IF;

  -- Supabase's refresh-token table. DELETE forces re-auth on next refresh.
  WITH deleted AS (
    DELETE FROM auth.refresh_tokens
     WHERE user_id::uuid = p_user_id
     RETURNING id
  )
  SELECT count(*) INTO v_count FROM deleted;

  RETURN jsonb_build_object('revokedTokenCount', v_count);
END;
$$;

-- 3. disable_member.
CREATE OR REPLACE FUNCTION public.disable_member(
  p_caller_user_id uuid,
  p_org_id uuid,
  p_target_user_id uuid
) RETURNS jsonb
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path = public, audit, auth, pg_temp
AS $$
DECLARE
  v_caller_role text;
  v_target_role text;
  v_target_membership_id uuid;
  v_target_status text;
  v_revoke_result jsonb;
  v_api_key_count bigint;
BEGIN
  IF p_target_user_id = p_caller_user_id THEN
    RAISE EXCEPTION 'cannot_disable_self' USING ERRCODE = 'P0001';
  END IF;

  SELECT role INTO v_caller_role
    FROM public.memberships
   WHERE organization_id = p_org_id
     AND user_id = p_caller_user_id
     AND status = 'active';
  IF v_caller_role IS NULL OR v_caller_role NOT IN ('admin', 'owner') THEN
    RAISE EXCEPTION 'not_admin' USING ERRCODE = 'P0001';
  END IF;

  SELECT id, role, status
    INTO v_target_membership_id, v_target_role, v_target_status
    FROM public.memberships
   WHERE organization_id = p_org_id
     AND user_id = p_target_user_id
   FOR UPDATE;

  IF v_target_membership_id IS NULL THEN
    RAISE EXCEPTION 'target_not_member' USING ERRCODE = 'P0001';
  END IF;

  -- Codex pass-1 WARNING #5 — explicit transition matrix.
  IF v_target_status = 'disabled' THEN
    -- Idempotent noop, no duplicate audit, no duplicate revocation.
    RETURN jsonb_build_object(
      'membershipId', v_target_membership_id,
      'status', 'disabled',
      'noop', true
    );
  END IF;
  -- Codex pass-2 WARNING #2 — only 'active' can be disabled. 'pending'
  -- members go through the invite-revocation flow (separate from
  -- lockout); 'inactive' members are already removed.
  IF v_target_status <> 'active' THEN
    RAISE EXCEPTION 'invalid_status_transition' USING ERRCODE = 'P0001';
  END IF;

  -- Owner protection: only an owner can disable another owner; admins
  -- cannot disable owners.
  IF v_target_role = 'owner' AND v_caller_role <> 'owner' THEN
    RAISE EXCEPTION 'cannot_disable_owner' USING ERRCODE = 'P0001';
  END IF;

  -- Last-owner guard: org must have at least one active owner remaining.
  IF v_target_role = 'owner' THEN
    PERFORM 1 FROM public.memberships
     WHERE organization_id = p_org_id
       AND status = 'active'
       AND role = 'owner'
       AND user_id <> p_target_user_id
     LIMIT 1;
    IF NOT FOUND THEN
      RAISE EXCEPTION 'last_owner' USING ERRCODE = 'P0001';
    END IF;
  END IF;

  UPDATE public.memberships
     SET status = 'disabled',
         disabled_at = NOW(),
         disabled_by = p_caller_user_id
   WHERE id = v_target_membership_id;

  -- Revoke refresh tokens (codex pass-1 CRITICAL #1 latency caveat —
  -- access tokens stay valid until exp ≤1h).
  v_revoke_result := public.revoke_user_sessions(p_target_user_id);

  -- Codex pass-2 CRITICAL #1 — DROPPED global API-key revocation due
  -- to cross-tenant blast radius. API keys are user-scoped, so revoking
  -- in org A would also disable the user's keys in org B. Phase 5.8
  -- adds membership-active check in the API-key auth helper instead;
  -- once shipped, disabled members' keys auto-fail per-org.
  --
  -- Known 5.7 gap: a disabled member with a pre-existing API key can
  -- still call ingest endpoints scoped to this org until Phase 5.8.
  v_api_key_count := 0;

  PERFORM audit.append(
    p_org_id,
    p_caller_user_id,
    'org.member.disabled',
    'user',
    p_target_user_id::text,
    jsonb_build_object(
      'targetUserId', p_target_user_id,
      'previousRole', v_target_role,
      'previousStatus', v_target_status,
      'revokedTokenCount', v_revoke_result->'revokedTokenCount',
      'revokedApiKeyCount', v_api_key_count
    ),
    true
  );

  RETURN jsonb_build_object(
    'membershipId', v_target_membership_id,
    'status', 'disabled',
    'noop', false,
    'revokedTokenCount', v_revoke_result->'revokedTokenCount',
    'revokedApiKeyCount', v_api_key_count
  );
END;
$$;

-- 4. enable_member — re-activate (mirror of disable).
CREATE OR REPLACE FUNCTION public.enable_member(
  p_caller_user_id uuid,
  p_org_id uuid,
  p_target_user_id uuid
) RETURNS jsonb
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path = public, audit, auth, pg_temp
AS $$
DECLARE
  v_caller_role text;
  v_existing_status text;
  v_target_role text;
  v_membership_id uuid;
BEGIN
  SELECT role INTO v_caller_role
    FROM public.memberships
   WHERE organization_id = p_org_id
     AND user_id = p_caller_user_id
     AND status = 'active';
  IF v_caller_role IS NULL OR v_caller_role NOT IN ('admin', 'owner') THEN
    RAISE EXCEPTION 'not_admin' USING ERRCODE = 'P0001';
  END IF;

  SELECT id, status, role
    INTO v_membership_id, v_existing_status, v_target_role
    FROM public.memberships
   WHERE organization_id = p_org_id
     AND user_id = p_target_user_id
   FOR UPDATE;
  IF v_membership_id IS NULL THEN
    RAISE EXCEPTION 'target_not_member' USING ERRCODE = 'P0001';
  END IF;
  IF v_existing_status <> 'disabled' THEN
    -- Only disabled members can be re-enabled. inactive/pending have
    -- their own re-invitation flows.
    RAISE EXCEPTION 'invalid_status_transition' USING ERRCODE = 'P0001';
  END IF;

  -- Codex pass-2 WARNING #3 — symmetric owner protection. Only owners
  -- can re-enable a previously-disabled owner. Without this, an admin
  -- could escalate by re-enabling an owner-disabled-by-another-owner.
  IF v_target_role = 'owner' AND v_caller_role <> 'owner' THEN
    RAISE EXCEPTION 'cannot_enable_owner' USING ERRCODE = 'P0001';
  END IF;

  UPDATE public.memberships
     SET status = 'active',
         disabled_at = NULL,
         disabled_by = NULL
   WHERE id = v_membership_id;

  PERFORM audit.append(
    p_org_id,
    p_caller_user_id,
    'org.member.enabled',
    'user',
    p_target_user_id::text,
    jsonb_build_object(
      'targetUserId', p_target_user_id,
      'targetRole', v_target_role,
      'previousStatus', v_existing_status
    ),
    true
  );

  RETURN jsonb_build_object('membershipId', v_membership_id, 'status', 'active');
END;
$$;

-- 5. cleanup_expired_sso_states.
CREATE OR REPLACE FUNCTION public.cleanup_expired_sso_states(
  p_state_age_hours int DEFAULT 24,
  p_event_age_days int DEFAULT 30
) RETURNS jsonb
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path = public, pg_temp
AS $$
DECLARE
  v_state_count bigint;
  v_event_count bigint;
BEGIN
  IF p_state_age_hours < 1 OR p_state_age_hours > 720 THEN
    RAISE EXCEPTION 'invalid_state_age' USING ERRCODE = 'P0001';
  END IF;
  IF p_event_age_days < 1 OR p_event_age_days > 365 THEN
    RAISE EXCEPTION 'invalid_event_age' USING ERRCODE = 'P0001';
  END IF;

  WITH d1 AS (
    DELETE FROM public.sso_authentication_states
     WHERE (consumed_at IS NOT NULL AND consumed_at < NOW() - (p_state_age_hours || ' hours')::interval)
        OR (consumed_at IS NULL AND expires_at < NOW() - (p_state_age_hours || ' hours')::interval)
     RETURNING id
  )
  SELECT count(*) INTO v_state_count FROM d1;

  WITH d2 AS (
    DELETE FROM public.processed_workos_events
     WHERE processed_at IS NOT NULL
       AND processed_at < NOW() - (p_event_age_days || ' days')::interval
     RETURNING event_id
  )
  SELECT count(*) INTO v_event_count FROM d2;

  RETURN jsonb_build_object(
    'expiredStatesDeleted', v_state_count,
    'oldEventsDeleted', v_event_count
  );
END;
$$;

-- 6. Modify record_workos_sign_in (Phase 5.6) to refuse disabled members
--    (codex pass-1 WARNING #1).
--
-- This is a CREATE OR REPLACE that re-defines the function with the
-- same signature but adds a status check after the membership upsert.
-- The disabled-block must happen BEFORE returning a successful link so
-- the SSO callback gets `member_disabled` and refuses to mint a session.
--
-- Schema change: instead of "upsert membership if missing", the function
-- now also explicitly checks for an existing 'disabled' membership and
-- refuses with member_disabled. We DO NOT auto-flip disabled→active
-- on SSO sign-in.
CREATE OR REPLACE FUNCTION public.record_workos_sign_in(
  p_organization_id uuid,
  p_email text,
  p_normalized_email_domain text,
  p_workos_user_id text,
  p_workos_organization_id text,
  p_workos_connection_id text
) RETURNS jsonb
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path = public, audit, auth, pg_temp
AS $$
DECLARE
  v_settings_workos_org text;
  v_existing_link_user_id uuid;
  v_local_user_id uuid;
  v_existing_membership_status text;
  v_membership_created boolean := false;
  v_identity_created boolean := false;
BEGIN
  IF p_email IS NULL OR p_normalized_email_domain IS NULL OR length(trim(p_normalized_email_domain)) = 0 THEN
    RAISE EXCEPTION 'invalid_email' USING ERRCODE = 'P0001';
  END IF;

  -- Phase 5.6 spec-pass-1 CRITICAL #3 — verified-domain match required.
  PERFORM 1
    FROM public.organization_domain_claims
   WHERE organization_id = p_organization_id
     AND lower(domain) = lower(p_normalized_email_domain)
     AND status = 'verified';
  IF NOT FOUND THEN
    RAISE EXCEPTION 'email_domain_not_claimed_for_org' USING ERRCODE = 'P0001';
  END IF;

  -- Phase 5.6 — workos_org binding check.
  SELECT workos_organization_id INTO v_settings_workos_org
    FROM public.organization_settings
   WHERE organization_id = p_organization_id;
  IF v_settings_workos_org IS NULL OR v_settings_workos_org <> p_workos_organization_id THEN
    RAISE EXCEPTION 'unknown_org' USING ERRCODE = 'P0001';
  END IF;

  -- Identity-link path (Phase 5.6 spec-pass-1 WARNING #6).
  SELECT user_id INTO v_existing_link_user_id
    FROM public.workos_user_identities
   WHERE workos_user_id = p_workos_user_id
     AND workos_organization_id = p_workos_organization_id;

  IF v_existing_link_user_id IS NOT NULL THEN
    v_local_user_id := v_existing_link_user_id;
  ELSE
    SELECT id INTO v_local_user_id
      FROM auth.users
     WHERE lower(email) = lower(p_email)
     LIMIT 1;
    IF v_local_user_id IS NULL THEN
      RETURN jsonb_build_object(
        'result', 'user_not_provisioned',
        'email', p_email,
        'organizationId', p_organization_id
      );
    END IF;
    INSERT INTO public.workos_user_identities (
      user_id, workos_user_id, workos_organization_id, workos_connection_id, email_at_link
    ) VALUES (
      v_local_user_id, p_workos_user_id, p_workos_organization_id, p_workos_connection_id, lower(p_email)
    );
    v_identity_created := true;
  END IF;

  -- Phase 5.7 — disabled-membership check (codex pass-1 CRITICAL #2 + WARNING #1).
  -- If membership exists and is 'disabled', refuse the sign-in BEFORE
  -- the audit append + return. SSO callback handles 403 → /login/sso?reason=member_disabled.
  SELECT status INTO v_existing_membership_status
    FROM public.memberships
   WHERE organization_id = p_organization_id
     AND user_id = v_local_user_id;
  IF v_existing_membership_status = 'disabled' THEN
    RAISE EXCEPTION 'member_disabled' USING ERRCODE = 'P0001';
  END IF;
  -- Codex pass-2 WARNING #1 — explicit refusal for inactive (removed)
  -- and pending (invited but not accepted). Without this, sign-in
  -- "succeeds" but downstream RLS/dashboard guards quietly 404.
  IF v_existing_membership_status = 'inactive' THEN
    RAISE EXCEPTION 'member_inactive' USING ERRCODE = 'P0001';
  END IF;
  IF v_existing_membership_status = 'pending' THEN
    RAISE EXCEPTION 'invite_pending' USING ERRCODE = 'P0001';
  END IF;

  -- Upsert membership only if no row exists. Existing 'inactive' /
  -- 'pending' rows are caught above; the only path that reaches the
  -- INSERT is a fresh first-time SSO sign-in.
  IF v_existing_membership_status IS NULL THEN
    INSERT INTO public.memberships (
      organization_id, user_id, role, status, joined_at
    ) VALUES (
      p_organization_id, v_local_user_id, 'member', 'active', NOW()
    );
    v_membership_created := true;
  END IF;

  PERFORM audit.append(
    p_organization_id,
    v_local_user_id,
    'org.sso.user.signed_in',
    'user',
    v_local_user_id::text,
    jsonb_build_object(
      'email', p_email,
      'workosUserId', p_workos_user_id,
      'workosOrganizationId', p_workos_organization_id,
      'membershipCreated', v_membership_created,
      'identityCreated', v_identity_created
    ),
    true
  );

  RETURN jsonb_build_object(
    'result', 'linked',
    'userId', v_local_user_id,
    'membershipCreated', v_membership_created,
    'identityCreated', v_identity_created
  );
END;
$$;
```

### SSO disconnect cascade — modify `apply_workos_event`

When `apply_workos_event` (Phase 5.4) processes `connection.deleted`, after the existing settings update, also revoke refresh tokens for all members whose email domain is verified for that org. **Codex pass-1 WARNING #3 — set-based DELETE in one statement, not per-user loop. Audit append includes counts.**

```sql
-- INSIDE apply_workos_event, BEFORE the audit append on connection.deleted:
DECLARE
  v_revoked_user_count bigint;
  v_revoked_token_count bigint;
BEGIN
  IF p_event_type IN ('connection.deleted', 'dsync.connection.deleted') THEN
    WITH affected_users AS (
      SELECT DISTINCT m.user_id
        FROM public.memberships m
        JOIN auth.users u ON u.id = m.user_id
       WHERE m.organization_id = v_org_id
         AND m.status = 'active'
         AND lower(split_part(u.email, '@', 2)) IN (
           SELECT lower(domain)
             FROM public.organization_domain_claims
            WHERE organization_id = v_org_id
              AND status = 'verified'
         )
    ),
    deleted_tokens AS (
      DELETE FROM auth.refresh_tokens
       WHERE user_id::uuid IN (SELECT user_id FROM affected_users)
       RETURNING id, user_id
    )
    SELECT count(DISTINCT user_id), count(*)
      INTO v_revoked_user_count, v_revoked_token_count
      FROM deleted_tokens;
  ELSE
    v_revoked_user_count := 0;
    v_revoked_token_count := 0;
  END IF;
  -- Codex pass-2 WARNING #5 — capture sample of user IDs in audit
  -- (bounded to avoid jsonb bloat for large orgs).
  --   'cascadeRevokedUserCount', v_revoked_user_count,
  --   'cascadeRevokedTokenCount', v_revoked_token_count,
  --   'cascadeRevokedUserIdsSample', (SELECT jsonb_agg(user_id)
  --      FROM (SELECT DISTINCT user_id FROM affected_users LIMIT 50) s)
END;
```

## Risks + tradeoffs

- **Coupling to `auth.refresh_tokens`.** Documented in `revoke_user_sessions` body. Codex pass-1 WARNING #4 — runbook entry added: when Supabase Auth is upgraded, run the existing privilege grep tests + manually verify the table still exists with the same columns. The Supabase admin SDK does NOT expose a "revoke all sessions for user" call as of @supabase/supabase-js v2.97 (only per-JWT signOut), so direct `auth.refresh_tokens` deletion is the only available path. If Supabase ships an admin API for this, swap in the SDK call and remove the SQL coupling.
- **≤1h revocation latency** — access tokens stay valid until `exp` (Supabase default 1h). For real-time revocation, would need middleware check on every request against a server-side denylist. Out of scope for 5.7; documented as a known limitation in the audit metadata + UI copy. Goal text updated to reflect actual latency (codex pass-1 CRITICAL #1).
- **Cleanup RPC has no automation in 5.7.** Phase 6 wires a cron. For now, manually invocable via `npx tsx scripts/cleanup-expired-sso-state.ts` (service-role client).
- **autopilot.dev = Vercel deployment** (not delegance-app/ECS). Skip directives + future rate-limit choices reference Vercel.

### disable_member
1. Owner disables admin → 200, status=disabled, audit appended, refresh_tokens deleted.
2. Admin disables admin → 200.
3. Member tries → 403 not_admin.
