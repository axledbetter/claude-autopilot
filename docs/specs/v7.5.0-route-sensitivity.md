# v7.5.0 — route-sensitivity-tiered membership revocation

## Goal

Address W4 from the v7.4.1 codex strategic review without
infrastructure additions. The original framing ("collapse ≤60s
window to instant globally") requires new infra (Redis or
Realtime). The codex pass-1 on this work surfaced a better answer:

> "Split the policy by route/action sensitivity. Keep the 60s
> cookie for low-risk dashboard navigation, but require per-request
> `assertActiveMembership` or a server-side invalidation check for
> privileged mutations and sensitive reads."

That's exactly the right shape. Most dashboard traffic is
low-risk (rendering pages, listing runs); the mutations + sensitive
reads are a small subset where the per-request RPC cost is
acceptable.

Risk tier: **medium**. Per v7.4.2 policy → 2 codex passes
(this is pass-1 done; pass-2 happens after the code is written).

## Deployment context

`apps/web/` is the **autopilot.dev** Vercel deployment (NOT
delegance-app, which is the AWS ECS / BullMQ-Redis codebase that
codex passes have repeatedly conflated — 8th false positive). No
Redis available, no Realtime websocket. This rules out
infrastructure-dependent invalidation; the route-sensitivity-
tiered approach is the right serverless-native fit.

## Architecture

### Existing v7.0 Phase 6 cookie cache

`cao_membership_check` HMAC-signed cookie, 60s TTL, returned by
middleware on every dashboard request. Fast path: cookie hit →
no RPC. Slow path: cookie miss → RPC + new cookie minted.

This stays in place for **low-sensitivity routes**.

### New v7.5.0 — bypass-cookie list

Define a route-sensitivity classification:

| Sensitivity | Routes | Behavior |
|---|---|---|
| **Low** (read-only navigation) | `/dashboard`, `/dashboard/runs/*`, `/dashboard/billing` (read), all GET handlers under `/api/dashboard/runs/[runId]/artifact` | Existing 60s cookie cache. ≤60s revocation. |
| **High** (mutations + sensitive reads) | All POST/PATCH/DELETE under `/api/dashboard/orgs/*`, `/api/dashboard/api-keys/*`, GET `/api/dashboard/orgs/*/audit`, GET `/api/dashboard/orgs/*/cost*`, GET `/api/dashboard/orgs/*/sso/*` | Per-request `check_membership_status` RPC, ≤1 request revocation. |

Implementation: middleware checks the request path against a
declarative `HIGH_SENSITIVITY_PATTERNS` list. On match, **always**
runs the RPC (skips the cookie cache entirely). Cookie still gets
minted/refreshed for the user's other low-sensitivity tabs.

### Why route-sensitivity

The ≤60s cache exists because dashboard navigation is bursty — a
single page load can fan out to 10+ API calls in 100ms. RPC-on-
every-call would add ~20ms × 10 = 200ms to every navigation. For
mutations the same overhead is invisible (mutation already takes
hundreds of ms server-side). For sensitive reads (audit logs,
cost reports, SSO config) the user expects accuracy over speed.

This makes the trade-off explicit per route, instead of accepting
≤60s globally.

## Components

### `apps/web/lib/middleware/route-sensitivity.ts` (new)

```ts
const HIGH_SENSITIVITY_PATTERNS: RegExp[] = [
  // Org admin mutations (POST/PATCH/DELETE)
  /^\/api\/dashboard\/orgs\/[^/]+\/members(\/|$)/,
  /^\/api\/dashboard\/orgs\/[^/]+\/sso(\/|$)/,
  /^\/api\/dashboard\/orgs\/[^/]+\/(disable|enable|update)$/,
  // API key mutations
  /^\/api\/dashboard\/api-keys\//,
  // Sensitive reads (audit, cost, billing settings)
  /^\/api\/dashboard\/orgs\/[^/]+\/audit/,
  /^\/api\/dashboard\/orgs\/[^/]+\/cost/,
  /^\/api\/dashboard\/orgs\/[^/]+\/billing\//,
];

export function isHighSensitivityRoute(pathname: string, method: string): boolean {
  // All non-GET methods on org-scoped routes are high-sensitivity by default.
  if (method !== 'GET' && pathname.startsWith('/api/dashboard/')) return true;
  // GET routes only match the explicit pattern list.
  return HIGH_SENSITIVITY_PATTERNS.some(re => re.test(pathname));
}
```

### `apps/web/middleware.ts` (modified)

Insert sensitivity check BEFORE the cookie verify path:

```ts
const sensitivity = isHighSensitivityRoute(req.nextUrl.pathname, req.method);

if (sensitivity) {
  // Skip cookie cache entirely; always RPC.
  const membership = await checkMembershipStatus(activeOrg, userId);
  if (membership.status !== 'active') {
    return buildRevokeResponse(membership.status, req);
  }
  // Continue WITHOUT minting a new cookie (the cookie path is for
  // low-sensitivity caching only).
} else {
  // Existing v7.0 Phase 6 cookie path: verify HMAC, check exp, fall
  // through to RPC on miss.
  const cookieResult = verifyMembershipCookie(req.cookies.get(...));
  if (cookieResult.kind === 'hit' && /* not expired */) {
    return NextResponse.next();
  }
  // Fall through to RPC + mint new cookie.
}
```

### Policy: which routes are "high"?

Locked in code (`HIGH_SENSITIVITY_PATTERNS` array), NOT
operator-configurable. Reasoning: misconfiguring this is a security
hole that's hard to detect; the surface is small enough to enumerate
in source. Adding a new high-sensitivity route = add a regex + ship
a PR.

  or autopilot.dev gains real traffic).

## Why this approach over the alternatives

* **Vercel KV** (~2hr ship if/when needed): adds infra, requires
  KV env config. Right answer when there's a security/compliance
  conversation that needs sub-second invalidation everywhere.
  Not the right answer when no real customers exist yet.
* **Postgres LISTEN/NOTIFY**: doesn't fit serverless model.
* **Module-level pub/sub**: cross-instance staleness; doesn't help.
* **Route-sensitivity tiers (this PR)**: zero new infra; explicit
  per-route trade-off; ≤1-request revocation on the routes that
  ACTUALLY matter for security/compliance. The codex pass on the
  prior spec attempt explicitly recommended this shape.
