# v6.2 — Multi-phase orchestrator (`claude-autopilot autopilot`)

**Status:** spec drafted 2026-05-07
**Origin:** v6.0.x wrapped each pipeline phase under its own runId. v6.1 flipped the engine on by default. The next move is **one runId across an entire pipeline** — the experience the YC demo promises ("watch your $25 budget tick down as autopilot ships a PR") needs a single run to tick down.

## What we're building

A new top-level CLI verb, `claude-autopilot autopilot [options]`, that runs the existing pipeline phases sequentially **under one engine run**. It's the multi-phase analogue of `runPhaseWithLifecycle`: create the run once, iterate over phases with monotonic `phaseIdx`, emit `run.complete` exactly once at the end.

```
$ claude-autopilot autopilot
[autopilot] runId=01HQK8...  budget=$25.00
[autopilot] phase 1/4: scan
[autopilot] phase 2/4: spec
[autopilot] phase 3/4: plan
[autopilot] phase 4/4: implement
[autopilot] ✔ run complete in 4m12s, $3.47 of $25.00
```

`runs watch <runId>` already exists (v6.1) — it tails events.ndjson with a live cost meter. The orchestrator's events stream into the same file under one runId, so `runs watch` becomes the live demo for the full pipeline.

## Why this matters

**For users:** today's pipeline is a chain of separate `claude-autopilot scan && claude-autopilot spec && claude-autopilot plan && ...` invocations. Each invocation creates its own runId, so resume / budget / `runs watch` work *per phase* but not *per pipeline*. A user who hit a budget cap on phase 3 has no easy way to inspect total spend across phases 1+2+3. A failure mid-pipeline leaves three orphan runs with no parent. The autopilot verb fixes this with one runId.

**For YC demo:** the pitch is "watch a $25 budget tick down as autopilot ships a PR." That demo needs a single `runs watch` window showing the full pipeline. Today you'd have to chain four windows.

**For resume:** v6 phase 6 wired idempotency + readback per phase. The orchestrator inherits this for free — a failed pipeline at phase 3 of 4 can resume at phase 3 with phases 1+2 short-circuiting via their `phase.success` events, all in the same run dir.

## Phase ordering (default)

Sequential, with three branches based on what the user wants:

| Mode | Phases (in order) | Default? |
|---|---|---|
| `--mode=full` | scan → spec → plan → implement → migrate → pr | yes |
| `--mode=fix` | scan → fix → pr | no |
| `--mode=review` | scan → review | no |
| `--phases=<csv>` | user-supplied list, validated against the registry | no |

`--mode` is sugar over `--phases`. The full list is the YC-demo flow. `--mode=fix` is the "find a bug + fix it + ship the PR" flow we've been calling out as the differentiator vs Cursor / Copilot.

`brainstorm`, `costs`, `validate` are intentionally out of the default `--mode=full` list — they're advisory / read-only verbs, not pipeline steps. Users can include them via `--phases` if they want.

## Architecture

### One run, many phases

`createRun({ phases: [allPhaseNames] })` already accepts a phase array — Phase 1 of v6 designed for this from day one. The orchestrator passes the full ordered list at run creation. Each subsequent `runPhase` call uses the matching `phaseIdx` from that array.

### Where each phase's `RunPhase<I, O>` definition lives

Today every wrapped CLI verb (`scan`, `spec`, ...) builds its `RunPhase<I, O>` *inside* its `runX(options)` entry function — opaque to other modules. The orchestrator can't call `runScan(options)` because that would invoke `runPhaseWithLifecycle` and create a *second* run.

**Refactor required:** each verb gets a new exported builder named `build<Phase>Phase(deps)` that returns `{ phase: RunPhase<I, O>; input: I; renderResult: (output: O) => number }`. The existing `runX(options)` entry calls the new builder internally — no behavior change for direct CLI users. The orchestrator imports the builder, runs `runPhase` itself with the right `phaseIdx`, and prints the per-phase banner.

11 verbs need this surgery (`scan`, `costs`, `fix`, `brainstorm`, `spec`, `plan`, `review`, `validate`, `implement`, `migrate`, `pr`). Mechanical — every verb already constructs the same shape inline. Estimated ~30 lines moved per verb, no logic changes.

### Phase registry

A new module, `src/core/run-state/phase-registry.ts`, exports a typed map. Registry uses `as const` + a generic helper to preserve per-phase I/O types (per codex NOTE #5) — `Record<PhaseName, PhaseRegistration<unknown, unknown>>` is the wrong shape because it discards the per-phase builder signature:

```typescript
type PhaseRegistration<I, O> = {
  build: (deps: PhaseBuildDeps) => { phase: RunPhase<I, O>; input: I; renderResult: (output: O) => number };
  displayName: string;
};

export const PHASE_REGISTRY = {
  scan: { build: buildScanPhase, displayName: 'Scan' } satisfies PhaseRegistration<ScanInput, ScanOutput>,
  spec: { build: buildSpecPhase, displayName: 'Spec' } satisfies PhaseRegistration<SpecInput, SpecOutput>,
  // ... 9 more
} as const;

export type PhaseName = keyof typeof PHASE_REGISTRY;
export type PhaseInput<N extends PhaseName> = Parameters<typeof PHASE_REGISTRY[N]['build']>[0] extends never ? never : ReturnType<typeof PHASE_REGISTRY[N]['build']>['input'];
```

The orchestrator reads from this map. New phases register here. The `runs` CLI uses the same map for its phase-name validation.

### Phase input wiring

Each phase reads its inputs from `.guardrail-cache/` (where prior phases wrote them) — that's already true today, just intra-process via filesystem. The orchestrator does *not* thread typed outputs into typed inputs. Every phase's input comes from its `build<Phase>Phase()` builder, which discovers state from the run cwd the same way the standalone CLI verb does. **This is a deliberate choice:** keeping the inter-phase contract on the filesystem (instead of in TypeScript types) means partial pipelines, resume, and external tooling all work without coordination.

The orchestrator does pass two cross-cutting things:
1. **The run dir** — phases that want to write engine artifacts know where to land them.
2. **A budget config** — a single `BudgetConfig` covers the whole pipeline; per-phase `phase.cost` events accumulate against it.

### Failure semantics + exit codes

If phase N throws:
1. `phase.failed` is emitted by `runPhase` (existing behavior).
2. Orchestrator catches, emits `run.complete` with `status: 'failed'`, `failedAtPhase: N`, `failedPhaseName: <name>`.
3. Refresh state.json snapshot.
4. Print stderr banner: `[autopilot] failed at phase N/M (<name>): <message>`.
5. Exit per the matrix below.

**Exit code matrix** (per codex WARNING #3 — must be deterministic for CI/scripts):

| Failure source | Exit code |
|---|---|
| Pre-run validation (`--phases` invalid, `--mode` unknown, engine disabled) | `1` (`invalid_config`) |
| Phase failure where `phase.failed.errorCode` is `budget_exceeded` | `78` (matches v6.1 budget exit code) |
| Phase failure where `phase.failed.errorCode` is `lock_held` / `corrupted_state` / `partial_write` | `2` (engine error) |
| Any other phase failure (LLM error, network, etc.) | `1` (generic phase failure) |
| Orchestrator-level failure AFTER phase N succeeded (state.json write, lock release) | `1`, but `run.complete` already shows `status: 'success'` for phase N |

Precedence: if both a phase failure AND a finalization error occur, the **phase failure wins** (the phase's exit code is returned). The finalization error is recorded as `run.warning(reason: 'finalization-error')` so observers can find it.

No automatic retry. `claude-autopilot run resume <runId>` already exists (v6 Phase 3) — the user runs that to retry from the failed phase.

### Resume semantics

`run resume` already inspects events.ndjson, finds the last successful phase, and replays. With the orchestrator emitting one runId across N phases, resume works the same way: phases 1..K-1 short-circuit via their `phase.success` events, phase K retries, phases K+1..N run fresh. **No new resume code needed** — the orchestrator inherits this from `runPhase`'s existing replay machinery.

The one nuance: `run resume` today picks "the last failed phaseIdx" and resumes from there. Multi-phase runs make this richer — a resume from phaseIdx=3 implies phases 0..2 are short-circuit-replays. Each replay is a new `phase.start` with `attempts++` and the idempotent / side-effect gating decides whether to actually re-execute or skip.

### Budget enforcement

Today each wrapped phase can accept a `BudgetConfig`. Today it's per-phase — phase A can't see phase B's spend. The orchestrator passes ONE `BudgetConfig` to every phase in sequence; the budget consults `sumPhaseCost(events, '*')` (cross-phase) instead of per-phase. This needs a small change to `checkPhaseBudget`: an optional `scope: 'phase' | 'run'` (default 'phase' for back-compat).

`run.complete` gains a `totalCostUSD` field summing across all phases.

### `--json` mode

The orchestrator emits one outer envelope on stdout when `--json` is set:

```json
{
  "version": "1",
  "verb": "autopilot",
  "runId": "01HQK8...",
  "phases": [
    {"name": "scan", "status": "success", "costUSD": 0.42, "durationMs": 12340},
    {"name": "spec", "status": "success", "costUSD": 1.10, "durationMs": 18200},
    {"name": "plan", "status": "success", "costUSD": 0.85, "durationMs": 14500},
    {"name": "implement", "status": "failed", "errorCode": "budget_exceeded", "costUSD": 1.10, "durationMs": 22000}
  ],
  "totalCostUSD": 3.47,
  "exitCode": 1
}
```

Per-phase NDJSON events go to stderr (existing channel discipline from v6 Phase 5). Sub-phases don't emit their own JSON envelopes when running under the orchestrator — the orchestrator owns the stdout channel.

## What gets built (PR breakdown)

A single PR is too large; spec ships as 4 PRs.

### v6.2.0 — Phase registry + scan/spec/plan/implement under one runId + run-scope budget
Extract `build<Phase>Phase()` for the four core read-only-feeding-into-implement phases. New `phase-registry.ts`. New `src/cli/autopilot.ts` orchestrator that drives them. **Run-scope budget lands in v6.2.0** (not deferred to v6.2.2 per codex review WARNING #2): `checkPhaseBudget` gains `scope: 'phase' | 'run'` — orchestrator passes `scope: 'run'`, per-phase callers keep `scope: 'phase'` default. The YC-demo budget narrative ("$25 across the pipeline") works on day one. New tests for the registry + orchestrator (4-phase happy path + 1-phase-fail + resume + run-scope budget cap).

**Builder-extraction parity tests required** (per codex WARNING #4): each of the 4 extracted verbs gets a `tests/cli/<verb>-builder-parity.test.ts` asserting that `runX(options)` and `buildXPhase + runPhaseWithLifecycle` produce identical (a) stdout output, (b) stderr output, (c) events.ndjson event sequence — for the same inputs. No drift is acceptable.

### v6.2.1 — Side-effecting phases (migrate + pr) with explicit idempotency contracts
Add migrate + pr to the registry — but ONLY after each phase publishes an explicit idempotency contract (per codex CRITICAL):

- **`migrate`** must persist a per-migration-version externalRef BEFORE invoking the migration runner. Resume preflight: query `migration_state` (Supabase) / equivalent (other adapters) to confirm `applied_at IS NOT NULL`. If yes → emit `phase.success` with `replayed: true`. If no but externalRef exists → resume retries the migration (it didn't actually apply last time). Documented in `migrate.ts` doc comment + spec.
- **`pr`** must record the `github-pr` externalRef BEFORE the `gh pr create` call. Resume preflight: `gh pr view <number>` — if PR exists and is open → short-circuit to `phase.success`. If closed/merged → emit `run.warning(reason: 'side-effect-already-applied')` and short-circuit. If 404 → retry. Same shape as v6 Phase 6 readback (provider-readback.ts) — wire it through.

`migrate` and `pr` cannot land in the orchestrator until both contracts are implemented + tested. v6.2.1 is gated on this.

### v6.2.2 — `--json` envelope with deterministic failure shape + docs
Outer JSON envelope (per codex NOTE #6): "if `--json` is set, stdout emits exactly one valid envelope including for pre-run failures (config validation, registry errors, uncaught exceptions)." Pre-run failure envelope:

```json
{"version": "1", "verb": "autopilot", "runId": null, "status": "failed", "errorCode": "invalid_config", "errorMessage": "..."}
```

Migration guide section "v6.1 → v6.2: one runId across the pipeline." `runs watch <id>` doc page gains the demo transcript showing the full 4-phase tick-down. Cache contract version marker added to `.guardrail-cache/runs/<ulid>/state.json` (`schemaVersion` already exists; document its bump policy + add explicit "older runs are not resumable across major schema bumps" check).

## Stabilization / scope guardrails

- **No new phases.** This spec wires existing phases together. Adding new phases is out-of-scope for v6.2.
- **No parallel phases.** Phases run sequentially. Parallel scan + lint, etc., is out-of-scope.
- **No interactive prompts inside the orchestrator.** Phases that need them (`fix` has readline) keep their own prompt; the orchestrator doesn't re-prompt at the run level.
- **Engine-off:** the orchestrator REQUIRES engine-on. v6.1 made engine-on the default; the orchestrator throws `invalid_config` exit 1 if the user explicitly disables the engine via `--no-engine`. Documented in the migration guide.

## Open questions

1. Should `--mode=fix` auto-include `validate` between `fix` and `pr`? (Pro: catches obviously-broken patches. Con: another LLM call, doubles cost on already-cheap fix runs.)
2. Should resume cross-phase support `--from-phase <name>` in addition to "from last failure"? (v6 Phase 3 has the lookup-only `run resume --from-phase`; orchestrator could make it actionable.)

## Out of scope (deliberate)

- Streaming progress to a TUI (reserved for v6.3 — `claude-autopilot tui`).
- Per-phase retry budgets distinct from the run budget.
- Cancellation. Today Ctrl-C kills the process; the run dir is left behind for `run resume`. Graceful cancellation is reserved for v6.3.
- Web dashboard. v6.x is CLI-only; the self-service product wires this for v7.
