# v7.0 Phase 5.3 — Org switcher

**Status:** ready for impl
**Repo:** `claude-autopilot` (Vercel)
**Builds on:** Phase 5.1 (memberships), Phase 5.2 (admin pages)
**Splits:** Phase 5.4 will be WorkOS SSO. Phase 5 is too large for a single PR.

## Goal

Replace the "first admin/owner membership" hack across `/dashboard` + `/dashboard/admin/*` with a real org switcher. Caller picks active org from a header dropdown; choice persists per-session via HTTP-only cookie. All page-level membership lookups use the active org.

## Why now

- Phase 5.1 + 5.2 hardcode `memberships[0]` in 5 places (`/dashboard/page.tsx`, `/dashboard/billing/page.tsx`, `/dashboard/layout.tsx`, `/dashboard/admin/layout.tsx`, `/dashboard/admin/members/page.tsx`). Users with multi-org membership see only their first org.
- Phase 5.4 (WorkOS SSO) needs an explicit "active org" concept to know which org's connection to use during sign-in.
- Org switcher is small and cosmetic compared to SSO — good 5.3 boundary.

## Non-goals

- WorkOS SSO (Phase 5.4)
- Personal vs org context split (the dashboard handles both today; switcher just picks among orgs)
- Org create UI (still server-side only via Phase 1 RPCs)
- Cross-org bulk operations
- Multi-org dashboard view ("show all orgs at once")

## Surfaces

### Backend
- `POST /api/dashboard/active-org` — sets the cookie. Body `{ orgId: string | null }`. Validates caller is active member of orgId; clears cookie if `null`. Returns 200 / 403 / 422 / 404.
- New helper `apps/web/lib/dashboard/active-org.ts` — `resolveActiveOrgId(supabase, userId)` reads cookie, validates against memberships, falls back to first active org if cookie missing/stale.

### Frontend
- `<OrgSwitcher>` (client) — dropdown in dashboard header. Shows org names. Calling `setActive(orgId)` POSTs to the API + reloads.
- `apps/web/app/dashboard/layout.tsx` — adds the switcher above the sidebar.
- All pages that previously used `memberships[0]` now call `resolveActiveOrgId()`.

## Cookie contract

- Name: `cao_active_org` (same prefix as `cao_*` Phase 4 cookies)
- Value: UUID (or empty string to clear)
- Attributes: `HttpOnly; Secure; SameSite=Lax; Path=/; Max-Age=1209600` (14 days)
- Validation: every read checks the value against `memberships` for the caller; if missing/stale → fallback to first active org membership; if no active membership at all → return null

## Authorization

POST validates:
1. assertSameOrigin (cookie-auth POST → CSRF concern)
2. resolveSessionUserId; 401 if not signed in
3. UUID validation on body.orgId
4. caller is active member of orgId via direct memberships query (no RPC; this is just a write to a cookie)
5. write cookie + 200

If body.orgId is null → clear cookie (200).

## Test plan (~10 tests)

### Backend (api/dashboard/active-org/route.ts)

1. POST { orgId } as active member → 200 + cookie set
2. POST { orgId } as non-member → 404 (no enumeration)
3. POST { orgId: null } → 200 + cookie cleared (Max-Age=0)
4. POST without session → 401
5. POST with malformed UUID → 422
6. POST with mismatched origin → 403

### Helper (lib/dashboard/active-org.ts)

7. resolveActiveOrgId returns cookie value when valid
8. resolveActiveOrgId falls back to first membership when cookie missing
9. resolveActiveOrgId falls back to first membership when cookie value isn't a current active membership
10. resolveActiveOrgId returns null when caller has no active memberships

## Codex passes folded inline

Anticipated:
- **CRITICAL: cookie-auth CSRF.** assertSameOrigin must run BEFORE any DB lookup. Folded.
- **CRITICAL: stale-cookie enumeration.** If a removed member's cookie still has the orgId, must NOT pass the membership check. Folded as Test 9.
- **WARNING: cookie-name collision.** Use `cao_active_org` to match existing `cao_*` namespace. Folded.
- **WARNING: SameSite=Strict vs Lax.** Lax is correct for this — POST happens from same-site dashboard, GET continues to work cross-site if user clicks back from email links. Folded.
