# v7.0 Phase 5.1 — Members management + RBAC enforcement

**Status:** ready for plan
**Repo:** `claude-autopilot` (NOT delegance-app)
**Deploys to:** Vercel
**Builds on:** Phase 1 (`memberships` schema), Phase 4 (dashboard shell)
**Gates:** Phase 5.2 (audit viewer), Phase 5.3 (WorkOS SSO)

## Goal

Org admins can invite + remove members and change roles inside `/dashboard/admin/members`. Owner-only actions (transfer ownership, demote owners) are gated. Cross-org enumeration is blocked. The page also surfaces basic org metadata (name) editable by the owner.

This is the **first user-visible Org tier surface** — Phase 4 was solo-user only. After 5.1 ships, an Org-tier (small/mid via Stripe Phase 3) admin can actually manage their team.

## Non-goals

- **Tokenized invitation links** (invite-by-email-of-non-user). Deferred to a later phase. 5.1's invite path requires the invitee to have already signed in to `autopilot.dev` via Google.
- **WorkOS SSO** (Phase 5.3).
- **Audit log viewer** (Phase 5.2) — the audit_events writes happen here, but the viewer UI is separate.
- **Cost report export** (Phase 5.2).
- **Soft-delete recovery** (admin un-removes a removed member). Removed members stay removed; re-invite if needed.
- **Email notifications** on invite/remove. Pure UI flow for now.

## Surfaces

### Backend (5 routes, all under `/api/dashboard/orgs/:orgId/...`)

| Method | Path | Auth | Role | Returns |
|---|---|---|---|---|
| GET | `/members` | session cookie | active member of orgId | `{ members: [{ userId, email, role, status, joinedAt }] }` |
| POST | `/members/invite` | session cookie + assertSameOrigin | admin OR owner | 200 `{ membership }` / 404 `user not found` / 409 `already member` |
| PATCH | `/members/:userId` | session cookie + assertSameOrigin | admin (member↔admin) OR owner (any role transition) | 200 `{ membership }` / 422 if last-owner demoted |
| DELETE | `/members/:userId` | session cookie + assertSameOrigin | admin (remove member) OR owner (remove admin/owner) | 200 / 422 if last owner |
| PATCH | `/` (org root) | session cookie + assertSameOrigin | owner | 200 `{ organization }` |

### Frontend (3 pages + 4 components)

- `/dashboard/admin/members` — server component, role-gated to admin/owner. Lists members, has invite form + role dropdown + remove button per row.
- `/dashboard/admin/settings` — server component, owner-only. Edits org name. (Placeholder for SSO/billing-tab in 5.2/5.3.)
- `/dashboard/admin/layout.tsx` — sidebar shows Members / Settings / (Audit, Cost report later); 404s if user has no admin/owner membership in any org.

Components:
- `<MembersList>` (client) — table with invite form, role dropdown per row, remove button.
- `<InviteMemberForm>` (client) — email input + role select.
- `<OrgSettingsForm>` (client) — org name input.
- `<RoleSelector>` (client) — dropdown that disables transitions the caller can't perform (e.g. member can never see "owner" option).

## Authorization model

### Role transition matrix (PATCH role)

Caller role | Target → member | → admin | → owner
:--|:-:|:-:|:-:
member | (no admin access) | — | —
admin | ✓ | ✓ | ✗
owner | ✓ | ✓ | ✓

**Codex pass 2: matrix locked. Admin can promote/demote between member↔admin including demoting another admin to member or promoting another member to admin. Admin cannot touch owner role. Owner can do anything subject to last-owner safeguard.**

### Removal matrix (DELETE)

Removal is more destructive than role change. Admin can only remove members; owner can remove anyone subject to last-owner.

Caller role | Remove member | Remove admin | Remove owner
:--|:-:|:-:|:-:
admin | ✓ | ✗ (`not_owner`) | ✗ (`not_owner`)
owner | ✓ | ✓ | ✓ (subject to `last_owner`)

**Code path:** all writes go through the SECURITY DEFINER RPCs in the migration delta. Route handlers do NOT issue direct service-role `update()` calls.

### Last-owner safeguard (atomic via SECURITY DEFINER RPC)

**Codex pass 1 CRITICAL — TOCTOU race.** A naive `SELECT count(*)` followed by `UPDATE` lets two owners concurrently demote/remove themselves and observe `count = 2` on both calls, leaving the org with zero active owners. **Fix: every membership write goes through a SECURITY DEFINER Postgres RPC that holds `FOR UPDATE` locks on the relevant rows and does the count check + write + audit insert in one transaction.**

RPCs (all with `SET search_path = public, pg_temp`, all in the migration delta below):

- `invite_member(p_caller_user_id uuid, p_org_id uuid, p_invitee_email text, p_role text) returns memberships`
- `change_member_role(p_caller_user_id uuid, p_org_id uuid, p_target_user_id uuid, p_new_role text) returns memberships`
- `remove_member(p_caller_user_id uuid, p_org_id uuid, p_target_user_id uuid) returns memberships`
- `update_org_name(p_caller_user_id uuid, p_org_id uuid, p_new_name text) returns organizations`

Each RPC:
1. Validates caller has admin/owner (or owner-only as applicable) membership in p_org_id; raises if not.
2. Locks the relevant membership rows: `SELECT ... FROM memberships WHERE organization_id = p_org_id FOR UPDATE` (locks all org members; cheap because typical org < 50 members).
3. Performs the count check (last-owner) under the lock.
4. INSERT/UPDATE the membership row.
5. INSERT into audit_events using `audit.append()` chain pattern.
6. Returns the updated row.

If validation fails, raises a coded exception (`P0001` codes: `not_admin`, `not_owner`, `last_owner`, `role_transition`, `user_not_found`, `already_member`, `target_not_member`, `bad_name`). Route handlers map the SQLSTATE → HTTP 403/404/409/422 with a stable error code in the JSON body.

### Cross-org enumeration

`:orgId` route param validated against caller's active memberships. If caller has no active membership for `orgId` → 404 (NOT 403, mirrors Phase 4 visibility/artifact pattern). Avoids enumeration of valid org IDs.

### CSRF

All mutating routes (POST, PATCH, DELETE) call `assertSameOrigin(req)` first. Cookie-only auth = vulnerable to CSRF without it. Pattern matches Phase 4.

## Data flow

All mutating routes follow the same shape: `assertSameOrigin → resolveSessionUserId (cookie) → svc.rpc('<verb>', { p_caller_user_id, p_org_id, ... }) → map SQLSTATE → HTTP`. The RPC owns authorization, locking, last-owner check, write, and audit. Route handlers are thin.

### Invite — `invite_member` RPC

1. POST `{ email, role: 'member' | 'admin' }` to `/orgs/:orgId/members/invite`
2. Route: assertSameOrigin → resolve caller → call `invite_member(caller, orgId, email, role)`.
3. RPC: assert caller is admin/owner. Assert role ∈ {'member','admin'} (admins can never invite-as-owner — only role transition by existing owner can promote). Email lookup `WHERE lower(email) = lower(p_invitee_email)`; raise `user_not_found` if 0, `bad_email` if >1 (defense; auth.users has UNIQUE on email).
4. Look up existing membership for (orgId, invitee). If status='active' → raise `already_member` (409). If status='removed' → UPDATE status='active', role=p_role, joined_at=NOW() (reactivation path). Else → INSERT new row.
5. INSERT `audit.append('org.member.invited', orgId, caller, target_user_id, jsonb_build_object('role', p_role, 'previousStatus', <removed|null>))` — metadata distinguishes reactivation from first invite.
6. Return updated/inserted membership row.

### Change role — `change_member_role` RPC

1. PATCH `{ role: 'member' | 'admin' | 'owner' }` to `/orgs/:orgId/members/:userId`
2. Route: standard. Call `change_member_role(caller, orgId, target_user_id, new_role)`.
3. RPC: lock memberships rows for orgId. Resolve caller_role + target_membership (must be active; raise `target_not_member` if removed/missing). Apply role transition matrix:
    - caller=admin → target_role ∈ {'member','admin'} AND new_role ∈ {'member','admin'}; else `role_transition`.
    - caller=owner → any transition allowed.
4. Last-owner check: IF target_membership.role='owner' AND new_role != 'owner' AND `(SELECT count(*) FROM memberships WHERE organization_id=p_org_id AND role='owner' AND status='active') = 1` → raise `last_owner`.
5. **Idempotent same-role** PATCH: if target_membership.role = p_new_role, return the row unchanged with NO audit event. (200 with `{ noop: true }` flag in JSON.)
6. Else UPDATE + audit.append `org.member.role_changed` with `{ oldRole, newRole }`.

### Remove — `remove_member` RPC

1. DELETE `/orgs/:orgId/members/:userId`
2. RPC: lock + caller role check. Admin may remove targets where role='member' only; owner may remove role∈{'member','admin','owner'} (subject to last-owner).
3. **Idempotent already-removed**: if target_membership not found OR status='removed' → return 404 `target_not_member`. We do NOT distinguish "never was member" from "was removed" — that's an enumeration leak.
4. Last-owner check (as above).
5. UPDATE status='removed' + audit.append `org.member.removed`.

### Org name update — `update_org_name` RPC

1. PATCH `{ name }` to `/orgs/:orgId`
2. Owner-only via RPC.
3. Validate `length(trim(p_new_name))` ∈ [1, 100]; raise `bad_name` else.
4. UPDATE organizations SET name=trim(p_new_name).
5. audit.append `org.settings.updated` with `{ field: 'name', oldValue, newValue }`.

### Target-membership existence is always checked

All PATCH/DELETE RPCs include `WHERE organization_id = p_org_id AND user_id = p_target_user_id AND status = 'active'` in their lookup. Cross-org/non-member target → `target_not_member` (404).

## Migration delta — `data/deltas/20260508140000_phase5_1_member_rpcs.sql`

**Codex pass 1 CRITICAL fold-in: 4 SECURITY DEFINER RPCs replace direct service-role writes.** No new tables. No new columns. RPCs hold `FOR UPDATE` locks on `memberships` rows for `p_org_id` so the count check + write + audit insert are one atomic unit.

```
CREATE FUNCTION public.invite_member(p_caller_user_id uuid, p_org_id uuid, p_invitee_email text, p_role text) RETURNS public.memberships
CREATE FUNCTION public.change_member_role(p_caller_user_id uuid, p_org_id uuid, p_target_user_id uuid, p_new_role text) RETURNS public.memberships
CREATE FUNCTION public.remove_member(p_caller_user_id uuid, p_org_id uuid, p_target_user_id uuid) RETURNS public.memberships
CREATE FUNCTION public.update_org_name(p_caller_user_id uuid, p_org_id uuid, p_new_name text) RETURNS public.organizations
```

All four:
- `LANGUAGE plpgsql SECURITY DEFINER SET search_path = public, pg_temp`
- **Codex pass 2 CRITICAL**: `REVOKE ALL ON FUNCTION ... FROM PUBLIC, anon, authenticated` THEN `GRANT EXECUTE ... TO service_role` ONLY. Routes call via `createServiceRoleClient()` and pass `p_caller_user_id` derived from the verified Supabase cookie session (`supabase.auth.getUser().data.user.id`). Direct client RPC calls from `authenticated` JWTs are blocked at the GRANT level — even if a user crafted a `supabase.rpc('invite_member', { p_caller_user_id: '<owner-uuid>', ... })` call, Postgres rejects with `permission denied for function`.
- **Codex pass 2 CRITICAL — lock before authorize**: every RPC body acquires `FOR UPDATE` lock on `memberships` rows for `p_org_id` FIRST, then re-reads `caller_role` from the locked rows, then authorizes. Without this ordering an admin could be mid-flight in `remove_member`, get demoted by a concurrent transaction, and still complete the original write under stale authorization.
- Returns jsonb `{ membership: jsonb, noop: bool, org?: jsonb }` so route handlers can detect same-role no-op without a pre-RPC SELECT.
- Raise `P0001` exceptions with codes mapped to HTTP in route handlers

```
P0001 codes -> HTTP:
  not_admin, not_owner -> 403
  user_not_found, target_not_member -> 404
  already_member -> 409
  last_owner, bad_name, role_transition, bad_role -> 422
```

Route handlers parse `error.code === 'P0001'` and read `error.message` for the keyword.

`audit.append()` from Phase 1 already exists; RPCs call it directly (same transaction).

## Codex passes folded

### Pass 1 — design (1 CRITICAL + 7 WARNING + 2 NOTE — all folded)

1. **CRITICAL: Last-owner TOCTOU** — moved entire write path into 4 SECURITY DEFINER RPCs that hold `FOR UPDATE` locks; count check + write + audit insert are one atomic transaction. Test 31 is the concurrency proof.
2. **WARNING: Audit-write atomicity** — RPCs call `audit.append()` in same transaction; never separated.
3. **WARNING: Target-membership existence** — every PATCH/DELETE RPC explicitly looks up target by (orgId, userId, status='active'); raises `target_not_member` (404) if missing/removed.
4. **WARNING: Service-role bypasses RLS** — RPCs centralize all authorization. Route handlers just translate SQLSTATE → HTTP. No direct service-role .update() calls.
5. **WARNING: Idempotency under-specified** — same-role PATCH returns 200 `{ noop: true }` with no audit event. Already-removed DELETE returns 404 `target_not_member` (no info-leak distinguishing "never was" from "was removed").
6. **WARNING: Email lookup ambiguity** — switched from ILIKE to exact `lower(email) = lower(p_invitee_email)`; UNIQUE constraint on auth.users.email guarantees ≤1 row. Defensive `bad_email` if >1 (impossible but explicit).
7. **WARNING: Test plan misses edge cases** — expanded from 22 to 35 tests. Added: target-not-member (15, 23), same-role no-op (16), CSRF on PATCH role + DELETE + org PATCH (17, 24, 29), admin-demoting-admin (18), owner-removing-1-of-many (22), case-insensitive email (30), concurrent demotion (31).
8. **WARNING: settings owner vs admin inconsistency** — settled: settings page is owner-only, admin gets 404. Test 34 asserts.
9. **NOTE: deployment target** — claude-autopilot deploys to **Vercel** (not ECS — recurring confusion with delegance-app). `assertSameOrigin` already correctly uses `AUTOPILOT_PUBLIC_BASE_URL` from config, not `req.url`.
10. **NOTE: scope** — settings stays in 5.1; uses same hardened RPC pattern as members.

### Pass 2 — final (2 CRITICAL + 3 WARNING — all folded)

1. **CRITICAL: SECURITY DEFINER caller spoofing** — RPCs grant only `service_role` after `REVOKE ALL ON FUNCTION ... FROM PUBLIC, anon, authenticated`. Routes use service-role client; pass `p_caller_user_id` from cookie-verified `getUser()`. Direct RPC spoofing rejected by GRANT. Test 31b proves.
2. **CRITICAL: Lock-before-authorize ordering** — every RPC body acquires `FOR UPDATE` on memberships first, re-reads caller_role from locked rows, then authorizes. Prevents stale-role races.
3. **WARNING: Admin role-transition consistency** — Test 18 was wrong relative to the matrix; matrix locked, test 18 now expects 200. Removal matrix added separately (admin removes member only).
4. **WARNING: noop signaling** — RPC return type changed to `jsonb { membership, noop, org? }` so route can detect same-role no-op without pre-RPC SELECT.
5. **WARNING: error code contract** — handlers parse `error.code === 'P0001'` AND `error.message` as the keyword (e.g. `not_admin`, `last_owner`). Stable enum.

## Hard rules (carry-forward from prior phases)

- All mutating routes call `assertSameOrigin(req)` before auth
- 404 (not 403) on non-member access to avoid org-ID enumeration
- Service-role client for writes (RLS will allow via memberships_write_admin policy, but explicit role check makes errors readable)
- Audit events for every write (invited, role_changed, removed, settings_updated)
- No new schema. No new env vars. No CLI changes (5.1 is dashboard-only).
