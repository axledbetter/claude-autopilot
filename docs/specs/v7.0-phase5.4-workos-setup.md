**Status:** ready for plan
**Repo:** `claude-autopilot` (Vercel — confirmed correct per `docs/specs/v7.0-hosted-product-mvp.md` Stack rationale; recurring cross-repo confusion with delegance-app's ECS deployment)

## Goal

Org admins can connect their WorkOS SSO connection through the Admin Portal flow. The setup phase ends when WorkOS marks the connection active and our system records the connection state. **No user-facing sign-in flow yet** — that's Phase 5.6.

This phase ships the durable data model + the secure webhook + the admin UI surface so 5.6 can wire in `/login/sso` against a working setup.

## Non-goals (deferred)

- `/login/sso` route / WorkOS OAuth callback (Phase 5.6)
- Domain claim + DNS TXT verification (Phase 5.6)
- `sso_required` enforcement toggle (Phase 5.6)
- Admin disable-user + Supabase session revocation (Phase 5.7)
- Directory Sync (SCIM) — explicitly out of scope per v7.0-mvp spec
- ~~WorkOS organization auto-creation~~ — **codex pass 2 CRITICAL: contradiction with portal-link flow.** Setup route MUST server-create the WorkOS org via `workos.organizations.createOrganization()` before generating the Admin Portal link. `workos_organization_id` is server-owned, never client-supplied. Persist BEFORE returning portal URL.
- Multi-connection per org — one connection per org for now (most enterprise customers consolidate to one IDP)

## Surfaces

### Backend
- `POST /api/dashboard/orgs/:orgId/sso/setup` — admin/owner only. **Sequence (codex pass 2 CRITICAL — order matters because WorkOS API + Postgres can't be one transaction):**
  1. Authenticate caller + assertSameOrigin
  2. Pre-RPC: read existing `workos_organization_id` for the local org (idempotent path — reuse if already bound)
  3. Call `workos.organizations.createOrganization()` ONLY IF none stored yet
  4. Call `workos.portal.generateLink({ organization, intent: 'sso' })` — short-lived, single-use admin URL
  5. Call `record_sso_setup_initiated` RPC (atomic: validates admin role, persists `workos_organization_id` + `sso_connection_status='pending'`, writes audit event)
  6. Return `{ portalUrl }`. If step 5 fails, DO NOT return the URL — the admin can retry; the WorkOS org is reused on the next attempt via step 2.
- `POST /api/workos/webhook` — receives WorkOS lifecycle events. Signature-verified via `workos-node` SDK or manual HMAC check. Idempotent via `processed_workos_events` table. Handles:
  - `connection.activated` → store connection_id + status='active'
  - `connection.updated` → refresh stored state
  - `connection.deactivated` / `connection.deleted` → status='inactive', clear connection_id
- `DELETE /api/dashboard/orgs/:orgId/sso` — owner-only. Disconnects the WorkOS connection (calls WorkOS DELETE /connections/:id, clears local state). Used when an org wants to switch IDPs or stop using SSO.

### Frontend
- `/dashboard/admin/sso/page.tsx` — Server Component, owner-only. Shows current state (connected/not-connected/disconnected) + "Set up SSO" button (calls setup route, opens portal URL in new tab with `rel="noopener noreferrer"` to prevent opener-context leak — codex pass 2 NOTE) + "Disconnect" button (admin confirmation modal). Server response sets `Cache-Control: private, no-store` so the short-lived portal URL isn't cached by any proxy.
- Sidebar admin nav adds "SSO".

Plus 1 SECURITY DEFINER RPC for the setup route to atomically check admin role + record audit event:

```sql
CREATE OR REPLACE FUNCTION public.record_sso_setup_initiated(
  p_caller_user_id uuid,
  p_org_id uuid,
  p_workos_organization_id text
) RETURNS jsonb
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path = public, audit, pg_temp
AS $$
DECLARE
  v_caller_role text;
BEGIN
  SELECT role INTO v_caller_role
    FROM public.memberships
   WHERE organization_id = p_org_id
     AND user_id = p_caller_user_id
     AND status = 'active';
  IF v_caller_role IS NULL OR v_caller_role NOT IN ('admin', 'owner') THEN
    RAISE EXCEPTION 'not_admin' USING ERRCODE = 'P0001';
  END IF;

  -- Codex pass 2 CRITICAL — reject WorkOS org reassignment to prevent
  -- cross-tenant state. If a different workos_organization_id is already
  -- bound and the connection is active, reject. Owner must DELETE first.
  DECLARE
    v_existing_workos_org_id text;
    v_existing_status text;
  BEGIN
    SELECT workos_organization_id, sso_connection_status
      INTO v_existing_workos_org_id, v_existing_status
      FROM public.organization_settings
     WHERE organization_id = p_org_id;
    IF v_existing_workos_org_id IS NOT NULL
       AND v_existing_workos_org_id <> p_workos_organization_id
       AND v_existing_status = 'active' THEN
      RAISE EXCEPTION 'workos_org_already_bound' USING ERRCODE = 'P0001';
    END IF;
  END;

  -- Persist WorkOS org id so we can correlate later webhooks even if the
  -- connection_id isn't yet known. Only overwrite when no existing binding
  -- or the org is in a non-active state.
  INSERT INTO public.organization_settings (
    organization_id, workos_organization_id, sso_connection_status, updated_by
  )
  VALUES (
    p_org_id, p_workos_organization_id, 'pending', p_caller_user_id
  )
  ON CONFLICT (organization_id) DO UPDATE SET
    -- Only overwrite the workos_organization_id if not already bound to a
    -- different active connection (the pre-check above guards that path).
    workos_organization_id = CASE
      WHEN public.organization_settings.workos_organization_id IS NULL THEN EXCLUDED.workos_organization_id
      WHEN public.organization_settings.sso_connection_status = 'active' THEN public.organization_settings.workos_organization_id
      ELSE EXCLUDED.workos_organization_id
    END,
    sso_connection_status = CASE
      WHEN public.organization_settings.sso_connection_status = 'active' THEN 'active'
      ELSE 'pending'
    END,
    updated_at = NOW(),
    updated_by = p_caller_user_id;

  PERFORM audit.append(
    p_org_id, p_caller_user_id, 'org.sso.setup_initiated',
    'organization', p_org_id,
    jsonb_build_object('workosOrganizationId', p_workos_organization_id)
  );

  RETURN jsonb_build_object('ok', true);
END;
$$;

REVOKE ALL ON FUNCTION public.record_sso_setup_initiated(uuid, uuid, text) FROM PUBLIC, anon, authenticated;
GRANT EXECUTE ON FUNCTION public.record_sso_setup_initiated(uuid, uuid, text) TO service_role;
```

## Webhook lifecycle ordering (codex pass 2 WARNING)

WorkOS retries / delayed delivery can produce out-of-order events. Each lifecycle handler MUST:

1. Look up the org by `workos_organization_id` (preferred — present in every connection event payload)
2. Compare `payload.created_at` to `sso_last_workos_event_at`. If event is older, treat as no-op and mark processed.
3. Apply the state transition only if older event. State machine:
   - `inactive` → `pending` (setup_initiated) → `active` (connection.activated)
   - `active` → `disabled` (connection.deactivated) — keeps `workos_connection_id` for replay
   - any → `inactive` (connection.deleted) — clears `workos_connection_id`
4. Update `sso_last_workos_event_at` + `sso_last_workos_event_id`

Strict rule: a `connection.updated` event older than the last-seen `connection.deleted` MUST NOT resurrect the connection.

## Disconnect semantics (codex pass 2 WARNING — two-step)

DELETE route does NOT immediately clear `workos_connection_id`. Instead:
1. Set `sso_connection_status = 'disabled'`, `sso_disabled_at = NOW()`, leave `workos_connection_id` intact
2. Call WorkOS `DELETE /connections/:id`
3. Wait for `connection.deleted` webhook to clear `workos_connection_id` AND set status='inactive'

If the WorkOS DELETE call fails, local state is `disabled` (gating any future use) but the connection_id is preserved for retry/reconciliation.

## Webhook signature verification

`@workos-inc/node` SDK exposes `workos.webhooks.constructEvent(payload, signature, secret, tolerance)`. We use the official path. Required env vars: `WORKOS_API_KEY`, `WORKOS_WEBHOOK_SECRET`. Webhook route runs `runtime = 'nodejs'` because we need raw body bytes for HMAC verification (matches Phase 3's Stripe webhook pattern).

**Idempotency (codex pass 2 CRITICAL — claim/lease/complete):** Same shape as Phase 3's Stripe webhook. Upon receipt:

1. INSERT into `processed_workos_events` with status='processing', `processing_started_at`, `locked_until = NOW() + 5 min`. `INSERT ... ON CONFLICT (event_id) DO UPDATE SET attempt_count = attempt_count + 1, locked_until = ... WHERE status = 'failed' OR locked_until < NOW()` — only re-claim failed or stale-lock rows.
2. If the insert reports an existing 'processed' row, return 200 immediately.
3. Apply the lifecycle state transition (with ordering rules above) AND audit append in one transaction.
4. UPDATE the event row to status='processed', processed_at=NOW(). If transaction throws, status='failed' + last_error captured for retry.

A webhook crash between step 1 and step 4 leaves the event in 'processing' state with `locked_until` set; the next retry will recover after the 5-minute lock expires.

### Webhook
11. valid `connection.activated` → connection_id stored, status='active', sso_connected_at set, audit event emitted
12. valid `connection.updated` for known org → updates timestamps
13. valid `connection.deactivated` → status='disabled', connection_id NOT cleared (preserve for replay)
14. valid `connection.deleted` → connection_id cleared, status='inactive'
15. invalid signature → 401, no DB write
16. duplicate event_id → 200 idempotent (no second audit event)
17. unknown event type → 200 acked, ignored
18. event for unknown workos_organization_id → 200 acked + structured warning log (don't 4xx — would cause WorkOS retry storm)

## Codex pass 2 (anticipated WARNINGs to fold inline)

1. **Webhook payload size + memory** — Vercel limits payload to ~4MB; WorkOS events are small. No streaming needed.
2. **Race: simultaneous setup + connection.activated** — RPC's `CASE WHEN sso_connection_status = 'active'` prevents downgrading on a stale setup-initiated call landing after activation.
3. **Replay attack window** — `processed_workos_events.event_id` PK + payload_hash; we accept events within WorkOS's recommended 5-minute tolerance.
4. **Setting cookie on redirect** — N/A in 5.4; portal URL opens in new tab, no Supabase session change.
5. **WORKOS_API_KEY rotation** — env var; doc that key rotation requires no migration.
