# v7.0 Phase 2.1 — Next.js scaffold + Supabase Auth (Free tier sign-in)

**Status:** spec drafted 2026-05-07
**Origin:** v7.0 Phase 1 shipped the database foundation (PR #115). Phase 2 is "Ingest API + tamper-evident events + CLI integration", which decomposes into three sub-PRs. **This spec covers Phase 2.1 only**: the Next.js + Supabase Auth foundation that 2.2 (ingest endpoints) and 2.3 (CLI integration) build on. NO API endpoints related to ingest in this PR — that's deliberate scope discipline.

## What we're building

A new `apps/web/` directory at the repo root containing a Next.js 16 App Router application that:

- Renders a minimal landing page (signed-out: "Sign in with Google"; signed-in: "Welcome, <email>" + sign-out)
- Handles Supabase Auth Google OAuth via the standard PKCE callback pattern
- Reads/writes session cookies via `@supabase/ssr` server + browser clients
- Refreshes sessions on protected routes via scoped middleware (NOT on every request)
- Deploys to Vercel pointing at `apps/web/` as the project root

The app does NOT yet display run history, dashboards, or any of the v7.0 product surface. Phase 2.1 is **pure foundation** — once it ships, Phase 2.2 adds the ingest API endpoints, Phase 2.3 wires the CLI.

## Why this matters

Three concrete things land that 2.2 + 2.3 + Phases 4-6 all depend on:

1. **A server surface in the repo for the first time.** All v6.x code is CLI-only. Phase 1 added Supabase migrations but no HTTP layer. 2.1 is where that changes.
2. **Auth infrastructure usable by 2.2's ingest endpoints.** The `@supabase/ssr` server client + cookie management is what 2.2's `POST /api/upload-session` will call to identify the user issuing the upload session.
3. **Vercel deployment plumbing shaken out.** Build output, env var conventions, preview deployments — all real-world quirks better discovered with a hello-world page than with the cryptographic-verification surface of 2.2.

### Auth flow (PKCE callback)

1. User clicks "Sign in with Google" on `/`.
2. Browser-side `supabase.auth.signInWithOAuth({ provider: 'google', options: { redirectTo: `${origin}/api/auth/callback` }})` → redirects to Google.
3. Google redirects back to `/api/auth/callback?code=<auth_code>&next=<safe_path>`.
4. `app/api/auth/callback/route.ts`:
   - `supabase.auth.exchangeCodeForSession(code)` → sets session cookies (Supabase manages PKCE verifier + state correlation internally)
   - On missing `code` query param: `redirect('/?error=auth_no_code')`
   - On `exchangeCodeForSession` throw (bad code / expired / replay attempt): `redirect('/?error=auth_failed')`
   - On `?error=...` from upstream provider (user denied, etc.): pass through to `/?error=<sanitized_code>`
   - On success: `redirect(safeRedirect(next))`
   - **NEVER trust query-provided origins**: `redirectTo` in step 2 is computed from `req.nextUrl.origin` server-side; the callback handler's redirect URL is derived from `safeRedirect`, never from any query param origin.
5. Subsequent requests go through `middleware.ts` which refreshes the session if needed (matcher excludes static + healthcheck routes + non-auth API routes).
6. Server components / route handlers use `lib/supabase/server.ts` to read the authenticated user.

**Security notes** (codex final-pass NOTE #1): we rely on `@supabase/ssr`'s built-in PKCE/state machinery. We do NOT mint our own state token or perform our own correlation. The integration test suite includes a "missing code", "invalid code", and "upstream error pass-through" case to pin the contract. If a future Supabase SDK change loosens these guarantees, the tests fail and we block the upgrade.

### `safeRedirect` whitelist (codex WARNING)

```typescript
const ALLOWED_REDIRECT_PATHS = [
  /^\/$/,
  /^\/dashboard(\/|$)/,
  /^\/runs(\/|$)/,
  /^\/settings(\/|$)/,
];

export function safeRedirect(next: string | null | undefined): string {
  if (!next) return '/';
  if (next.startsWith('//') || next.includes('://')) return '/';  // reject absolute / scheme-relative
  if (!ALLOWED_REDIRECT_PATHS.some(re => re.test(next))) return '/';
  return next;
}
```

Tested with: valid path, malicious `//evil.com`, scheme-relative `https://attacker.com`, unknown path, null, empty.

**Change policy** (codex final-pass WARNING #4): the `ALLOWED_REDIRECT_PATHS` array is the single source of truth for post-auth destinations. Any new authenticated route added in 2.2+ that should accept a `?next=` redirect MUST update this array AND the `safeRedirect` test fixture in the same PR. Reviewers reject PRs that add a route but don't update the allowlist (the silent fallback-to-`/` would otherwise look like lost navigation state). The phrasing of this rule lives in `apps/web/lib/auth/redirect.ts` as a JSDoc comment so the next contributor sees it inline.

### Middleware matcher (codex WARNING)

```typescript
export const config = {
  matcher: [
    // Match all request paths EXCEPT:
    // - /_next/static, /_next/image (build artifacts)
    // - /favicon.ico, public files (.svg, .png, etc.)
    // - /api/health (platform health check)
    // - /api/* OTHER than /api/auth/* — Phase 2.2's ingest endpoints will
    //   handle auth themselves via signed-session JWT, not via cookie
    //   refresh. Forcing middleware on every ingest call would add
    //   latency + cookie churn that's invisible to the operator (codex
    //   final-pass WARNING #3).
    '/((?!_next/static|_next/image|favicon\\.ico|.*\\.(?:svg|png|jpg|jpeg|gif|webp)$|api/health|api/(?!auth/)).*)',
  ],
};
```

Session refresh happens on page navigation, sign-in callback, sign-out, and `/api/auth/*` — but NOT on static asset fetches OR ingest API calls. A test asserts ingest endpoint paths are excluded once 2.2 introduces them.

### Cookie handling

Sign-in: `@supabase/ssr` writes httpOnly cookies for `sb-<project_ref>-auth-token`. Cookie names match Supabase's defaults so the browser client and server client agree on the same session.

Sign-out: `supabase.auth.signOut()` is the primary action. As a fallback for inconsistent clearing across environments, the response handler additionally deletes **only** the cookies for the configured project ref — `sb-<NEXT_PUBLIC_SUPABASE_PROJECT_REF>-auth-token` and any chunked variants (`sb-<ref>-auth-token.0`, `.1`, etc.) — with explicit path / domain attributes matching what Supabase set. **No `sb-*` wildcard sweep** (codex final-pass WARNING #2): on shared parent domains, a wildcard delete would clear cookies for unrelated Supabase projects (local dev → preview → staging → prod can all coexist), causing hard-to-debug cross-app auth bugs. Test asserts both server and browser sides agree post-signout (no stale session) and that a second project's cookies under a different ref are untouched.

### CI workflow

`.github/workflows/web-tests.yml`:

1. Checkout + Node 22 + npm ci (workspaces install both CLI and web deps).
2. **Required env-var validation**: fail with a clear message if `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY`, `NEXT_PUBLIC_SUPABASE_PROJECT_REF`, or `NEXT_PUBLIC_SITE_URL` are missing. Also validate `NEXT_PUBLIC_SITE_URL` matches `^https?://` and the callback path resolves to a known route (codex final-pass NOTE #2 — misalignment between Supabase Auth provider's redirect allowlist and the actual `NEXT_PUBLIC_SITE_URL` is the most common production OAuth break that passes build but fails real auth). (CI uses Phase 1's local-Supabase keys; production uses Vercel's env config.)
3. `npm run typecheck --workspace apps/web` — strict TS check.
4. `npm run build --workspace apps/web` — Next.js production build (catches missing env at build time, broken route handlers, etc.).
5. `npm run test:web --workspace apps/web` — the 17 tests.
6. **Tarball boundary check**: `npm pack --dry-run` and assert `apps/` does NOT appear. Explicit guard against accidental web sources in the published CLI.

### Failure modes

| Scenario | Expected behavior |
|---|---|
| Missing `NEXT_PUBLIC_SUPABASE_URL` env at build | Build fails with clear message |
| Google OAuth returns `?error=access_denied` | Callback redirects to `/?error=auth_failed` (no crash) |
| Supabase rejects `code` (expired/replay) | `exchangeCodeForSession` throws → callback catches → redirect with error param |
| User passes `?next=//evil.com` | safeRedirect rejects → redirect to `/` |
| User passes `?next=/admin` (path not in allowlist) | safeRedirect rejects → redirect to `/` |
| Session cookie tampered/forged | Supabase Auth rejects on next request → redirect to `/?error=session_invalid` |
| User signs out while another tab is open | Middleware sees no session → redirects to `/`; both tabs sign out cleanly |

1. `apps/web/` builds cleanly via `next build` with required env vars set.

3. Existing CI green: `ci.yml` (CLI tests), `bin-parity.yml` (npm tarball smoke), `db-tests.yml` (RLS), `delegance-regression.yml`, `adapter-cert.yml`.

5. Vercel preview deployment succeeds end-to-end against a Vercel-staging Supabase project: sign in with Google → redirect to `/` → see "Welcome, <email>" → sign out → session cleared.
6. `package.json` version bumps to `6.3.0-pre.2` on the `next` dist-tag (latest stays 6.2.2).
