# v7.0 Phase 2.3 — CLI dashboard verbs + auto-upload at run.complete

**Status:** spec drafted 2026-05-07
**Origin:** v7.0 Phase 2.1 shipped the Next.js scaffold + Supabase Auth (PR #117). v7.0 Phase 2.2 shipped the ingest API endpoints with hash-chain tamper-evidence (PR #119). Phase 2.3 is the **CLI side** that connects the existing v6.x autopilot pipeline to the hosted ingest API. After this phase, every engine-on `claude-autopilot autopilot --mode full` automatically lands at `https://autopilot.dev/runs/<runId>`.

> **Anchor — platform context.** This spec lives in the `claude-autopilot` repo. The v7.0 hosted product deploys to **Vercel** per `docs/specs/v7.0-hosted-product-mvp.md`. The CLAUDE.md guidance about AWS ECS applies to the unrelated delegance-app repo. Ignore any review feedback that flags Vercel as a platform mismatch.

## What we're building

Three new CLI verbs in `src/cli/dashboard/`:

1. **`claude-autopilot dashboard login`** — opens a browser to authenticate via Supabase Auth, mints a long-lived CLI API key, stores it locally at `~/.claude-autopilot/dashboard.json` (mode 0600). Falls back to manual paste if browser/loopback fails.
2. **`claude-autopilot dashboard logout`** — revokes the local key server-side and deletes the local file.
3. **`claude-autopilot dashboard status`** — shows account email, key fingerprint, org membership list, last upload timestamp.
4. **`claude-autopilot dashboard upload <runId>`** — manual upload of a previously completed run (codex WARNING — recovery path for offline / interrupted / `--no-upload` cases).

Plus the **auto-upload integration** at `run.complete` in the v6.2.0 multi-phase orchestrator: when dashboard config exists and engine is on, freeze a snapshot of the run directory, chunk events.ndjson with the Phase 2.2 hash chain, upload via the three Phase 2.2 endpoints (`/api/upload-session` → `/api/runs/:runId/events/:seq` → `/api/runs/:runId/finalize`), and print `[autopilot] uploaded to https://autopilot.dev/runs/<runId>`.

## Why this matters

Phase 2.2 built the credible-audit boundary. Phase 2.3 is what makes it **default**: every YC-demo run, every dogfood run from the maintainer, every bug-fix iteration — they all land at `autopilot.dev` with hash-chain proof. Without Phase 2.3, Phase 2.2 is dead code.

This is also the phase that **proves the hash-chain canonicalization spec across language boundaries**. Phase 2.2's fixtures at `apps/web/lib/upload/__fixtures__/{chain-vectors,state-canonicalization-vectors}.json` get imported by the CLI test suite to assert byte-identical output. If the two implementations ever diverge, the parity test fails before any cross-language upload corruption can land.

## Login flow — nonce-bound loopback (codex CRITICAL #1)

The CSRF/CSRF-equivalent for a long-lived API key handoff has to be hard, not loose. Using a referer check or a secret cookie isn't enough — another local process or a malicious page in another browser tab could race the legitimate callback. The fix is a **128-bit ephemeral nonce** generated CLI-side, embedded in the launch URL, and required in the loopback POST body. The server (the user's CLI listening on a one-shot HTTP listener) accepts exactly ONE successful POST whose body carries the matching nonce, then closes the listener.

Concrete protocol:

1. CLI generates `nonce = randomBytes(16).toString('hex')` (32-char hex, 128 bits of entropy).
2. CLI binds an HTTP listener on the first available port in `127.0.0.1:56000-56050`. Same port range every time keeps firewall prompts predictable.
3. CLI opens the user's default browser to `https://autopilot.dev/cli-auth?cb=http://127.0.0.1:<port>/cli-callback&nonce=<nonce>`.
4. The web page authenticates the user via existing Phase 2.1 Supabase Auth flow.
5. After successful Supabase Auth, the page calls server-side `POST /api/dashboard/api-keys/mint` with `{ nonce }` (nonce traveled via the URL → browser → POST body). Server verifies user, mints a new `api_keys` row, returns the raw `clp_<32-byte-hex>` key.
6. The page POSTs `{ apiKey, nonce, fingerprint, accountEmail }` to `http://127.0.0.1:<port>/cli-callback`. Content-Type `application/json`. Method MUST be POST. Max body size 4 KiB.
7. CLI listener:
   - Asserts `req.method === 'POST'`.
   - Asserts `req.headers['content-type']?.startsWith('application/json')`.
   - Reads ≤4 KiB body; rejects oversized.
   - Asserts `body.nonce === expectedNonce` (constant-time comparison via `crypto.timingSafeEqual`).
   - On any mismatch: respond 400, do NOT close listener — wait for legitimate callback OR 60s timeout.
   - On match: write the api_key to `~/.claude-autopilot/dashboard.json` atomically (codex WARNING #4 — temp file + rename), respond 200 with `<html>You can close this tab.</html>`, close the listener.
8. CLI prints `Logged in as <email>. Key fingerprint clp_xxxxxxxx...` and exits 0.

**Manual paste fallback** (browser unavailable, no display, restricted firewall):
- CLI prints "Open this URL in any browser: `https://autopilot.dev/cli-auth?manual=1`"
- Web page after auth shows the raw key once with "copy" button.
- User pastes back into terminal. Same atomic-write + permissions handling.

**Why not OAuth device-code?** Future option; Phase 2.3 keeps surface minimal. The nonce-bound loopback is sufficient for MVP because the threat model is "another local process races the callback" — a 128-bit nonce at 1024 attempts/sec would take ~10^28 years to brute force.

## Auto-upload at run.complete

### Trigger

`runAutopilot()` (v6.2.0 multi-phase orchestrator in `src/cli/autopilot.ts`) emits `run.complete` and currently exits. After Phase 2.3, immediately before exit:

```typescript
if (shouldAutoUpload(runDir, options)) {
  const result = await uploadRun(runId, runDir);
  if (result.ok) {
    console.log(`[autopilot] uploaded to ${result.url}`);
  } else {
    console.warn(`[autopilot] upload failed: ${result.error}. Resume with: claude-autopilot dashboard upload ${runId}`);
    // codex WARNING #2 — non-fatal. Exit code preserved from run, not upload.
  }
}
```

`shouldAutoUpload()` checks: dashboard.json exists, engine is on (events.ndjson exists), `--no-upload` flag absent, `CLAUDE_AUTOPILOT_UPLOAD` env var != `'off'`, `engine.dashboard.uploadOnComplete` config != `false`.

Engine OFF → silent no-op (codex open question #6 — log nothing, the user opted out of engine, they don't want hosted upload either).

### Snapshot before upload (codex CRITICAL #2)

The auto-upload module **must not write its own progress events to the same `events.ndjson` it's about to chunk**. Self-mutation during streaming makes chunk boundaries shift, hash chain non-deterministic, retries produce different roots.

Solution: copy `events.ndjson` and `state.json` into `<runDir>/.upload-snapshot/` immediately after the trigger, then chunk from the snapshot. Progress events go to:
- stderr lines (humans see them).
- A separate `<runDir>/.upload-snapshot/upload.ndjson` (cold storage for `runs watch <runId>` if anyone wants it).
- NEVER back into the run's own `events.ndjson`.

The orchestrator's `events.ndjson` is **frozen at run.complete**. After the run is final, no more writes to it from any code path. This makes the snapshot a defensive copy, not a hard requirement, but the defense-in-depth is cheap.

### Chunking + hash chain (codex NOTE #1)

Chunks are byte-based, not line-based. Stream-read the snapshot file in 1 MiB `Buffer` chunks; trailing partial chunk is whatever's left. Each chunk:

```
H_0 = 0x00 * 32  (32 zero bytes)
H_i = SHA256(chunk_bytes_i || H_{i-1})  (where || means raw byte concat, prev hash is 32 binary bytes)
chain_root = H_{N-1}  (lowercase hex, 64 chars)
```

This is byte-identical to `apps/web/lib/upload/chain.ts` (Phase 2.2). The CLI implementation lives at `src/dashboard/upload/chain.ts` and shares fixtures.

Chunk size `MAX_CHUNK_BYTES = 1024 * 1024` matches Phase 2.2's server limit. Tests cover edges:
- Empty file (zero chunks → finalize is `expectedChunkCount: 0`).
- Exactly 1 MiB (1 chunk).
- 1 MiB + 1 byte (2 chunks; second chunk = 1 byte).
- 100 MiB long-running pipeline (100 chunks).
- Non-ASCII event payloads (utf-8 bytes ≠ char count).

### Server contract — idempotency (codex CRITICAL #3)

Phase 2.2's claim_chunk_slot RPC already supports same-payload retry: same `(session_id, seq)` with byte-identical hash/bytes/path returns success silently; differing payload → 409 P0005. Finalize is idempotent by `consumed_at` CAS + state_sha256 pinning (Phase 2.2 fix).

Phase 2.3 leans on these existing guarantees. The CLI uploader does:
- On chunk PUT 5xx → exponential backoff retry (1s, 4s, 16s; 4 attempts max).
- On chunk PUT 4xx → fail-fast with deterministic error.
- On chunk PUT 409 from existing-payload-mismatch → fail-fast (means another process already advanced this slot with different bytes — should not happen in practice).
- On finalize 5xx → retry 4 attempts.
- On finalize 200 idempotent (server's `consumed_at IS NOT NULL` branch) → return success.

**Resume after Ctrl-C:** server-side, the partial upload session has `consumed_at IS NULL` and some `upload_session_chunks` rows. The Phase 2.2 partial-unique-index `(run_id) WHERE consumed_at IS NULL` blocks a new mint for the same runId. The recovery path is **`claude-autopilot dashboard upload <runId>`**: the manual command queries the existing in-flight session via `GET /api/dashboard/runs/:runId/upload-session` (NEW endpoint), gets the same JWT back (or mints a fresh one if expired), and resumes from `next_expected_seq`. Server's existing-pending-row-with-matching-payload branch in `claim_chunk_slot` makes this safe.

### Concurrent uploads (codex open question #8)

Two parallel `autopilot --mode=full` from different terminals get different runIds (ULID-based). No conflict in the upload session table; no conflict in storage paths. Sanity test: simulate two parallel uploads, assert both succeed independently.

## API key contract

### Generation

Raw key shape: `clp_` + 64 hex chars = 68 chars total. Stored only as `SHA256(raw)` hex in `api_keys.key_hash`. Constant-time comparison via existing `apps/web/lib/upload/auth.ts` helper.

### Display fingerprint (codex WARNING #5)

Don't display "first 8 chars" — that's just `clp_xxxx` for any key (4 entropic chars). Display `clp_<first-12-hex-chars>...` = 16 chars total, 48 bits of entropy in the visible portion (still keeps the rest secret). E.g. `clp_a1b2c3d4e5f6...`.

### Server endpoints (NEW in Phase 2.3)

- **`POST /api/dashboard/api-keys/mint`** — body `{ nonce, label? }`. Auth: Supabase session cookie. Mints a new key for the authed user, returns `{ apiKey, fingerprint, keyId }`. Idempotent on (user_id, nonce) — same nonce within 5 min returns the same key (handles browser-page reload during the loopback POST).
- **`POST /api/dashboard/api-keys/revoke`** — body `{ keyId }` OR `{ apiKey }` (either form acceptable). Auth: Supabase session cookie OR API key (you can revoke yourself). Sets `revoked_at = NOW()`. Idempotent: revoking an already-revoked key returns 200.
- **`GET /api/dashboard/me`** — auth: API key OR Supabase session. Returns `{ email, fingerprint, organizations: [{ id, name, role }], lastUploadAt }`.
- **`GET /api/dashboard/runs/:runId/upload-session`** — auth: API key. Returns the in-flight session for this run if one exists (resume scenario). 404 if no session and run not yet finalized.

### Authentication helper (codex WARNING #1)

Centralize API-key auth in one helper:

```typescript
async function authViaApiKey(req: Request): Promise<{ userId: string; keyId: string } | null> {
  const auth = req.headers.get('authorization');
  if (!auth?.startsWith('Bearer clp_')) return null;
  const raw = auth.slice('Bearer '.length).trim();
  if (!/^clp_[0-9a-f]{64}$/.test(raw)) return null;
  const hash = createHash('sha256').update(raw).digest();
  const supabase = createServiceRoleClient();
  const { data } = await supabase.from('api_keys')
    .select('id, user_id, key_hash, revoked_at')
    .is('revoked_at', null);  // codex WARNING — every protected endpoint filters revoked
  if (!data) return null;
  for (const row of data as { id: string; user_id: string; key_hash: string }[]) {
    let stored: Buffer;
    try { stored = Buffer.from(row.key_hash, 'hex'); } catch { continue; }
    if (stored.length !== hash.length) continue;
    if (timingSafeEqual(stored, hash)) {
      // Update last_used_at (fire-and-forget, non-blocking).
      supabase.from('api_keys').update({ last_used_at: new Date().toISOString() }).eq('id', row.id).then();
      return { userId: row.user_id, keyId: row.id };
    }
  }
  return null;
}
```

Phase 2.2's existing `apps/web/lib/upload/auth.ts` already does most of this for the upload-session endpoint; Phase 2.3 generalizes it into `apps/web/lib/dashboard/auth.ts` and rewires the upload-session route to use the shared helper.

## Database additions (delta migration)

`data/deltas/20260507120000_phase2_3_api_keys.sql`:

```sql
-- API keys for CLI dashboard authentication.
CREATE TABLE api_keys (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  key_hash TEXT NOT NULL UNIQUE CHECK (key_hash ~ '^[0-9a-f]{64}$'),
  prefix_display TEXT NOT NULL CHECK (prefix_display ~ '^clp_[0-9a-f]{12}$'),
  label TEXT,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  last_used_at TIMESTAMPTZ,
  revoked_at TIMESTAMPTZ
);

CREATE INDEX api_keys_user_active ON api_keys (user_id) WHERE revoked_at IS NULL;

ALTER TABLE api_keys ENABLE ROW LEVEL SECURITY;

-- Owner can SELECT/UPDATE/DELETE their own keys (revoke is via UPDATE).
CREATE POLICY api_keys_select_own ON api_keys
  FOR SELECT USING (user_id = auth.uid());

CREATE POLICY api_keys_update_own ON api_keys
  FOR UPDATE USING (user_id = auth.uid());

-- INSERT is service-role only (mint goes through /api/dashboard/api-keys/mint).
-- No INSERT policy → blocked for authed users by default.

-- Mint nonce dedup (5-min window).
CREATE TABLE api_key_mint_nonces (
  user_id UUID NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  nonce TEXT NOT NULL CHECK (nonce ~ '^[0-9a-f]{32}$'),
  api_key_id UUID NOT NULL REFERENCES api_keys(id) ON DELETE CASCADE,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  PRIMARY KEY (user_id, nonce)
);

CREATE INDEX api_key_mint_nonces_cleanup ON api_key_mint_nonces (created_at);
-- GC sweeper deletes rows older than 5 min (Phase 5 follow-up).
```

## Local config file

`~/.claude-autopilot/dashboard.json`:

```json
{
  "schemaVersion": 1,
  "apiKey": "clp_<64-hex>",
  "fingerprint": "clp_<first-12-hex>",
  "accountEmail": "alex@example.com",
  "loggedInAt": "2026-05-07T22:00:00Z",
  "lastUploadAt": null
}
```

Atomic write protocol (codex WARNING #4):
1. `fs.mkdir('~/.claude-autopilot', { recursive: true, mode: 0o700 })`.
2. Write to `<file>.tmp` via `fs.writeFile(..., { mode: 0o600 })`.
3. `fs.chmod(<file>.tmp, 0o600)` (defensive — some platforms ignore the writeFile mode).
4. `fs.rename(<file>.tmp, <file>)`.

On read, if directory or file mode is group/world-readable on POSIX, emit a stderr warning. Don't refuse — user might have a custom umask intentionally.

Windows: `fs.chmod` is largely a no-op. Document that the config file inherits Windows ACLs from the user's home directory; sensitive-file warnings are POSIX-only.

## Hard constraints

- **Auto-upload is non-fatal.** Failure prints a resume command and preserves the run's exit code. NEVER fail the run because the upload failed.
- **Snapshot before upload.** Never read the live `events.ndjson` while it could be written. Copy to `.upload-snapshot/` first.
- **Login nonce is mandatory.** No nonce-less callback path. No referer-only validation.
- **API key auth helper is centralized.** Every dashboard route imports from `apps/web/lib/dashboard/auth.ts`. No bespoke key validation per route.
- **Atomic config write.** Always temp-file + rename. Never partial JSON on disk.
- **Streaming chunk reads.** Don't buffer the whole events.ndjson in memory; stream-read 1 MiB at a time.
