# v5.6 — Fly.io + Render deploy adapters

**Status:** spec drafted 2026-04-30
**Builds on:** v5.4 Vercel adapter (shipped via PRs #59, #61, #63, #64) — same `DeployAdapter` contract, same factory, same auth-doctor pattern.

## What we're shipping

Two more first-class deploy adapters: **Fly.io** and **Render**. Both implement the existing `DeployAdapter` interface from `src/adapters/deploy/types.ts`, plug into the existing factory in `src/adapters/deploy/index.ts`, and integrate with the existing `--watch` log-streaming, `deploy status`, `deploy rollback`, and `deploy doctor` CLI surfaces. No new contract — just two new implementations that close the "framework-agnostic deploy" pitch.

After this, `deploy.adapter:` accepts `vercel | fly | render | generic`, covering ~80% of the cloud-deploy market that doesn't run its own platform.

## Why this matters

Today's `DeployAdapter` contract is already abstract enough that the architecture work is done. The only thing missing for the YC pitch is breadth credibility: "framework-agnostic" reads stronger when you can point at three platforms instead of one. Fly and Render are the obvious next picks because (a) both have well-documented REST APIs, (b) both target the same "deploy code without a control plane" user we already serve, and (c) neither overlaps with Vercel's frontend-heavy positioning.

## Architecture (no changes)

`DeployAdapter` interface, factory, log-streaming primitives, rollback semantics, and CLI subverbs all stay the same. Each new adapter is a single file under `src/adapters/deploy/` plus a config-validation case in the factory.

```ts
export interface DeployAdapter {
  readonly name: string;
  deploy(input: DeployInput): Promise<DeployResult>;
  status?(input: DeployStatusInput): Promise<DeployStatusResult>;
  rollback?(input: DeployRollbackInput): Promise<DeployResult>;
  streamLogs?(input: DeployStreamLogsInput): AsyncIterable<DeployLogLine>;
}
```

## Fly.io adapter

### API surface

Fly's [Machines API](https://fly.io/docs/machines/api/) is the modern path; `flyctl` itself is a wrapper. We use the REST API directly to keep the dependency surface small.

> **Pre-implementation gate (Codex CRITICAL #1, #2):** Phase 1 starts with capturing real request/response fixtures for the deploy and rollback endpoints below. The exact field names + image-binding semantics in this section are intent-level; the captured fixtures are the source of truth for the implementation. If the captured behavior diverges from this spec, the spec is updated before code.

- **Deploy (image-based):** Image must already be in the Fly registry. The adapter accepts `image` (e.g. `registry.fly.io/my-app:deployment-01`) and binds it to a new release. The exact endpoint + body shape — whether that's a `POST .../releases` with `image` in the body, a per-machine `POST .../machines/{id}` config update, or a `flyctl deploy --image` shell-out — is decided by the captured fixtures in Phase 1. Build/push of the image is the user's responsibility; we don't shell out for the build.
- **Status:** poll the deploy/release endpoint returned by the deploy call until terminal. Map Fly's terminal states → `pass | fail | in-progress`.
- **Logs:** Fly's log streaming uses WebSockets. Phase 3 opens a WS subscription, parses NDJSON-encoded log lines, fans out to the AsyncIterable. Reconnect once with exponential backoff on disconnect, then bail with a "log stream lost" warning to stderr (deploy itself continues).
  - **Machine-mapping deviation (resolved by deferral, not fix):** Phase 3's WS URL builder uses `wss://api.machines.dev/v1/apps/{app}/machines/{machineId}/logs` with `deployId` substituted as `machineId`. Fly's release→machine mapping isn't strictly 1:1 — a release can fan out to multiple machines, and the deploy ID is not a machine ID. The Phase 3 agent flagged this in code (`buildLogsWsUrl` is overridable via constructor option) so that fixture-capture against the real Fly API can either (a) drive a machine-list lookup pre-subscribe and pick one, or (b) switch to a release-scoped logs endpoint if Fly exposes one. **Until that fixture work happens, log streaming on Fly is best-effort and may attach to the wrong machine in multi-machine deployments.** Tracked for the post-merge fixture pass.
- **Rollback:** Fly historically had native rollback (`POST .../releases/{release_id}/rollback`), but the API has evolved across Nomad → Machines eras. **The adapter implements two paths**: (a) try native rollback if the API still supports it; (b) fall back to "redeploy previous known-good image" by listing prior releases, finding the most recent one with status `succeeded`, and triggering a new deploy with that image. Phase 4 captures fixtures for both paths and picks whichever is current.

### Auth

`FLY_API_TOKEN` env var (created at `fly.io/dashboard/personal/tokens`). Same shape as `VERCEL_TOKEN` — the doctor adds a check that mirrors Phase 6:

```ts
if (deployAdapter === 'fly') {
  const token = process.env.FLY_API_TOKEN;
  // warn-not-fail when missing, link to fly.io/dashboard/personal/tokens
}
```

### Config shape

```yaml
deploy:
  adapter: fly
  app: my-app                          # Fly app name
  image: registry.fly.io/my-app:latest # OR set image at deploy time via --image flag
  region: ord                          # optional pin; uses app default
  watchBuildLogs: true
  healthCheckUrl: https://my-app.fly.dev/health
  rollbackOn: [healthCheckFailure]
```

### Failure modes

Codex review (#3) flagged that the v5.4 pattern of mapping any auth-adjacent failure to `auth` is too broad. v5.6 introduces a tighter taxonomy that keys off HTTP status, not just feel-of-the-error:

| Trigger | Code | Notes |
|---|---|---|
| 401 / 403 | `auth` | Token missing/expired/revoked. Hint: regenerate at fly.io/dashboard/personal/tokens |
| 404 (app or release) | `not_found` (new code) | App slug typo, wrong org scope, or release ID belongs to another app. Hint includes the request-id from `Fly-Request-Id` header |
| 422 / 400 (config) | `invalid_config` | Missing `image`, malformed slug, region not in app's allowed list |
| 5xx | `transient_network` | Retryable per existing taxonomy |
| Image not in registry | `invalid_config` | Hint: `fly deploy --build-only --push` |
| Release failed (post-deploy) | `status: 'fail'` | Last 50 log lines (post-redaction, see below) in `output`, dashboard link in `buildLogsUrl` |
| Health check fails | bounded auto-rollback (see Health-check policy below) |  |
| WebSocket disconnect mid-stream | warning only — deploy continues | One reconnect with exp backoff, then bail to stderr |

The `not_found` error code is added to `src/core/errors.ts` (`ErrorCode` union) as part of Phase 5. Existing adapters continue to map 404 to `auth` until they're audited; v5.6 only changes the new adapters.

## Health-check policy (cross-adapter, addresses Codex #5)

`rollbackOn: [healthCheckFailure]` triggers exactly one auto-rollback per deploy attempt — never a chain. The flow:

1. Deploy succeeds, status = `pass`
2. Run health check: GET `healthCheckUrl`, accept any 2xx, retry up to 5× with 6s backoff (max ~30s window)
3. If all health checks fail → trigger rollback exactly once
4. After rollback completes (success or failure), the adapter returns; **no second rollback attempt**, regardless of whether the rollback target is itself healthy
5. Final `status` reflects the bounded outcome:
   - `pass` — deploy + health check OK
   - `fail` — deploy succeeded, health check failed, no rollback configured
   - `fail_rolled_back` (new status, added to `DeployResult.status` union) — deploy succeeded, health check failed, rollback completed
   - `fail_rollback_failed` (new) — deploy succeeded, health check failed, rollback also failed
   - `fail` (deploy itself) — deploy never reached running

The two new status values land in `src/adapters/deploy/types.ts` in Phase 5. PR comment formatter renders both clearly.

## Log redaction (cross-adapter, addresses Codex #6)

Provider logs frequently contain accidental env dumps, JWTs, and tokens. v5.4 emits "last 50 lines" verbatim into PR comments — which would leak any such secret into the public PR. v5.6 introduces a mandatory redaction pass:

- Every log line surfaced into `DeployResult.output` or PR-comment body runs through the redaction patterns already defined in `src/core/logging/redaction.ts` (`AKIA...`, `sk-...`, `eyJ...`, `ghp_`, `xoxb-`, plus user-configurable `config.persistence.redactionPatterns`)
- Full unredacted logs remain available in `.guardrail-cache/deploy-logs/<deploy-id>.ndjson` for local debugging — never copied into PR artifacts
- Tests assert that a synthetic log line containing `AKIAEXAMPLE...` is redacted to `[REDACTED:...]` in both `output` and the PR-comment string

Apply the same redaction to the v5.4 Vercel adapter as part of Phase 5 — it's a real existing hazard, not a v5.6-only concern.

## Render adapter

### API surface

Render's [REST API](https://api-docs.render.com/) is straightforward, no streaming WS.

- **Deploy:** `POST /v1/services/{serviceId}/deploys` — triggers a new deploy from the latest commit on the configured branch. Body accepts `clearCache: 'do_not_clear' | 'clear'`. Returns `id`, `commit`, `status`.
- **Status:** `GET /v1/services/{serviceId}/deploys/{deployId}` — polls `status`: `created`, `build_in_progress`, `update_in_progress`, `live`, `deactivated`, `build_failed`, `update_failed`, `canceled`. Map → `pass | fail | in-progress`. (Note: an earlier draft of this spec used the shorthand `/v1/deploys/{id}` — that endpoint does not exist on Render's API. Cursor Bugbot caught the discrepancy on PR #73 / Phase 2 and the fix landed in commit `567d6c5`.)
- **Logs:** `GET /v1/services/{serviceId}/logs?deployId=...&direction=forward&limit=100` — REST-paginated, no streaming. The adapter polls every 2s while deploy is in-progress and yields new lines via the AsyncIterable. Less elegant than Fly's WS, but adequate.
- **Rollback:** Render doesn't have a one-shot rollback API. To roll back, you re-deploy a specific commit: `POST /v1/services/{serviceId}/deploys` with `commitId`. The adapter implements rollback by listing deploys (`GET /v1/services/{serviceId}/deploys`), finding the previous successful deploy's `commit.id`, and deploying that.

### Auth

`RENDER_API_KEY` env var (created at `dashboard.render.com/u/settings#api-keys`). Doctor check mirrors Vercel/Fly pattern.

### Config shape

```yaml
deploy:
  adapter: render
  serviceId: srv-abc123                 # Render service ID
  clearCache: do_not_clear              # optional, default 'do_not_clear'
  watchBuildLogs: true
  healthCheckUrl: https://my-app.onrender.com/health
  rollbackOn: [healthCheckFailure]
```

### Failure modes

Same five categories as Fly, with one difference:

5. **Health check fails after deploy** → auto-rollback re-deploys previous commit (no native rollback verb). Surfaces both deploy IDs in the PR comment plus the commit SHAs that were swapped.

## Capability metadata (addresses Codex #4)

Codex review correctly noted that Render's REST polling is not equivalent to Fly's WebSocket streaming, and pretending they are degrades user trust. v5.6 adds an optional `capabilities` field to `DeployAdapter`:

```ts
export interface DeployAdapter {
  readonly name: string;
  readonly capabilities?: {
    streamMode?: 'websocket' | 'polling' | 'none';
    /** Native rollback verb available, vs. simulated via re-deploy */
    nativeRollback?: boolean;
  };
  // ... existing methods
}
```

When `--watch` is used against a `polling` adapter, the CLI prints a one-line notice to stderr before iteration begins:

```
[deploy] note: render uses 2s log polling — lines may arrive in batches and could include short gaps. See docs/deploy/adapters.md#log-streaming for details.
```

Adapters with `streamMode: 'polling'` MUST cursor by `(timestamp, logId)` to dedupe and avoid gaps under pagination. Tests assert no duplicates and no missed lines across pagination boundaries.

## CLI changes

Zero new flags. The existing `deploy`, `deploy status`, `deploy rollback`, `deploy doctor` subverbs already work against any registered adapter via the factory. Two minor surface updates:

```diff
- --adapter <vercel|generic>   Override deploy.adapter from config
+ --adapter <vercel|fly|render|generic>   Override deploy.adapter from config
```

The polling-mode notice (see Capability metadata above) is the only new stderr output.

## Implementation phases

1. **Phase 1 (~3h)** — `src/adapters/deploy/fly.ts` with deploy + status. POST release, poll status. Auth + error mapping.
2. **Phase 2 (~3h)** — `src/adapters/deploy/render.ts` with deploy + status. Same shape, different endpoints.
3. **Phase 3 (~2h)** — Log streaming for both. Fly via WebSocket, Render via REST polling. Both wired to `--watch`.
4. **Phase 4 (~2h)** — Rollback for both. Fly via native `/rollback`; Render via list-deploys + re-deploy-prior-commit.
5. **Phase 5 (~1h)** — Factory registration + config validation in `src/adapters/deploy/index.ts`. CLI help text update. Tests for the factory error paths.
6. **Phase 6 (~1h)** — Doctor extension: detect `adapter: fly` → check `FLY_API_TOKEN`; detect `adapter: render` → check `RENDER_API_KEY`. Same shape as Phase 6 of v5.4.
7. **Phase 7 (~1h)** — Docs: README adapter list updated, `skills/migrate/SKILL.md`–style examples added to a new `docs/deploy/adapters.md`.

**Total: ~13h.** Roughly 1.3× the Vercel work because there's two of them, but each one is half the design effort because the contract is settled.

## Test plan

- **Unit (per adapter):** 12+ tests against mocked `fetch`/WS — success, build-fail, 401/403/404/422/5xx mapping, network-blip retry, log-streaming, rollback (both native and simulated paths). Same fixtures pattern as `tests/adapters/deploy/vercel.test.ts`.
- **Race / concurrency tests (Codex #8):** per adapter, add 3 cases —
  1. Two concurrent `deploy()` calls on the same adapter instance must not corrupt shared state (or must explicitly serialize)
  2. Out-of-order status transitions (`succeeded` arriving before `running`) handled deterministically
  3. `rollback()` invoked when no successful prior release exists → returns `no_previous_deploy` cleanly, no fall-through to a non-existent ID
- **Redaction tests:** synthetic log line `AKIAIOSFODNN7EXAMPLE secret token` must appear redacted in `DeployResult.output` and the PR-comment string for both adapters.
- **Capability surface tests:** `streamMode: 'polling'` adapter prints the one-line notice to stderr; `streamMode: 'websocket'` adapter does not.
- **Integration:** 1 test per platform against real APIs with sandbox accounts (skipped in CI without `FLY_API_TOKEN_TEST` / `RENDER_API_KEY_TEST`).
- **E2E:** real deploy of a hello-world app on each platform during release — ~$0 cost on free tiers, ~30-60s wall clock.

## Per-adapter config schema (addresses Codex #7)

Existing `DeployConfig` is a flat union. v5.6 introduces strict per-adapter required-fields validation in the factory, with field-named error messages:

| Adapter | Required | Optional |
|---|---|---|
| `vercel` | `project` | `team`, `target` |
| `fly` | `app`, `image` | `region`, `watchBuildLogs`, `healthCheckUrl`, `rollbackOn` |
| `render` | `serviceId` | `clearCache` (enum: `do_not_clear` \| `clear`), `watchBuildLogs`, `healthCheckUrl`, `rollbackOn` |
| `generic` | `deployCommand` | `healthCheckUrl` |

Factory throws `GuardrailError({ code: 'invalid_config' })` with the exact missing field name, e.g. `"deploy.adapter=fly requires deploy.app (Fly app slug)"`. Mirrors the v5.4 Vercel error.

## What v5.6 deliberately does NOT do

- **Heroku adapter.** Heroku's git-push-to-deploy model maps to the generic adapter cleanly; no first-class adapter needed.
- **AWS / GCP / Azure.** Their deploy stories are too divergent (CDK, Cloud Run, App Service, ECS, Lambda, etc.) to fit one adapter shape. Future work, possibly v6.
- **Build step.** Adapters trigger a deploy from a pre-built image (Fly) or a git ref (Render); building the image / committing the code is the user's responsibility, same as v5.4.
- **Per-environment Fly/Render configs.** v5.6 ships single-environment adapters. Multi-env (production vs preview) is doable but not required for the framework-agnostic pitch.

## Open questions for review

1. Should the Fly adapter shell out to `flyctl deploy` as a fallback when `FLY_API_TOKEN` is missing but `flyctl` is installed and authenticated? Mirrors `gh` CLI fallback in vcs-host. Adds a shell-injection surface we don't currently have.
2. Render's WebSocket-less log experience is noticeably worse than Fly's. Should we degrade `--watch` for Render to "polling banner every 5s" with a one-line warning, or pretend the experience is uniform?
3. Auto-rollback on Render replays a previous deploy. The new deploy ID will differ from the prior one. PR comment should make clear: "rolled back to commit X (new deploy ID Y)".
