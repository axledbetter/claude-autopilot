# v7.1 Ingest-API JWT membership re-check

> **Status:** spec — implementation pending PR.
> **Track:** v7.1 follow-ups to v7.0 hosted product MVP.
> **Builds on:** Phase 2.2 (ingest API + signed-session JWTs), Phase 5.7 (admin lifecycle / `memberships.status`), Phase 5.8 (API-key org-membership check), Phase 6 (cookie-auth ≤60s revocation).
>
> **Deployment target:** `apps/web/` is the autopilot.dev hosted product, deployed on **Vercel**. (Recurring codex false-positive note: this is NOT the AWS-ECS-deployed `delegance-app` repo. Do NOT rewrite for ECS.)

## Goal

Close the symmetric gap that Phase 6 explicitly deferred: Phase 2.2's
upload-session JWTs (15min TTL) keep working for members who get
disabled mid-session. Phase 6 collapsed the cookie-authenticated
revocation window from ≤60min to ≤60s; this spec collapses the
JWT-authenticated ingest revocation window from ≤15min to **≤1
request** for org-scoped runs.

## Scope

**Single PR. Ingest API only.**

1. **JWT mint embeds membership status snapshot.** At
   `POST /api/upload-session`, after the existing entitlement check
   succeeds, look up the org membership status for `(r.organization_id,
   r.user_id)`. Embed as new claim `mint_status: 'active'` (always
   `'active'` — if the lookup returns anything else, mint is refused
   with `403 member_not_active` and audit-logged). Personal runs
   (`r.organization_id IS NULL`) skip the lookup and embed
   `mint_status: 'personal'`.

2. **Re-check at every event-write.** In
   `PUT /api/runs/:runId/events/:seq`, after `verifyUploadToken()`
   succeeds, call new helper `assertActiveMembership(claims)` BEFORE
   `claim_chunk_slot`. Returns 403 `member_disabled` (or
   `member_inactive`/`no_membership`) if the current
   `memberships.status` for `(claims.org_id, claims.sub)` is anything
   other than `'active'`. Personal runs skip the check
   (`claims.mint_status === 'personal'`).

3. **Re-check at finalize.** Same helper, same point in flow:
   `POST /api/runs/:runId/finalize`, after `verifyUploadToken()`,
   before the chain-verify + manifest write.

4. **Re-use existing `check_membership_status` RPC** from Phase 6 — no
   new migration. The RPC is already SECURITY INVOKER + service-role-
   only and returns `{status, role, checked_at}`.

5. **No JWT secret rotation.** The existing JWT secret + 15min TTL
   stay unchanged. The `mint_status` claim is additive and is
   observability-only — `claims.org_id` remains the sole authority
   for membership-check dispatch (codex pass-1 CRITICAL #2). v7.0
   tokens still minted with `org_id: '<uuid>'` get the new check
   on every event-write/finalize after deploy.

## Non-goals

- **Not** lowering JWT TTL below 15min. The TTL is a hard cap on
  blast radius if our middleware fails open; the per-request re-check
  collapses the typical case to ≤1 request.
- **Not** revoking in-flight JWT tokens server-side. There's no
  blacklist; the per-request re-check is the enforcement mechanism.
- **Not** wiring the membership check into the dashboard runs API
  routes (already done by Phase 5.8).
- **Not** wiring the check into Phase 5.6 SSO callback (already
  refuses disabled members at sign-in via `record_workos_sign_in`).
- **Not** caching the membership-status RPC result per request (the
  ingest endpoints make one RPC call per HTTP request — adding cache
  state would complicate without meaningful win because each request
  = 1 chunk = ~256KB of state at most).

## Architecture

### Mint-time snapshot + per-request re-check

```
CLI mint:                                   CLI event-write:
POST /api/upload-session              PUT /api/runs/:id/events/:seq
  ├── auth (cookie or API key)          ├── verifyUploadToken(JWT)
  ├── runs lookup                       ├── assertActiveMembership(claims)  ← NEW
  ├── entitlement check                 │     ├── if claims.mint_status==='personal': allow
  ├── check_membership_status RPC       │     ├── checkMembershipStatus(org_id, sub) — Phase 6 helper
  │     ├── status='active': continue   │     ├── 'active' → allow
  │     ├── 'disabled'/etc: 403         │     ├── 'disabled' → 403 member_disabled
  │     │   member_not_active           │     ├── 'inactive'|'invite_pending' → 403 member_inactive
  │     └── personal run: skip          │     ├── 'no_row' → 403 no_membership
  ├── mint JWT with                     │     └── RPC error/timeout → 403 member_check_failed
  │     {mint_status: 'active'|         ├── claim_chunk_slot RPC (existing)
  │      'personal', ...rest}           ├── upload to Storage (existing)
  └── insert upload_sessions row        └── mark_chunk_persisted RPC (existing)
```

**Why the mint-time snapshot (codex pass-2 NOTE #2):** observability
only — `claims.org_id` is the sole authorization-dispatch signal.
The `mint_status` claim is logged on every ingest event so operators
can correlate revocations of "members who were active at mint" vs
"members who were already disabled but somehow got a JWT" (the
latter shouldn't happen because mint refuses non-active members).

**Why fail-closed for RPC errors:** mirrors the Phase 6 middleware
rationale — a Supabase outage during ingest is rare; locking out
ingest for 1 RPC TTL (1.5s timeout same as Phase 6) per attempt is
preferable to silently letting disabled members continue uploading.

**Why no separate JWT-blacklist table:** the per-request
`check_membership_status` RPC IS the revocation. A blacklist would
add write amplification on every disable_member call (one row per
in-flight session) for a sub-second saving vs the existing approach.

## Helper module

New `apps/web/lib/upload/membership-recheck.ts`:

```typescript
import { checkMembershipStatus, MembershipCheckError }
  from '@/lib/supabase/check-membership';
import type { UploadTokenClaims } from './jwt';

export type IngestMembershipReason =
  | 'member_disabled'
  | 'member_inactive'
  | 'no_membership'
  | 'member_check_failed';

export class IngestMembershipError extends Error {
  constructor(public readonly reason: IngestMembershipReason) {
    super(`ingest membership refused: ${reason}`);
  }
}

/**
 * Assert that the JWT principal is an active member of the run's org.
 *
 * Authority: `claims.org_id`. Personal runs (org_id absent / null /
 * empty string) skip the check; org-scoped runs ALWAYS call the RPC,
 * regardless of the (cosmetic) `mint_status` claim. This is the codex
 * pass-1 CRITICAL #2 fix — mint_status is observability-only, NOT
 * authorization. (Codex pass-1 CRITICAL #1: explicit fail-closed
 * default branch on unknown statuses.)
 */
export async function assertActiveMembership(claims: UploadTokenClaims): Promise<void> {
  // Personal-run shortcut. org_id is the SOLE authority.
  if (!claims.org_id) return;

  let result;
  try {
    result = await checkMembershipStatus(claims.org_id, claims.sub);
  } catch (err) {
    // Codex pass-3 WARNING #4 — broad catch: ANY RPC/network/timeout
    // failure becomes member_check_failed (retryable 503). Programmer
    // errors (TypeError on undefined etc.) bubble up as 500 — no
    // attempt to silence those.
    if (err instanceof MembershipCheckError) {
      throw new IngestMembershipError('member_check_failed');
    }
    // Non-MembershipCheckError but also non-programmer (network/timeout)
    // — wrap defensively so one ill-classified Supabase error doesn't
    // turn into a 500.
    if (err && typeof err === 'object' && 'message' in err) {
      logger.warn({ msg: 'ingest.unexpected_rpc_error', err: String((err as Error).message) });
      throw new IngestMembershipError('member_check_failed');
    }
    throw err;
  }

  switch (result.status) {
    case 'active':         return;
    case 'disabled':       throw new IngestMembershipError('member_disabled');
    case 'inactive':
    case 'invite_pending': throw new IngestMembershipError('member_inactive');
    case 'no_row':         throw new IngestMembershipError('no_membership');
    default:
      // Codex pass-1 CRITICAL #1 — fail-closed on any unknown status.
      // Codex pass-2 NOTE #3 — structured log line (run_id, jti, org_id
      // included; sub deliberately omitted to avoid PII spread).
      logger.warn({
        msg: 'ingest.unknown_membership_status',
        status: result.status,
        runId: claims.run_id,
        jti: claims.jti,
        orgId: claims.org_id,
      });
      throw new IngestMembershipError('member_check_failed');
  }
}
```

The helper is the single chokepoint — both event-write + finalize
import and call it identically. Future ingest endpoints touching
org-scoped runs MUST call this helper (added to the
`lib/dashboard/auth.ts` audit comment block).

**Wrapper enforcement (codex pass-2 WARNING #5; pass-3 CRITICAL #1+#2):**
introduce `apps/web/lib/upload/auth.ts` exporting one orchestrator
that EVERY ingest route calls:

```typescript
export async function verifyTokenAndAssertRunMembership(
  rawToken: string,
  routeRunId: string,
  supabase: SupabaseServiceClient,
): Promise<{ claims: UploadTokenClaims; run: RunRow }> {
  // 1. JWT shape + signature.
  const claims = verifyUploadToken(rawToken);
  // 2. Path/JWT run_id consistency (existing invariant — surfaced here).
  if (claims.run_id !== routeRunId) {
    throw new IngestMembershipError('run_mismatch');
  }
  // 3. Persisted-run lookup — single source of truth for organization_id.
  const { data: run } = await supabase.from('runs')
    .select('id, user_id, organization_id, deleted_at')
    .eq('id', routeRunId)
    .maybeSingle();
  if (!run || run.deleted_at) throw new IngestMembershipError('run_not_found');
  // 4. JWT org_id MUST match persisted run.organization_id (codex pass-3
  //    CRITICAL #2 — closes cross-org JWT replay AND personal-shortcut
  //    bypass). Both must be NULL/empty (personal) or both non-null
  //    matching uuid (org-scoped).
  const tokenOrg = claims.org_id || null;
  if (tokenOrg !== run.organization_id) {
    throw new IngestMembershipError('run_org_mismatch');
  }
  // 5. Membership re-check (only for org-scoped — personal short-circuits
  //    safely now that step 4 proves the run is genuinely personal).
  await assertActiveMembership(claims);
  return { claims, run };
}
```

Both ingest routes (events + finalize) MUST call
`verifyTokenAndAssertRunMembership()` and use the returned
`{claims, run}`. The two new error reasons (`run_mismatch`,
`run_not_found`, `run_org_mismatch`) all map to **404
not_found** at the HTTP layer (no enumeration leakage) per existing
ingest convention.

This eliminates the "personal-shortcut bypass" (pass-3 CRITICAL #2)
because step 4 proves the persisted run is genuinely personal
BEFORE step 5's `if (!claims.org_id) return;` shortcut runs. It also
covers finalize (pass-3 CRITICAL #1) because finalize calls the
SAME orchestrator.

**No SQL migration needed (codex pass-3 WARNING #1+#2 corrections):**
the persisted-run consistency check moves from "extend
`claim_chunk_slot`" to "do a `runs.select` in TypeScript before
side-effect RPCs." Avoids changing RPC signatures, avoids `uuid =
text` type pitfalls, avoids coordinating SQL-side null semantics.
The previous `data/deltas/20260509220000_phase7_1_*` migration is
**dropped** from the spec.

**Pre-existing route-side `runs` lookup**: both event-write and
finalize already do a `runs` lookup early in the handler (event-write
to fetch organization_id for storage path; finalize to fetch
state). The orchestrator pulls those lookups into one shared call
that ALSO performs the consistency assertion — net code reduction,
not addition.

**Enforcement (codex pass-3 WARNING #5):** primary enforcement is
that `verifyUploadToken()` becomes file-private to `lib/upload/jwt.ts`
(unexported); only `verifyTokenAndAssertRunMembership()` is the
public API. ESLint `no-restricted-imports` rule on
`apps/web/app/api/runs/**` rejects any direct import from
`@/lib/upload/jwt`. The grep test stays as defense-in-depth but is
not the primary mechanism.

**Mint-time vs ingest-time identity invariant (codex pass-1 WARNING #1):**
the JWT `sub` is always set to the user whose membership was checked
at mint time (`r.user_id`). For API-key-authenticated mints,
`r.user_id` is the run's owner (the original creator), NOT the
API-key holder if those differ — this matches Phase 5.8's
authorization model where API keys are scoped to the user that
created them and the run's `user_id` IS the API-key owner. Test
asserts this invariant.

## JWT claim addition

`UploadTokenClaims` interface extended (codex pass-1 CRITICAL #3 — `org_id`
type matches `MintInput.orgId`):

```typescript
export interface UploadTokenClaims {
  sub: string;
  run_id: string;
  /** Organization ID for org-scoped runs; null/empty for personal.
   *  Authority for membership-recheck dispatch (truthy → check). */
  org_id: string | null;
  jti: string;
  aud: string;
  iss: string;
  exp: number;
  iat: number;
  /** Observability-only snapshot of membership status at mint time.
   *  NOT used for authorization (codex pass-1 CRITICAL #2). */
  mint_status?: 'active' | 'personal';
}
```

`MintInput` extended:

```typescript
export interface MintInput {
  userId: string;
  runId: string;
  orgId: string | null;
  jti: string;
  mintStatus: 'active' | 'personal';     // NEW — required
}
```

Existing serialization keeps personal runs as `org_id: ''` on the
wire to match the v7.0 token shape; verify normalizes empty string
to `null` so consumers always see `string | null`. Helper's `if
(!claims.org_id)` shortcut handles both.

Mint is updated to thread `mintStatus` through. Verify is unchanged
in required-claims set (`mint_status` stays optional for forward-
compat with v7.0 tokens during rollout).

## Rollout safety

The JWT format change is **forward-only and backward-compatible** —
revised after codex pass-1 CRITICAL #2:

- v7.0 tokens (no `mint_status`): helper STILL runs the membership
  RPC if `claims.org_id` is truthy, because `org_id` is the sole
  authorization authority. Effect during rollout: every in-flight
  v7.0 org-scoped session is now bound to the new revocation rule
  immediately on the first event-write/finalize after deploy. No
  rollout window where org-scoped tokens skip the check.
- v7.0 personal-run tokens (`org_id: ''` empty string): helper
  short-circuits via `!claims.org_id` falsy check.
- v7.1 tokens: mint embeds `mint_status` for observability/audit.
  Authorization behaves identically to v7.0 tokens of the same
  org-scoped/personal shape.

**No coordinated cutover required.** The `mint_status` claim is
purely observability — operators can grep ingest logs to distinguish
revocations of "members who were active at mint" from "members who
were already disabled but somehow got a JWT" (the latter shouldn't
happen because mint refuses non-active members per Scope item #1).

**HTTP status for transient failures (codex pass-1 WARNING #2):**
the event-write + finalize routes return **`503 member_check_failed`**
(NOT `403`) when the helper throws `IngestMembershipError('member_check_failed')`.
Permanent membership refusals (`member_disabled`, `member_inactive`,
`no_membership`) keep `403`. CLI uploader's existing 5xx-retry path
covers the 503 case automatically; 403s remain non-retryable.

## Observability + capacity notes

(Codex pass-3 NOTE #1+#2+#3 — folded inline rather than deferred.)

**Structured log contract** for ingest event-write + finalize:
every request emits one `info` line on success and one `warn` line
on `IngestMembershipError`, both with fields:

```
{ msg, runId, jti, orgId, mintStatus, endpoint, result, durationMs }
```

`mint_status` is logged on success too — operators can filter
`result=member_disabled mintStatus=active` to see "members who were
active at mint, then disabled mid-session" vs `mintStatus=active
result=success` baseline.

**Capacity:** 1 `check_membership_status` RPC per ingest chunk =
~256KB. A 1GB upload → ~4096 RPC calls; at typical
`apps/web/lib/supabase/check-membership.ts` 50-200ms latency that's
~2-13min added to the upload wall clock if chunks are serial. CLI
uploader uses concurrent chunks (~4 in flight), so realistic
worst-case is ~30s added per 1GB. For typical autopilot-run
state-export uploads (~1-50 MB), the added latency is sub-second.
Alert thresholds: `member_check_failed` rate > 1/min sustained
indicates Supabase outage; `member_disabled` rate > 5/min sustained
indicates either a script revoking memberships in a loop or a real
attack pattern.

## Post-launch follow-ups (v7.2+)

- Configurable `MEMBERSHIP_CHECK_TTL_SECONDS` env var so enterprise
  customers can tighten the 60s cookie cache from Phase 6.
- Dual-secret rotation for `MEMBERSHIP_CHECK_COOKIE_SECRET`.
- Server-side cache invalidation on `change_member_role` /
  `disable_member` (collapses Phase 6 ≤60s cookie window to instant).
- Sentry / observability for `member_check_failed` rate to
  distinguish "Supabase outage" from "real revocation events".
