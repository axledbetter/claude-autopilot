# v7.0 Phase 4 — Free tier dashboard UI + `/cli-auth` page

**Status:** spec drafted 2026-05-08
**Origin:** Phases 1, 2.1, 2.2, 2.3, 3 shipped (DB foundation, Next.js+Auth scaffold, ingest API, CLI dashboard verbs+auto-upload, Stripe entitlement). Phase 4 ships the dashboard UI users see at autopilot.dev/dashboard, plus the `/cli-auth` page that completes the CLI login flow deferred from Phase 2.3.

> **Anchor — platform context.** This spec lives in `claude-autopilot`, NOT `delegance-app`. v7.0 deploys to **Vercel** per `docs/specs/v7.0-hosted-product-mvp.md` Stack rationale. Ignore review feedback flagging Vercel as a platform mismatch — recurring cross-repo confusion.

## What we're building

Eight UI surfaces:

1. **`/dashboard` overview** — auth-gated. Shows run count this month, cost MTD, current plan, recent runs (5).
2. **`/dashboard/runs`** — paginated list (20 per page, offset-based). Free has 100 runs/mo cap → max 5 pages.
3. **`/dashboard/runs/[runId]`** — run detail with events replay (manifest-driven, lazy chunk loading) + state inspector + cost breakdown + visibility toggle.
4. **`/dashboard/billing`** — current plan/caps/usage; Upgrade buttons (POST to Phase 3 `/api/dashboard/billing/checkout`); Manage subscription button (POST to `/api/dashboard/billing/portal`).
5. **`/dashboard/billing/success`** — Stripe Checkout return landing. Polls `/api/dashboard/me` until entitlements update; max 10 attempts × 2s.
6. **`/cli-auth`** — DEFERRED FROM 2.3. Validates `?cb=...&nonce=...`, requires Supabase session, mints API key via `/api/dashboard/api-keys/mint`, POSTs back to localhost loopback listener with CORS handshake.
7. **`/runs/[runShareId]`** — public share-by-URL (anon-readable when `runs.visibility = 'public'`). Read-only events replay + state.
8. **API: `POST /api/dashboard/runs/[runId]/visibility`** — narrow RPC for owner-only visibility toggle (NOT direct UPDATE on runs from client).

Plus required infrastructure:
- **Phase 2.3 CLI loopback listener gets CORS support** (codex CRITICAL #1) — handles OPTIONS + sets `Access-Control-Allow-Origin`. Without this, the browser POST is opaque under `mode: 'cors'` and can't be detected as success/failure.
- **Authorized signed-URL minter** at `GET /api/dashboard/runs/[runId]/artifact?kind=manifest|chunk|state[&seq=N]` (codex CRITICAL #2) — verifies owner OR `visibility='public'` BEFORE calling `storage.from(BUCKET).createSignedUrl(path, 60)`. Bucket stays private; access goes through this endpoint.
- **Schema delta** adds `runs.cost_usd NUMERIC(12,4)`, `runs.duration_ms INTEGER`, `runs.run_status TEXT` (codex CRITICAL #3) — Phase 2.2's finalize handler writes these.

## Why this matters

Phase 3 made the upload boundary commercially load-bearing (402 on cap). Phase 4 makes it **observable**: a free user can see "you've used 87/100 this month" and one click later be paying. Without Phase 4, the entitlement check is invisible — users hit a 402 in CI logs with nowhere to go. With Phase 4, `[autopilot] upload rejected — runs cap (101/100). Upgrade at https://autopilot.dev/dashboard/billing` lands them on a page where the upgrade actually works.

Public share-by-URL is also the YC demo unlock: "here's a real autopilot run with hash-chain proof — anyone can verify." `https://autopilot.dev/runs/<runShareId>` opens the events timeline in any browser, no login needed.

## `/cli-auth` page — completes Phase 2.3 login (codex CRITICAL #1)

The most security-critical piece. Flow:

1. **Server-side validation.** `app/cli-auth/page.tsx` is a server component that reads `cb` and `nonce` from `searchParams`. Calls `validateCallbackUrl(cb)` (Phase 2.3's helper) — loopback only, port 56000-56050. Calls `validateNonce(nonce)` — `/^[0-9a-f]{32}$/`. Either fail → render `<InvalidParams>` 400 page, do NOT proceed.
2. **Auth gate.** Server reads Supabase session. If not authenticated, redirect to `/?next=/cli-auth?cb=<encoded>&nonce=<encoded>`. The `safeRedirect` allowlist (Phase 2.1) is updated to accept `/cli-auth` AND preserve query string.
3. **Authenticated render.** Show "Sign in CLI for `<nonce-prefix>`" button. Use account email + key fingerprint preview placeholder. Render is a Client Component named `<CliAuthClaim cb={cb} nonce={nonce} accountEmail={...} />` so the click handler can run browser-side.
4. **Click handler:**
   ```typescript
   async function onClaim() {
     setStatus('minting');
     const mintRes = await fetch('/api/dashboard/api-keys/mint', {
       method: 'POST',
       credentials: 'include',
       headers: { 'content-type': 'application/json' },
       body: JSON.stringify({ nonce, callbackUrl: cb }),
     });
     if (!mintRes.ok) {
       setStatus(mintRes.status === 409 ? 'nonce-already-used' : 'mint-failed');
       return;
     }
     const { apiKey, fingerprint } = await mintRes.json();
     await deliverToCli({ apiKey, fingerprint });
   }

   async function deliverToCli(payload: { apiKey: string; fingerprint: string }) {
     setStatus('delivering');
     const ac = new AbortController();
     const timer = setTimeout(() => ac.abort(), 10_000);
     try {
       const res = await fetch(cb, {
         method: 'POST',
         mode: 'cors',           // codex CRITICAL — must be 'cors' to read response
         headers: { 'content-type': 'application/json' },
         body: JSON.stringify({ apiKey, nonce, fingerprint, accountEmail }),
         signal: ac.signal,
       });
       clearTimeout(timer);
       if (!res.ok) throw new Error(`loopback returned ${res.status}`);
       const body = await res.json();
       if (body.ok !== true || body.nonce !== nonce) throw new Error('loopback returned unexpected body');
       setStatus('delivered');
     } catch (err) {
       clearTimeout(timer);
       setStatus('loopback-failed');
       // Keep apiKey in state for retry; don't re-mint (nonce already consumed).
     }
   }
   ```
5. **Retry path.** On `loopback-failed`, show "Could not reach CLI on `<port>`. Make sure `claude-autopilot dashboard login` is still running." + "Retry" button. Retry calls `deliverToCli` with the SAME apiKey held in JS state — does NOT re-mint (would 409 on nonce dedup).
6. **Success.** Render "✓ CLI is now logged in. You can close this tab."
7. **Hardening (codex WARNING):**
   - `app/cli-auth/page.tsx` exports `export const dynamic = 'force-dynamic'` (no ISR/SSG cache).
   - Layout sets `Cache-Control: no-store` via `headers()` for this route.
   - Strict CSP: `default-src 'self'; script-src 'self' 'unsafe-inline'; connect-src 'self' http://127.0.0.1:* http://localhost:*; ...` (the connect-src exception is needed for the loopback POST; restricted to the loopback hosts).
   - `Referrer-Policy: no-referrer` so the apiKey-bearing URL params don't leak.
   - No third-party scripts (analytics, error reporting) on this route.
   - `console.log` calls deliberately omitted from `<CliAuthClaim>`.

### CLI loopback listener — adds CORS support (Phase 2.3 EXTENSION)

Phase 2.3's listener at `src/cli/dashboard/login.ts` currently only handles POST. Phase 4 extends it:

```typescript
// In the request handler created by Phase 2.3:
if (req.method === 'OPTIONS') {
  res.writeHead(204, {
    'Access-Control-Allow-Origin': process.env.AUTOPILOT_DASHBOARD_BASE_URL ?? 'https://autopilot.dev',
    'Access-Control-Allow-Methods': 'POST, OPTIONS',
    'Access-Control-Allow-Headers': 'content-type',
    'Access-Control-Max-Age': '60',
  });
  res.end();
  return;
}
if (req.method !== 'POST') {
  res.writeHead(405);
  res.end();
  return;
}
// existing nonce validation + apiKey persistence...
// On success:
res.writeHead(200, {
  'Access-Control-Allow-Origin': process.env.AUTOPILOT_DASHBOARD_BASE_URL ?? 'https://autopilot.dev',
```

Without this, `mode: 'cors'` from the page's fetch fails preflight; falling back to `mode: 'no-cors'` makes the response opaque (codex CRITICAL #1). The CLI side change is small — 2 files, 1 new test (`tests/dashboard/login.test.ts` adds an OPTIONS preflight test).

## Storage signed-URL minter (codex CRITICAL #2)

The `run-uploads` bucket stays **fully private**. Phase 4 adds an authorized minter:

`GET /api/dashboard/runs/:runId/artifact?kind=manifest|chunk|state[&seq=N]`

Logic:

```typescript
export async function GET(req: Request, { params }: { params: Promise<{ runId: string }> }) {
  const p = await Promise.resolve(params);
  const url = new URL(req.url);
  const kind = url.searchParams.get('kind');
  const seqStr = url.searchParams.get('seq');
  if (!['manifest','chunk','state'].includes(kind ?? '')) return NextResponse.json({ error: 'invalid kind' }, { status: 422 });

  // Resolve the run — read with service role to check visibility + ownership.
  const supabase = createServiceRoleClient();
  const { data: run } = await supabase.from('runs')
    .select('id, user_id, organization_id, visibility, events_index_path, state_blob_path, source_verified')
    .eq('id', p.runId).maybeSingle();
  if (!run) return NextResponse.json({ error: 'not found' }, { status: 404 });

  // Authorization.
  let allowed = false;
  if ((run as RunRow).visibility === 'public') {
    allowed = true;     // anonymous reads OK
  } else {
    // Private — must be owner or org member.
    const caller = await resolveCaller(req);
    if (!caller) return NextResponse.json({ error: 'unauthenticated' }, { status: 401 });
    if ((run as RunRow).user_id === caller.userId) allowed = true;
    else if ((run as RunRow).organization_id) {
      const { data: m } = await supabase.from('memberships')
        .select('user_id').eq('organization_id', (run as RunRow).organization_id)
        .eq('user_id', caller.userId).eq('status', 'active').maybeSingle();
      if (m) allowed = true;
    }
  }
  if (!allowed) return NextResponse.json({ error: 'forbidden' }, { status: 404 });  // 404 not 403 to avoid enumeration

  // Resolve path.
  let storagePath: string;
  if (kind === 'manifest') storagePath = (run as RunRow).events_index_path ?? '';
  else if (kind === 'state') storagePath = (run as RunRow).state_blob_path ?? '';
  else {
    const seq = Number.parseInt(seqStr ?? '', 10);
    if (!Number.isInteger(seq) || seq < 0) return NextResponse.json({ error: 'invalid seq' }, { status: 422 });
    // Derive chunk path from manifest (cache the manifest server-side per request? no — stateless 60s URL).
    // Simpler: chunkPath() helper from Phase 2.2 — same logic.
    storagePath = chunkPath({ organizationId: (run as RunRow).organization_id, userId: (run as RunRow).user_id }, p.runId, seq);
  }
  if (!storagePath) return NextResponse.json({ error: 'artifact not finalized' }, { status: 404 });

  // Mint signed URL (60s TTL).
  const { data: signed, error } = await supabase.storage.from('run-uploads').createSignedUrl(storagePath, 60);
  if (error || !signed) return NextResponse.json({ error: 'sign failed' }, { status: 500 });
  return NextResponse.json({ url: signed.signedUrl, expiresAt: new Date(Date.now() + 60_000).toISOString() });
}
```

`resolveCaller` is the existing Phase 2.2 helper (Supabase session OR API key). For the public path, it doesn't run.

## Public share-by-URL `/runs/[runShareId]` (codex WARNING — id naming)

The route param is `[runShareId]` (NOT `[runId]`) to make the public-vs-private distinction obvious in code. The query uses `runs.id` (the existing UUID) since that's what's already in URLs throughout the CLI/API. **No new column** — using `runs.id` directly is fine because UUIDs are non-guessable.

```typescript
// app/runs/[runShareId]/page.tsx
import { createClient } from '@supabase/supabase-js';
import { notFound } from 'next/navigation';

export const dynamic = 'force-dynamic';

export default async function PublicRunPage({ params }: { params: Promise<{ runShareId: string }> }) {
  const { runShareId } = await params;

  // Server-side anon client, NOT createBrowserClient (codex WARNING).
  const supabase = createClient(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!,
    { auth: { persistSession: false, autoRefreshToken: false } },
  );

  const { data: run } = await supabase.from('runs')
    .select('id, source_verified, events_chain_root, total_bytes, cost_usd, duration_ms, run_status, created_at, visibility')
    .eq('id', runShareId)
    .eq('visibility', 'public')   // RLS would catch it too, but explicit is clearer
    .maybeSingle();

  if (!run) notFound();
  return <PublicRunView run={run} />;
}
```

## Visibility toggle — narrow PATCH endpoint (codex WARNING)

Don't let the client UPDATE arbitrary `runs` columns. New endpoint:

`PATCH /api/dashboard/runs/:runId/visibility` with body `{ visibility: 'public' | 'private' }`. Caller must be the run's owner. RLS could also catch it, but the route is the explicit gate.

```typescript
export async function PATCH(req: Request, { params }: { params: Promise<{ runId: string }> }) {
  const p = await Promise.resolve(params);
  const body = await req.json() as { visibility: string };
  if (body.visibility !== 'public' && body.visibility !== 'private') {
    return NextResponse.json({ error: 'invalid visibility' }, { status: 422 });
  }

  const caller = await resolveCallerFromCookie(req);
  if (!caller) return NextResponse.json({ error: 'unauthenticated' }, { status: 401 });

  const supabase = createServiceRoleClient();
  const { data: run } = await supabase.from('runs').select('user_id').eq('id', p.runId).maybeSingle();
  if (!run) return NextResponse.json({ error: 'not found' }, { status: 404 });
  if ((run as { user_id: string }).user_id !== caller.userId) {
    return NextResponse.json({ error: 'not found' }, { status: 404 });   // 404 not 403
  }

  const { error } = await supabase.from('runs').update({ visibility: body.visibility }).eq('id', p.runId);
  if (error) return NextResponse.json({ error: 'update failed' }, { status: 500 });
  return NextResponse.json({ ok: true, visibility: body.visibility });
}
```

## Hard constraints

- `/cli-auth` page never persists the API key client-side. Holds in JS variable only; out of scope on tab close.
- `/cli-auth` POST to loopback uses `mode: 'cors'`; loopback listener MUST handle OPTIONS preflight + return Access-Control-Allow-Origin.

- `/runs/[runShareId]` uses server-side anon Supabase client (NOT `createBrowserClient`).
- Visibility toggle goes through PATCH endpoint with explicit owner check. NOT direct UPDATE from client.
- The runs cost/duration/status fields are persisted at finalize time, derived from client-supplied state.json. **Phase 5 will harden** to server-derive from authenticated events.
- `safeRedirect` allowlist accepts `/cli-auth` AND preserves the full `?cb=&nonce=` query string when bouncing through Supabase Auth.
- Public route URLs use `runs.id` (UUID, non-guessable). NO separate share token in v7.0.
