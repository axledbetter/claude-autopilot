### Task 1 ŌĆö Adapter: rollback + listDeployments

**File:** `src/adapters/deploy/vercel.ts`

Append three additions to `VercelDeployAdapter` (no contract changes):

1. **Public type `VercelDeployListItem`** at the top of the file (next to
   `VercelDeployResponse`):
   ```ts
   export interface VercelDeployListItem {
     id: string;
     url?: string;
     state?: VercelState;
     createdAt?: number;
   }
   ```

2. **`async rollback(input: DeployRollbackInput): Promise<DeployResult>`** ŌĆö implements the existing optional contract method.
   - Capture `start = this.now()`.
   - If `input.to` provided, set `targetId = input.to`.
   - If omitted, call `findPreviousProdDeployment()`. If null, throw
     `GuardrailError({ code: 'no_previous_deploy', provider: 'vercel' })`
     with message `"No previous production deployment found for project '<project>' to roll back to"`.
   - POST `/v13/deployments/<targetId>/promote` (urlWithTeam-wrapped) via
     `fetchWithRetry`. Body is empty JSON `{}`.
   - Run through `assertOkOrThrow(res, 'promote deployment')`. The
     existing helper already maps 401/403 ŌåÆ `auth`, 404 ŌåÆ `invalid_config`,
     5xx ŌåÆ `adapter_bug`.
   - Parse the response (Vercel returns the promoted deployment object,
     same shape as `VercelDeployResponse`). Return:
     ```ts
     {
       status: 'pass',
       deployId: targetId,
       rolledBackTo: targetId,
       deployUrl: data.url ? `https://${data.url}` : undefined,
       buildLogsUrl: this.buildLogsUrl(targetId),
       durationMs: this.now() - start,
       output: `Vercel deployment ${targetId} promoted to production`,
     }
     ```

3. **`async listDeployments(limit = 5): Promise<VercelDeployListItem[]>`** (public).
   - GET `/v6/deployments?projectId=<encodeURIComponent(this.project)>&limit=<limit>&target=production`
     (urlWithTeam-wrapped ŌĆö note `urlWithTeam` already handles the `?` vs `&` separator).
   - Run through `assertOkOrThrow(res, 'list deployments')`.
   - Parse `{ deployments: VercelDeployListItem[] }` ŌåÆ return the array
     (default to `[]` if missing).

4. **`private async findPreviousProdDeployment(): Promise<VercelDeployListItem | null>`**
   - `const items = await this.listDeployments(5);`
   - Filter to `state === 'READY'`.
   - Sort by `createdAt` desc (treat missing `createdAt` as 0).
   - If `< 2` remain, return `null`.
   - Return `items[1]` (the one immediately under the current prod).

**Lines added:** ~120. Total file: ~440 ŌåÆ ~560.

### Task 2 ŌĆö CLI runners

**File:** `src/cli/deploy.ts`

1. **Extract `loadDeployConfig(opts)` private helper** that does the
   config-load + adapter-merge dance currently inline at the top of
   `runDeploy`. Returns either `{ adapter, merged }` or `{ error: number }`.
   Reuse from all three runners.

2. **Export `runDeployRollback(opts: RunDeployRollbackOptions): Promise<number>`**
   - `RunDeployRollbackOptions` extends the shared options shape with
     `to?: string`.
   - Build adapter via factory. If `typeof adapter.rollback !== 'function'`,
     print `[deploy] adapter "<name>" does not support rollback` (red),
     return 1.
   - Wrap in try/catch (mirrors `runDeploy`'s error handling for
     `formatErr`).
   - Print on success:
     `[deploy] rollback status=pass adapter=vercel rolledBackTo=<id> duration=<s>s`
     (green); on failure (status !== 'pass'), red.
   - Return 0 on `pass`, else 1.

3. **Export `runDeployStatus(opts: RunDeployStatusOptions): Promise<number>`**
   - Build adapter. If `typeof adapter.listDeployments !== 'function'`,
     print error, return 1.
   - Call `adapter.listDeployments(5)`.
   - Sort returned items by `createdAt` desc (defensive ŌĆö Vercel already
     returns newest-first, but don't rely on it).
   - Print:
     ```
     [deploy] status ŌĆö adapter=vercel
       current: <id> state=<state> url=<url> age=<age>
       recent builds:
         <id> state=<state> age=<age> url=<url>
         ...
     ```
   - Return 0 on success, 1 on failure.

> ## Summary
>
> Implements Phase 3 of the v5.4 Vercel deploy adapter spec ŌĆö rollback
> via `POST /v13/deployments/<id>/promote` plus the `claude-autopilot deploy status`
> and `claude-autopilot deploy rollback` CLI subcommands.

> Spec: [docs/specs/v5.4-vercel-adapter.md](../docs/specs/v5.4-vercel-adapter.md)
>
> ## What ships
>
> - `VercelDeployAdapter.rollback({ to? })` ŌĆö promotes a specific deploy
>   or auto-detects the previous prod deploy when `to` is omitted.
> - `VercelDeployAdapter.listDeployments(limit)` ŌĆö fetches the most
>   recent prod deployments.
> - CLI: `claude-autopilot deploy rollback [--to <id>]` and
>   `claude-autopilot deploy status`.
> - 8 new tests (6 adapter, 2 CLI) covering success, auth failure,
>   not-found, no-previous-deploy edge case, and status listing.
>
> ## Out of scope
>
> - **Phase 4** (auto-rollback wired to `rollbackOn: [healthCheckFailure]`)
>   is deferred to its own PR.
> - **Persistent deploy ID cache** (open question 3 in the spec): defer.
>   The Vercel `/v6/deployments?target=production` list endpoint is one
>   cheap round-trip and the source of truth; a local cache would add a
>   staleness failure mode if a deploy is promoted from outside this CLI.
